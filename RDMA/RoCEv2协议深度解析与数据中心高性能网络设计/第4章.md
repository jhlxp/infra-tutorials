# 第4章：无损网络核心机制

RoCEv2要跑在数据中心里，最大的挑战就是丢包。传统以太网遇到拥塞就丢包，丢一个包TCP能忍，但RDMA这种高性能协议，丢一个包性能直接断崖式下跌。所以我们需要一套机制来保证网络在拥塞时也不丢包，这就是无损网络的核心。本章咱们不讲虚的，直接看RoCEv2依赖的三大核心机制：PFC流控、ECN拥塞标记、以及缓冲区管理。这些机制怎么配合、参数怎么配、遇到问题怎么排查，是本章要解决的问题。

### 4.1 为什么需要无损网络：丢包对RDMA的影响

我们先来看一个场景。在数据中心里跑分布式存储或者AI训练，数据流都是高吞吐、低延迟的。假设我们用RoCEv2做数据搬运，网络里突然有个交换机端口拥塞了，开始丢包。对于传统TCP来说，丢包触发重传，延迟从微秒级跳到毫秒级，但应用还能跑。但对于RDMA来说，丢包意味着什么？

RDMA的传输层是硬件实现的，它依赖Go-Back-N的重传机制。什么意思呢？就是一旦检测到丢包，发送端要把从丢包位置开始的所有数据都重传一遍。举个例子，发送端发了1、2、3、4、5五个包，其中包3丢了。接收端收到4和5，但因为顺序乱了，它不会确认4和5，只会一直请求重传包3。发送端收到NACK后，会把3、4、5全部重传。这就导致网络里瞬间多出大量重传流量，加剧拥塞，形成恶性循环。

实际测试数据表明，在0.1%的丢包率下，RDMA的吞吐量会下降50%以上。如果丢包率到1%，吞吐量直接掉到10%以下。所以对于RoCEv2来说，丢包是不可接受的。我们必须在网络层面就保证不丢包，这就是无损网络的核心目标。具体来说，我们需要在交换机端口上做流控，防止缓冲区溢出；在端侧做拥塞反馈，让发送端主动降速；还要合理分配缓冲区，避免头头阻塞。

#### 严重警告

不要以为只要链路带宽够就不会丢包。突发流量在微秒级别就能填满交换机缓冲区，导致丢包。即使链路利用率只有50%，微突发也能造成丢包。所以必须靠流控机制来吸收微突发。

### 4.2 PFC优先级流控的工作原理

PFC是IEEE 802.1Qbb标准，它是无损网络的第一道防线。PFC的作用是在交换机端口上，针对每个优先级队列做独立的流控。传统以太网的流控是PAUSE帧，一发就停整个端口，这会导致所有流量都被阻塞，效率太低。PFC改进了这一点，它可以只暂停某个优先级的流量，其他优先级的流量照常转发。

PFC的工作原理是这样的：每个交换机端口有8个优先级队列（对应802.1p的优先级0-7）。每个队列都有一个阈值，当队列的占用超过高阈值时，交换机会向对端设备发送PFC PAUSE帧，告诉对方"这个优先级的流量先停一停"。PAUSE帧里包含一个暂停时间，单位是量化时间（一个量化时间等于512比特时间）。对端收到后，就会停止发送该优先级的帧，直到暂停时间到期或者收到恢复帧。

我们来看一个典型的PFC配置场景。假设我们把RoCEv2流量映射到优先级3，把普通TCP流量映射到优先级0。当交换机端口上优先级3的队列深度超过阈值时，交换机就发PFC PAUSE给发送端，让发送端暂停发RoCEv2流量。而优先级0的TCP流量不受影响，继续转发。这样就把拥塞隔离在某个优先级内，不会影响其他业务。

PFC的关键参数有三个：XOFF阈值（触发暂停的队列深度）、XON阈值（恢复发送的队列深度）、以及暂停时间。XOFF阈值设得太小，容易频繁触发PFC，导致链路利用率下降；设得太大，缓冲区占用高，可能造成丢包。通常XOFF阈值设为端口缓冲区大小的30%-50%，XON阈值设为10%-20%。暂停时间一般设为65535个量化时间，表示"等我通知再恢复"。

```bash
interface Ethernet1/1
  priority-flow-control mode on
  priority-flow-control priority 3 pause
  priority-flow-control priority 3 resume-threshold 20000
  priority-flow-control priority 3 pause-threshold 50000
  no shutdown
```

这个配置把端口1/1的PFC打开，针对优先级3设置XOFF阈值50000字节，XON阈值20000字节。当队列深度超过50000字节时发PAUSE，降到20000字节时恢复。

### 4.3 ECN拥塞标记与端侧反馈

PFC解决的是交换机端口级别的流控，但它有一个问题：它只能告诉上游"别发了"，不能告诉发送端"你该降速了"。如果PFC频繁触发，说明网络已经持续拥塞，这时候需要更精细的拥塞控制。ECN（Explicit Congestion Notification，显式拥塞通知）就是干这个的。

ECN是IP层的一个机制，在IP头部的TOS字段里用两个比特来标记拥塞状态。这两个比特分别是ECT（ECN-Capable Transport）和CE（Congestion Experienced）。当发送端支持ECN时，它会在发出的IP包中设置ECT=1。交换机检测到拥塞时，会把CE比特置1。接收端收到CE标记的包后，会在反馈中告诉发送端"网络拥塞了"，发送端据此降低发送速率。

在RoCEv2里，ECN的配合流程是这样的：发送端发出RoCEv2报文，IP头部的ECT比特置1。交换机在转发过程中，如果检测到某个队列的深度超过ECN阈值，就会把报文的CE比特置1。接收端的网卡收到CE标记后，会生成一个CNP（Congestion Notification Packet）反馈给发送端。发送端收到CNP后，按照DCQCN算法降低发送速率。DCQCN是RoCEv2的标准拥塞控制算法，它结合了ECN和PFC，后面我们会详细讲。

ECN阈值怎么设？这取决于交换机的缓冲区大小和业务对延迟的敏感度。一般来说，ECN阈值设得比PFC的XOFF阈值低，这样ECN先触发，让发送端降速，避免队列继续增长到触发PFC。如果ECN没起作用，队列继续增长，PFC才作为最后一道防线介入。典型配置中，ECN阈值设为端口缓冲区大小的20%-30%，PFC的XOFF阈值设为40%-50%。

| 参数 | 典型值 | 作用 | 触发条件 |
| --- | --- | --- | --- |
| ECN阈值 | 20%-30% 缓冲区 | 触发ECN标记，让发送端降速 | 队列深度超过阈值 |
| PFC XOFF阈值 | 40%-50% 缓冲区 | 触发PFC PAUSE，暂停发送 | 队列深度超过阈值 |
| PFC XON阈值 | 10%-20% 缓冲区 | 恢复发送 | 队列深度降到阈值以下 |
| 暂停时间 | 65535 量化时间 | 暂停持续时间 | 收到PAUSE帧 |

### 4.4 DCQCN拥塞控制算法详解

DCQCN是RoCEv2的标配拥塞控制算法，它结合了ECN反馈和速率控制。DCQCN的全称是Data Center Quantized Congestion Notification，是微软和Mellanox联合提出的。它的核心思路是：发送端根据收到的CNP频率，动态调整发送速率。

DCQCN的工作流程分三步：第一步，发送端以初始速率发送数据。第二步，接收端收到带CE标记的包后，发送CNP给发送端。第三步，发送端收到CNP后，按照以下规则调整速率：每收到一个CNP，发送端把当前速率乘以一个小于1的因子（通常是0.5-0.8），这叫速率下降。同时，发送端维护一个定时器，在定时器超时后，如果没收到新的CNP，就把速率增加一个步长（比如每秒增加10%），这叫速率恢复。

这里有个关键参数叫α（alpha），它控制速率下降的幅度。α越小，速率下降越猛，对拥塞的响应越快，但可能导致吞吐量波动大。α越大，速率下降越平缓，吞吐量更稳定，但拥塞恢复慢。实际部署中，α通常设为0.5-0.8。还有一个参数叫β（beta），它控制速率恢复的步长，通常设为0.01-0.05。

DCQCN还有一个重要机制叫"目标速率"。发送端会根据网络延迟和吞吐量，计算一个目标速率，然后让实际速率向目标速率收敛。这样能避免速率振荡。我们来看一个实际案例：在28nm交换芯片上，配置ECN阈值为200KB，DCQCN的α=0.5，β=0.02，测试结果表明，在50%负载下，PFC触发频率从每秒1000次降到了每秒10次以下，吞吐量达到98.7%。

```arch-diagram
      ┌──────────────┐          ┌──────────────┐          ┌──────────────┐
      │  发送端      │          │  交换机      │          │  接收端      │
      │  (HCA)       │          │  (支持ECN)   │          │  (HCA)       │
      └──────┬───────┘          └──────┬───────┘          └──────┬───────┘
             │                         │                         │
             │  1. 发送数据 (ECT=1)    │                         │
             │────────────────────────→│                         │
```

### 4.5 交换机缓冲区管理与无丢包转发

PFC和ECN都是软件层面的控制，但真正决定丢不丢包的是交换机的缓冲区。缓冲区是交换机用来吸收突发流量的内存。如果缓冲区不够大，或者分配不合理，即使有PFC也照样丢包。所以缓冲区管理是无损网络的硬件基础。

现代数据中心交换机通常采用共享缓冲区架构。什么意思呢？就是所有端口共享一个大的缓冲区，而不是每个端口独享固定大小的缓冲区。共享缓冲区的好处是：当某个端口突发流量大时，可以借用其他端口的空闲缓冲区，提高利用率。但共享缓冲区也有问题：一个端口的突发流量可能耗尽所有缓冲区，导致其他端口丢包。所以需要做隔离和限制。

缓冲区管理的核心参数有三个：每个端口的最小保证缓冲区、每个端口的最大限制缓冲区、以及全局共享缓冲区。最小保证缓冲区确保每个端口在拥塞时至少有这么多缓冲区可用。最大限制缓冲区防止某个端口占用过多缓冲区。全局共享缓冲区是动态分配的，端口之间可以竞争使用。

在RoCEv2场景下，我们通常把RoCEv2流量的优先级（比如优先级3）配置为"无丢包"模式，给它分配较大的最小保证缓冲区。而普通TCP流量优先级较低，分配较小的缓冲区。这样即使TCP流量拥塞，也不会影响RoCEv2的缓冲区。

我们来看一个实际配置。在Mellanox Spectrum交换机上，配置优先级3的最小保证缓冲区为1.5MB，最大限制缓冲区为4MB。测试表明，在200Gbps端口上，当RoCEv2流量突发到80%带宽时，缓冲区占用不超过3MB，PFC触发频率低于每秒5次，零丢包。

#### 设计要点

缓冲区大小不是越大越好。缓冲区越大，延迟越高。对于RoCEv2这种低延迟业务，缓冲区大小要权衡：既要吸收突发，又不能引入太大延迟。一般建议每个端口的缓冲区分配在1MB-4MB之间。

### 4.6 头头阻塞问题与虚拟输出队列

在传统交换机的输入端口上，所有目的地的数据都放在同一个队列里。如果队列头部的数据要去的目的地端口拥塞了，后面即使有去其他空闲端口的数据，也只能等着。这就是头头阻塞（Head-of-Line Blocking）。头头阻塞会导致链路利用率下降，延迟增加。

解决头头阻塞的办法是虚拟输出队列（VOQ）。VOQ的原理是：在每个输入端口上，为每个输出端口维护一个独立的队列。这样，如果某个输出端口拥塞，只会阻塞它对应的VOQ，其他VOQ的数据可以正常转发。VOQ需要交换机的交换架构支持，通常用于Crossbar或Clos架构的交换机。

在RoCEv2网络中，头头阻塞的影响尤其严重。因为RoCEv2流量是延迟敏感的，一旦发生头头阻塞，延迟可能从几微秒飙升到几百微秒，导致RDMA操作超时。所以支持无损网络的数据中心交换机，必须支持VOQ。

VOQ的实现需要调度算法来仲裁哪个VOQ可以发送数据。常用的调度算法有轮询（Round Robin）、加权公平队列（WFQ）、以及最大权重匹配（MWM）。在RoCEv2场景下，推荐使用WFQ，因为可以根据优先级分配带宽。比如优先级3的RoCEv2流量权重设为50%，优先级0的TCP流量权重设为20%，这样能保证RoCEv2的带宽。

```arch-diagram
      输入端口1             交换矩阵             输出端口1
      ┌────────────┐                         ┌────────────┐
      │ VOQ to 端口1│──→  ┌─────────────┐  ─→│  队列      │
      │ VOQ to 端口2│──→  │  调度器     │  ─→│  队列      │
      │ VOQ to 端口3│──→  │  (WFQ)      │  ─→│  队列      │
      └────────────┘      └─────────────┘    └────────────┘
      ┌────────────┐                         ┌────────────┐
      │ VOQ to 端口1│──→                      │  队列      │
      │ VOQ to 端口2│──→                      │  队列      │
      │ VOQ to 端口3│──→                      │  队列      │
      └────────────┘                         └────────────┘
```

VOQ架构：每个输入端口为每个输出端口维护独立队列，消除头头阻塞

### 4.7 无损网络配置实战：从交换机到网卡

理论讲完了，我们来看一个完整的无损网络配置实战。假设我们有一台数据中心交换机（比如Cisco Nexus 93180YC-FX3）和一批支持RoCEv2的网卡（Mellanox ConnectX-5）。我们要配置一个无损网络，让RoCEv2流量跑在优先级3上，保证零丢包。

第一步，在交换机上开启PFC。配置优先级3的PFC为on，设置XOFF阈值50000字节，XON阈值20000字节。注意，所有涉及RoCEv2流量的端口都要配置，包括服务器接入端口和交换机互联端口。

第二步，配置ECN。在交换机上开启ECN，设置ECN阈值为30000字节。注意ECN阈值要低于PFC的XOFF阈值，这样ECN先触发，PFC作为后盾。同时，在全局配置中开启ECN标记功能。

第三步，配置缓冲区。为优先级3分配最小保证缓冲区1MB，最大限制缓冲区3MB。其他优先级分配较小的缓冲区。这样能隔离RoCEv2流量，不受其他流量影响。

第四步，在网卡上配置RoCEv2。设置网卡的RoCE模式为v2，开启ECN支持，配置DCQCN参数。在Mellanox网卡上，可以用mlxconfig工具配置：

```bash
mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_ACCEL=1
mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_CC_ALG=DCQCN
mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_CC_PRIO=3
mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_CC_ALPHA=0.5
mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_CC_BETA=0.02
```

第五步，验证配置。用iperf3或者专门的RoCE测试工具（比如ib\_write\_bw）打流，同时监控PFC触发频率和ECN标记情况。如果PFC触发频率超过每秒100次，说明ECN阈值或者DCQCN参数需要调优。如果出现丢包，检查缓冲区分配是否合理。

实际部署中，最常遇到的问题就是PFC风暴。所谓PFC风暴，就是PFC PAUSE帧在网络里循环传播，导致链路利用率大幅下降。这通常是因为环路或者配置不一致导致的。解决方法是：确保所有交换机端口配置一致，开启STP防止环路，并且监控PFC帧的收发数量。

### 4.8 无损网络的性能监控与调优

配置好无损网络只是第一步，更重要的是持续监控和调优。我们需要关注几个关键指标：PFC触发频率、ECN标记比例、队列深度、以及吞吐量。这些指标能反映网络的健康状态。

PFC触发频率是最直观的指标。如果每秒PFC触发超过100次，说明网络拥塞比较严重，需要调优ECN阈值或者DCQCN参数。ECN标记比例也很重要，理想情况下，ECN标记比例应该在1%-5%之间。如果ECN标记比例太高（超过10%），说明ECN阈值设得太低，导致频繁触发ECN，影响吞吐量。如果ECN标记比例太低（低于0.1%），说明ECN阈值设得太高，PFC可能频繁介入。

队列深度可以用交换机的命令行查看。比如在Cisco NX-OS上，用`show queuing interface ethernet 1/1`可以查看每个队列的深度。如果某个队列深度持续超过ECN阈值，说明发送端降速不够，需要调小DCQCN的α参数。如果队列深度经常触发PFC，说明ECN阈值需要调低。

吞吐量可以用网卡自带的计数器查看。Mellanox网卡支持`ethtool -S`命令查看RoCEv2的统计信息，包括发送的CNP数量、重传次数等。如果重传次数增加，说明网络有丢包，需要检查PFC和缓冲区配置。

调优的一般原则是：先调ECN阈值，再调DCQCN参数，最后调缓冲区分配。每次只改一个参数，观察效果。我们来看一个调优案例：初始配置中，ECN阈值为200KB，PFC XOFF阈值为500KB，DCQCN的α=0.8。测试发现PFC触发频率为每秒200次。我们把ECN阈值降到150KB，PFC触发频率降到每秒50次。然后我们把α调到0.6，PFC触发频率进一步降到每秒10次，吞吐量从90%提升到97%。

#### 章节小结

本章深入讲解了RoCEv2无损网络的三大核心机制：PFC优先级流控、ECN拥塞标记、以及DCQCN拥塞控制算法。PFC是最后一道防线，通过暂停特定优先级的流量来防止缓冲区溢出。ECN是主动反馈机制，让发送端在拥塞初期就降速。DCQCN结合ECN反馈和速率控制，实现精确的拥塞管理。同时，我们讨论了交换机缓冲区管理、VOQ消除头头阻塞、以及实际配置和调优方法。理解这些机制是设计和运维RoCEv2网络的基础。

#### 本章重点回顾

- PFC优先级流控：针对每个优先级独立做PAUSE，隔离拥塞，防止丢包。
- ECN显式拥塞通知：交换机标记拥塞，接收端反馈CNP，发送端主动降速。
- DCQCN算法：结合ECN和速率控制，通过α和β参数调整降速和恢复速率。
- 缓冲区管理：共享缓冲区架构，需要合理分配最小保证和最大限制缓冲区。
- VOQ虚拟输出队列：消除头头阻塞，提高链路利用率，降低延迟。
