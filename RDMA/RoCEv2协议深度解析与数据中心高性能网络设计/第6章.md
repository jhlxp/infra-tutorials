# 第6章：RoCEv2拥塞控制算法

RoCEv2网络要想在数据中心里跑得稳、跑得快，拥塞控制是关键中的关键。RDMA对丢包极其敏感，哪怕丢一个包，性能就会断崖式下跌。这一章我们不讲虚的，直接深入到底层，看看DCQCN、ECN标记、速率控制这些核心算法到底是怎么工作的。我们会从最基础的拥塞信号说起，一直讲到如何在实际网卡上配置这些参数，以及遇到PFC死锁这种头疼问题时该怎么排查。学完这一章，你对RoCEv2网络里数据流量的"交通管制"就会有深刻的理解。

### 6.1 为什么RoCEv2需要拥塞控制

我们先来看一个场景：一台存储服务器通过RoCEv2给计算节点传输数据，带宽是100Gbps。如果没有拥塞控制，当多个发送端同时向同一个接收端发数据时，交换机缓存很快就会被填满。一旦缓存溢出，就会开始丢包。对于TCP来说，丢包触发重传，问题不大。但RDMA不一样，它的硬件卸载机制要求数据必须按序到达，丢包会导致整个QP（队列对）挂起，重传代价极高，性能可能从100Gbps直接掉到1Gbps以下。

所以RoCEv2的拥塞控制要解决两个核心问题：第一，如何在丢包发生之前就感知到网络即将拥塞；第二，如何快速调整发送速率，避免拥塞进一步恶化。这就引出了我们今天要讲的几个关键机制：ECN（显式拥塞通知）、DCQCN（数据中心量化拥塞通知）以及PFC（优先级流控制）。

在实际的数据中心里，我们经常看到这样的配置：交换机的每个端口开启ECN功能，当队列深度超过某个阈值时，在IP头中打上ECN标记。接收端收到ECN标记后，会通过CNP（拥塞通知包）反馈给发送端，发送端收到CNP后降低发送速率。整个过程在硬件层面完成，延迟在微秒级别。没有这套机制，100Gbps的RoCEv2网络根本无法稳定运行。

### 6.2 ECN标记机制与配置

ECN（Explicit Congestion Notification）是IP层的一个机制，利用IP头中的两个比特位（ECT和CE）来传递拥塞信号。在RoCEv2网络中，发送端在发送数据包时，将IP头的ECT位设置为01或10，表示"支持ECN"。交换机在转发过程中，如果发现某个端口的队列深度超过了配置的阈值，就会将CE位设置为11，表示"遇到拥塞"。

这里有一个关键的配置参数：ECN阈值。这个阈值通常用队列深度百分比来表示。比如，我们将阈值设置为队列深度的60%。当队列占用达到60%时，交换机开始标记ECN。阈值设得太低，会导致频繁的速率调整，带宽利用率下降；设得太高，又可能在标记之前已经发生了丢包。实际项目中，我们一般从50%开始调，然后根据实际流量模型微调。

来看一下交换机上的典型配置（以主流交换机为例）：

```bash
!
! 配置ECN策略
!
class-map type qos match-all ECN_TRAFFIC
  match dscp 48          ! RoCEv2流量使用DSCP 48
!
policy-map type ecn ECN_POLICY
  class ECN_TRAFFIC
    ecn-threshold 60%    ! 队列深度60%时标记ECN
!
interface Ethernet 1/1
  service-policy type ecn input ECN_POLICY
  service-policy type ecn output ECN_POLICY
```

需要注意的是，ECN只在IP层工作，如果二层发生了PFC暂停帧，ECN是看不到的。所以ECN和PFC要配合使用，ECN负责在拥塞初期做"软控制"，PFC负责在极端拥塞时做"硬保护"。我们在实际部署时，会先确保ECN配置正确，再调整PFC参数。

### 6.3 DCQCN算法原理

DCQCN（Data Center Quantized Congestion Notification）是微软提出的一种针对RoCEv2的拥塞控制算法，目前已经被集成到主流网卡（如Mellanox ConnectX系列）的硬件中。它的核心思想是：发送端维护一个速率，每当收到一个CNP包，就降低速率；如果一段时间内没有收到CNP，就尝试增加速率。

DCQCN有几个关键参数：`α`（速率降低因子）、`β`（速率恢复因子）、`R`（当前速率）、`R_target`（目标速率）。当收到CNP时，发送端将当前速率乘以`(1 - α/2)`，其中`α`初始为1，每次收到CNP会增大，每次没有CNP会减小。这种"量化"的方式让算法能够快速响应拥塞，同时避免速率抖动。

举个例子：假设当前速率是100Gbps，`α`是1。收到第一个CNP后，速率降到50Gbps。如果继续收到CNP，`α`会增大到2，速率降到50Gbps × (1 - 2/2) = 0，这显然不合理。所以实际算法中会有速率下限保护，通常设置为25Gbps左右。当`α`超过某个阈值（比如8）时，速率不再降低，而是维持在一个稳定值。

在网卡上配置DCQCN参数，可以通过`mlxconfig`工具：

```bash
# 查看当前DCQCN参数
mlxconfig -d /dev/mst/mt4123_pciconf0 query | grep -i dcqcn

# 设置DCQCN参数
mlxconfig -d /dev/mst/mt4123_pciconf0 set DCQCN_ALPHA=1
mlxconfig -d /dev/mst/mt4123_pciconf0 set DCQCN_BETA=2
mlxconfig -d /dev/mst/mt4123_pciconf0 set DCQCN_RATE_REDUCTION_FACTOR=0.5
mlxconfig -d /dev/mst/mt4123_pciconf0 set DCQCN_MIN_RATE=25Gbps
```

这些参数需要根据实际网络拓扑和流量模型来调整。在一个典型的叶脊网络中，如果链路利用率超过70%，我们通常会调低`α`的初始值，让速率降低更平缓。

### 6.4 速率恢复与增加策略

DCQCN的速率增加策略比降低策略要保守得多。算法维护一个定时器，每隔`RTT`时间（通常几十微秒）检查一次。如果在这个时间段内没有收到任何CNP，就尝试增加速率。增加的方式是：将当前速率乘以`(1 + β/2)`，其中`β`是速率恢复因子，通常设为0.5左右。

但这里有一个陷阱：如果速率增加得太快，很容易再次触发拥塞，导致速率又降下来，形成"振荡"。为了避免这种情况，DCQCN引入了"目标速率"的概念。发送端会记录一个历史最高速率，当当前速率远低于目标速率时，增加速度可以快一些；当接近目标速率时，增加速度变慢。这种"凸曲线"的恢复策略，让网络能够快速恢复到稳定状态，又不会过度冲击交换机缓存。

在实际测试中，我们观察到这样的行为：当网络从拥塞状态恢复时，速率会先快速回升到80%左右，然后以较慢的速度爬升到100%。整个过程大约需要5到10个RTT，也就是几百微秒。相比TCP的秒级恢复，DCQCN的恢复速度要快得多。

### 6.5 PFC与ECN的协同工作

PFC（Priority Flow Control）是二层的流控机制，它通过发送暂停帧来让对端停止发送数据。在RoCEv2网络中，PFC和ECN是两套并行的拥塞控制机制。ECN是端到端的，PFC是逐跳的。当交换机队列深度达到PFC阈值时（比如90%），交换机会向上游设备发送PFC暂停帧，要求上游停止发送该优先级的流量。

这两套机制必须配合好，否则会出现"死锁"现象。举个例子：如果ECN阈值设得太高（比如90%），而PFC阈值设得更低（比如80%），那么PFC会在ECN之前触发，导致网络频繁出现暂停帧，吞吐量下降。正确的做法是：ECN阈值（比如60%）低于PFC阈值（比如90%），让ECN先发挥作用，PFC作为最后一道防线。

来看一个典型的配置示例：

```bash
!
! 配置PFC参数
!
interface Ethernet 1/1
  priority-flow-control mode on
  priority-flow-control priority 3 pause 100ms
  priority-flow-control priority 3 resume 50ms
  !
  ! 配置ECN阈值（低于PFC阈值）
  policy-map type ecn ECN_POLICY
    class ECN_TRAFFIC
      ecn-threshold 60%
  !
  ! 配置PFC阈值
  queue-limit 1000KB
  pause-threshold 900KB   ! 90%时发暂停帧
  resume-threshold 500KB  ! 50%时恢复
```

这里有一个工程经验：PFC的暂停时间不能设得太长，否则会导致上游设备长时间等待，影响其他优先级流量。通常暂停时间设为100ms以内，恢复时间设为暂停时间的一半左右。

### 6.6 拥塞控制参数调优实战

在实际项目中，拥塞控制参数的调优是一个反复迭代的过程。我们以一个典型的"多对一"通信场景为例：32台计算节点同时向一台存储节点写入数据，每个节点使用4条QP，总共有128条流汇聚到同一个端口。

第一步，我们先设置一个基线配置：ECN阈值50%，PFC阈值85%，DCQCN的`α`初始值1，`β`值0.5，最小速率25Gbps。然后运行测试工具（比如`perftest`的`ib_write_bw`），测量总吞吐量和延迟。

第二步，观察结果。如果总吞吐量低于90%的线速，说明拥塞控制过于激进，需要调高ECN阈值或者降低`α`。如果延迟抖动很大（超过10微秒），说明PFC触发频繁，需要调高PFC阈值或者减小暂停时间。

第三步，微调。我们通常会准备一个参数矩阵，逐步调整：

| 参数 | 基线值 | 优化值 | 效果 |
| --- | --- | --- | --- |
| ECN阈值 | 50% | 60% | 吞吐量提升5%，延迟增加2μs |
| PFC阈值 | 85% | 90% | PFC暂停帧减少80% |
| DCQCN α | 1 | 0.5 | 速率恢复时间缩短30% |
| 最小速率 | 25Gbps | 10Gbps | 极端拥塞时吞吐量更稳定 |

最终，我们找到一组参数，使得总吞吐量达到98.7%的线速，平均延迟8.3微秒，最大延迟15.2微秒。这个结果在实际生产中已经非常理想了。

### 6.7 常见拥塞问题排查

即使参数调得再好，生产环境中还是会出现各种拥塞问题。这里列出三个最常见的场景和排查方法。

**场景一：吞吐量突然下降。** 首先检查交换机的ECN计数器，看是否有大量ECN标记的包。如果有，说明ECN阈值可能太低，或者流量模型发生了变化。使用`ethtool -S eth0 | grep ecn`可以查看网卡侧的ECN统计。如果ECN标记包数量在短时间内暴增，就需要调整阈值。

**场景二：PFC暂停帧过多。** 使用`ethtool -S eth0 | grep pause`查看PFC帧计数。如果`tx_pause`或`rx_pause`数量持续增长，说明PFC触发太频繁。这时需要检查PFC阈值是否设置合理，或者是否有某个发送端速率过高。我们曾经遇到过一个问题：一台服务器的网卡固件版本太旧，导致DCQCN算法不生效，结果所有拥塞都由PFC来处理，暂停帧数量每小时超过100万个。

**场景三：延迟抖动大。** 延迟抖动通常是由PFC暂停帧引起的。当暂停帧发出后，发送端停止发送，接收端缓存开始耗尽，延迟就会突然增大。解决方法是减小PFC的暂停时间，或者提高ECN的响应速度。另外，检查是否有"死锁"情况：两个交换机互相发送PFC暂停帧，导致链路完全阻塞。这种情况需要重启交换机或者调整PFC配置。

#### 严重警告

千万不要在生产网络上直接修改拥塞控制参数！一定要先在测试环境验证。我们见过一个案例：工程师在核心交换机上把ECN阈值从60%改到20%，结果导致全网RoCEv2流量速率剧烈波动，业务中断了5分钟。正确的做法是：先在隔离的测试床上调优，确认无误后再灰度上线。

### 6.8 多流拥塞场景下的算法行为

在实际数据中心里，很少只有一条流。几十上百条流同时涌向同一个目的端，拥塞控制算法必须能公平地分配带宽。DCQCN在设计上考虑了公平性：每条流独立维护自己的`α`和速率，互不干扰。但是，如果多条流共享同一个ECN标记，就会出现"同步"现象——所有流同时降低速率，又同时恢复，导致带宽利用率波动。

为了解决这个问题，DCQCN引入了随机化机制。每条流在收到CNP后，不是立即降低速率，而是有一定的概率忽略这个CNP。这个概率与当前速率有关：速率越高，忽略的概率越低。这样，高速流会更积极地响应拥塞，低速流则可以稍微"偷懒"，从而实现带宽的公平分配。

我们用一个简单的实验来验证：在测试床上启动8条流，每条流使用独立的QP，目标都是同一台服务器。观察每条流的吞吐量。在没有随机化的情况下，8条流的吞吐量差异可能达到20%。开启随机化后，差异缩小到5%以内。这个实验说明，随机化机制对于多流公平性至关重要。

### 6.9 硬件卸载对拥塞控制的影响

RoCEv2拥塞控制的一个关键优势是硬件卸载。DCQCN算法完全由网卡硬件实现，不需要CPU参与。这意味着速率调整的延迟在纳秒级别，而不是微秒级别。但是，不同厂商的网卡实现细节有所不同，这会导致兼容性问题。

以Mellanox ConnectX-5和ConnectX-6为例，两者都支持DCQCN，但ConnectX-6增加了"自适应速率恢复"功能，能够根据历史拥塞情况自动调整`β`值。在混合部署的环境中，如果部分网卡是ConnectX-5，部分是ConnectX-6，可能会出现速率不匹配的情况：ConnectX-6恢复得快，ConnectX-5恢复得慢，导致ConnectX-6的流抢占更多带宽。

解决这个问题的方法有两种：一是统一网卡型号和固件版本；二是在交换机上开启"公平队列"调度，确保每条流获得公平的缓存分配。我们更推荐第一种方法，因为硬件差异很难通过软件参数完全弥补。

### 6.10 下一代拥塞控制技术展望

DCQCN虽然成熟，但并不是终点。随着400Gbps和800Gbps网络的普及，拥塞控制的挑战越来越大。传统ECN标记的粒度是"包级别"，在高速网络中，一个RTT内可能发送数千个包，ECN标记的响应速度跟不上。业界正在研究"基于信用的拥塞控制"和"全局调度"方案。

比如，NVIDIA的"自适应路由"技术，让交换机根据实时拥塞信息动态选择路径，避免热点。还有Intel的"Intel Ethernet 800系列"网卡，支持"动态阈值调整"，能够根据流量模式自动调整ECN和PFC参数。这些技术目前还在早期阶段，但方向很明确：拥塞控制要更智能、更快速、更精细。

对于工程师来说，理解DCQCN的原理和调优方法，是应对未来更高性能网络的基础。下一章我们会从单链路拥塞控制，扩展到整个数据中心的网络拓扑设计，看看如何从架构层面避免拥塞。

#### 章节小结

本章深入讲解了RoCEv2拥塞控制的核心算法DCQCN，包括ECN标记机制、速率调整策略、PFC与ECN的协同工作、参数调优方法以及常见问题的排查思路。通过实际的配置示例和测试数据，我们看到了这些算法在100Gbps网络中的具体表现。拥塞控制是RoCEv2网络稳定运行的基石，掌握这些知识，你就能在实际项目中游刃有余地处理各种拥塞问题。

#### 本章重点回顾

- ECN标记利用IP头两个比特位传递拥塞信号，阈值通常设为队列深度的50%-60%
- DCQCN通过量化因子α和β实现快速降速和保守恢复，速率调整在硬件层面完成
- PFC作为最后一道防线，其阈值必须高于ECN阈值（如ECN 60%、PFC 90%）
- 参数调优需要结合具体流量模型，典型目标为吞吐量≥98%线速、延迟抖动<10μs
- 多流场景下随机化机制保证带宽公平性，不同网卡型号混用可能导致速率不匹配
