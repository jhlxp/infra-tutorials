# 第15章：丢包检测与重传机制

在RoCEv2网络中，数据包丢失是影响性能的头号杀手。传统TCP/IP的丢包恢复机制需要几十毫秒，这对高带宽、低延迟的RDMA网络来说是不可接受的。本章我们聚焦于RoCEv2如何实现亚毫秒级的丢包检测与重传，核心围绕两个技术路径：基于硬件的Go-Back-N重传和基于E2E确认的选择性重传。我们会深入到具体的数据包格式、定时器配置、NACK生成逻辑，以及如何在实际的Mellanox CX-5/CX-6网卡上调整这些参数。理解这些机制，是设计无损数据中心网络的基础。

### 15.1 丢包对RoCEv2性能的影响模型

我们先来定量分析一下丢包对RoCEv2流量的具体影响。假设一条100Gbps的链路，RTT为10微秒（数据中心内部典型值）。如果发生一次丢包，采用Go-Back-N重传时，发送端需要重传从丢失包开始之后的所有数据包。假设窗口大小为256个256字节的包，一次丢包会导致重传约64KB数据。这个重传过程加上等待NACK的时间，大约会增加15-20微秒的延迟。但如果丢包率上升到0.1%，有效吞吐量会从100Gbps直接腰斩到50Gbps以下。

我们来看一个具体的工程参数。在Mellanox CX-6网卡上，当开启DCQCN拥塞控制时，丢包率被控制在10⁻⁶以下。一旦丢包率超过10⁻⁵，Go-Back-N机制就会频繁触发，导致吞吐量急剧下降。这是因为每次重传都要回退到丢包点，后续所有已经成功发送的包都要重新传输。在金融高频交易场景中，一个丢包可能导致交易延迟从5微秒飙升到50微秒，直接导致交易策略失效。

### 15.2 Go-Back-N重传协议实现细节

Go-Back-N是RoCEv2中最基础的重传策略，由硬件（网卡）直接实现，不需要软件干预。它的核心思想是：发送端维护一个发送窗口，窗口内的包可以连续发送。当接收端检测到某个包丢失时，会发送一个NACK（Negative ACK）给发送端，NACK中携带了期望接收的包序号（PSN）。发送端收到NACK后，立即回退到该PSN，重传从该PSN开始之后的所有包。

在硬件实现上，每个发送队列都有一个PSN计数器和一个重传缓冲区。重传缓冲区的大小决定了可以回退的最大深度。在CX-6网卡上，每个QP（Queue Pair）的重传缓冲区默认配置为512个64字节的包，即32KB。这个值可以通过`mlxconfig`工具调整。如果重传缓冲区太小，当窗口内丢包过多时，硬件可能无法完成重传，导致连接断开。

```verilog
// Go-Back-N重传控制状态机（简化硬件描述）
module go_back_n_retx (
    input  clk, rst_n,
    input  [23:0] tx_psn,          // 当前发送PSN
    input  [23:0] nack_psn,        // 收到的NACK中的PSN
    input  nack_valid,
    output reg retx_enable,
    output reg [23:0] retx_start_psn,
    output reg [7:0] retx_count    // 需要重传的包数量
);

    reg [23:0] last_acked_psn;     // 最后确认的PSN
    reg [7:0] retx_buf_depth;      // 重传缓冲区深度

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            retx_enable <= 1'b0;
            retx_start_psn <= 24'd0;
            retx_count <= 8'd0;
            last_acked_psn <= 24'd0;
        end else if (nack_valid) begin
            // 检查NACK的PSN是否在重传缓冲区范围内
            if ((nack_psn >= last_acked_psn) &&
                (nack_psn < (last_acked_psn + retx_buf_depth))) begin
                retx_enable <= 1'b1;
                retx_start_psn <= nack_psn;
                // 计算需要重传的包数：当前发送PSN - NACK PSN
                retx_count <= tx_psn[7:0] - nack_psn[7:0];
            end
        end else begin
            retx_enable <= 1'b0;
        end
    end
endmodule
```

这个状态机的关键点在于：NACK中携带的PSN必须是接收端期望收到的下一个PSN。例如，发送端发送了PSN=100,101,102,103，其中PSN=101丢失。接收端收到了100,102,103，它会发现PSN不连续（期望101，却收到了102），于是生成NACK，其中PSN=101。发送端收到NACK后，从PSN=101开始重传101,102,103。注意，102和103虽然已经被接收端收到，但接收端会丢弃它们，等待重传的101到来后再重新组装。

#### 严重警告

Go-Back-N在丢包率超过0.1%时会严重浪费带宽。因为每次重传都会回退，导致大量已经成功传输的包被重传。在RoCEv2网络中，如果观察到吞吐量异常下降，首先检查NACK计数器的值。使用`ethtool -S eth0 | grep nack`可以查看NACK统计。

### 15.3 NACK生成逻辑与PSN连续性检查

接收端如何判断一个包丢失了？这依赖于PSN连续性检查。每个RoCEv2数据包的BTH（Base Transport Header）中都包含一个24位的PSN。接收端硬件维护一个`expected_psn`寄存器，初始化为0。每收到一个数据包，硬件将包中的PSN与`expected_psn`比较。如果相等，说明包是连续的，`expected_psn`加1。如果收到的PSN大于`expected_psn`，说明有包丢失了，硬件立即生成一个NACK，其中包含`expected_psn`的值。

这里有一个工程细节：NACK的生成有延迟吗？在硬件实现中，NACK是在包到达的同一个时钟周期内生成的。但实际中，由于包解析和流水线延迟，通常会有几个时钟周期的延迟。在CX-5网卡上，这个延迟大约是40ns（8个时钟周期@200MHz）。这个延迟对于10微秒级别的RTT来说可以忽略不计。但需要注意的是，如果接收端同时收到多个乱序包，它只会在第一个乱序包到达时生成一个NACK，后续的乱序包不会触发额外的NACK，这可以避免NACK风暴。

NACK包本身也是一个RoCEv2传输层包，它的Opcode是0x81（针对RC服务的NACK）。NACK包中包含了`NACK_PSN`字段，即期望的PSN。发送端收到NACK后，必须立即停止发送新数据，并开始重传。在CX-6网卡上，从收到NACK到开始重传的硬件延迟约为100ns。这个速度远快于软件重传（毫秒级）。

### 15.4 重传定时器配置与超时处理

除了基于NACK的快速重传，RoCEv2还提供了基于定时器的重传机制，作为NACK机制的补充。当发送端发送一个包后，会启动一个重传定时器（Retransmission Timer）。如果在定时器超时前没有收到该包的ACK（或NACK），硬件就会自动重传该包。这个定时器主要用于应对NACK包本身丢失的情况。

定时器的超时值配置非常关键。在Mellanox网卡上，这个值通过`QP上下文`中的`retry_count`和`rnr_retry`字段间接控制。实际超时时间由硬件自动计算，基于当前链路的RTT估计值。在CX-5网卡上，默认的超时倍数是4.096微秒乘以一个系数（1到31）。对于10微秒RTT的链路，建议将超时倍数设置为4（即16.384微秒）。如果设置得太小，会导致不必要的重传；设置得太大，会延长丢包恢复时间。

| 参数名称 | 默认值 | 推荐值（10us RTT） | 说明 |
| --- | --- | --- | --- |
| retry\_count | 7 | 4 | 最大重传次数，超过则连接断开 |
| rnr\_retry | 7 | 3 | 接收端未就绪时的重试次数 |
| timeout\_scale | 4 | 4 | 超时倍数（1-31），单位4.096us |
| 实际超时值 | 16.384us | 16.384us | timeout\_scale × 4.096us |

配置这些参数可以通过`ibv_modify_qp`系统调用。在DPDK应用中，可以直接操作QP上下文结构体。需要注意的是，定时器超时重传和NACK触发的重传是两套独立的机制。如果NACK先到，定时器会被取消。如果定时器先超时，硬件会重传所有未确认的包，这实际上是一种保守的Go-Back-N行为。在工程实践中，我们通常希望NACK机制主导重传，定时器仅作为兜底。因此，将超时值设置得稍大一些（比如RTT的1.5倍）是合理的。

#### 信息

在Mellanox CX-6网卡上，可以通过`mlxreg`工具读取硬件重传计数器：`mlxreg -d /dev/mst/mt4123_pciconf0 --reg_name QPCRT --get`。该寄存器记录了每个QP的重传次数和超时次数。

### 15.5 选择性重传（SRQ）与硬件卸载

Go-Back-N的一个主要缺点是重传了大量不必要的包。为了优化这一点，RoCEv2的扩展规范（IBTA 1.4）引入了选择性重传（Selective Retransmission）的概念，但实际硬件实现中，这通常与SRQ（Shared Receive Queue）结合使用。在SRQ模式下，接收端可以预先发布多个接收缓冲区，硬件可以乱序接收数据包，并将它们放入对应的缓冲区中。当某个包丢失时，只有丢失的那个包需要重传，后续已经收到的包不需要重传。

这听起来很理想，但实际实现中，选择性重传需要接收端维护一个复杂的位图（Bitmap）来记录哪些包已经收到。在100Gbps速率下，每微秒可以收到约500个256字节的包，位图更新频率极高。目前，只有少数高端网卡（如CX-7及以上）支持硬件卸载的选择性重传。在CX-6上，虽然SRQ支持乱序接收，但重传仍然是Go-Back-N模式。也就是说，接收端可以乱序接收并存放在不同缓冲区，但发送端仍然会重传从丢失点开始的所有包。

```arch-diagram

│  选择性重传 vs Go-Back-N 对比                                │
├──────────────────────────────────────────────────────────────┤
│  发送序列：P0 P1 P2 P3 P4  (P2丢失)                        │
│                                                              │
│  Go-Back-N: 重传 P2 P3 P4  (浪费P3,P4的带宽)               │
```

在工程实践中，如果应用对带宽利用率要求极高（如HPC场景），建议使用支持选择性重传的网卡。但要注意，选择性重传需要更大的接收缓冲区（因为要缓存乱序包）。在CX-7上，每个QP的接收缓冲区建议配置为至少128KB，以容纳乱序到达的包。如果缓冲区不足，硬件会丢弃乱序包，导致性能反而下降。

### 15.6 重传缓冲区管理与内存占用

重传缓冲区是网卡内部用于暂存已发送但未确认的数据包的内存区域。每个QP都有自己的重传缓冲区，大小可以通过`ibv_create_qp`时的`sq_sig_all`和`max_send_wr`参数间接控制。在CX-6上，每个QP的重传缓冲区最大可以配置为1MB。对于512个QP的网卡，如果每个QP配置1MB，总共需要512MB的板载内存，这已经接近网卡的内存上限了。

因此，实际部署时需要权衡。对于延迟敏感的金融交易应用，建议为每个QP配置256KB的重传缓冲区，这可以容纳约1024个256字节的包。对于HPC应用，由于消息通常较大，建议配置512KB。重传缓冲区的大小直接影响可以回退的深度。如果重传缓冲区太小，当窗口内连续丢包较多时，硬件可能无法找到需要重传的包，导致连接错误。

```c
// 使用libibverbs创建QP时配置重传缓冲区大小
struct ibv_qp_init_attr init_attr = {
    .send_cq = cq,
    .recv_cq = cq,
    .cap = {
        .max_send_wr = 1024,       // 发送WR数量，影响重传缓冲区深度
        .max_recv_wr = 1024,
        .max_send_sge = 1,
        .max_recv_sge = 1,
        .max_inline_data = 64
    },
    .qp_type = IBV_QPT_RC,
    .sq_sig_all = 1               // 所有发送WR都需要信令完成
};

struct ibv_qp *qp = ibv_create_qp(pd, &init_attr);
if (!qp) {
    fprintf(stderr, "创建QP失败，可能重传缓冲区不足\n");
    exit(1);
}
// 实际重传缓冲区大小由驱动根据max_send_wr和MTU自动计算
// 对于MTU=1024字节，max_send_wr=1024，重传缓冲区≈1MB
```

需要注意的是，重传缓冲区是网卡内部资源，不占用主机内存。但每个QP的重传缓冲区是独立分配的，所以QP数量越多，占用的网卡内存越大。在CX-5上，板载内存为4MB，最多支持8个QP配置256KB重传缓冲区。在CX-6上，板载内存为16MB，可以支持64个QP配置256KB缓冲区。如果QP数量超过这个限制，驱动会自动降低每个QP的重传缓冲区大小，可能导致重传深度不足。

#### 设计要点

在规划QP数量时，务必计算重传缓冲区的总需求。一个简单的公式：`总重传缓冲区 = QP数量 × max_send_wr × MTU`。确保这个值不超过网卡板载内存的80%，留出余量给其他功能（如拥塞控制状态表）。

### 15.7 丢包检测的端到端确认（ACK/NACK）流程

RoCEv2的RC（Reliable Connection）服务使用端到端的确认机制。每个数据包发送后，接收端必须回复一个ACK或NACK。ACK包中包含了`ACK_PSN`字段，表示接收端期望的下一个PSN。发送端收到ACK后，可以释放重传缓冲区中对应PSN之前的所有包。NACK包则触发重传。

ACK的生成策略有两种：立即ACK和延迟ACK。立即ACK是每收到一个包就回复一个ACK，这会增加网络中的小包数量。延迟ACK是每收到多个包（通常是2-4个）才回复一个ACK，可以减少ACK包的数量。在RoCEv2中，默认使用延迟ACK，延迟的包数可以通过`QP上下文`中的`ack_freq`字段配置。在CX-6上，`ack_freq`的默认值为4，即每4个数据包回复一个ACK。

| ACK策略 | ACK包数量 | CPU开销 | 延迟影响 | 适用场景 |
| --- | --- | --- | --- | --- |
| 立即ACK | 与数据包1:1 | 高 | 低（及时确认） | 低延迟交易 |
| 延迟ACK(4) | 数据包的1/4 | 低 | 中等（延迟一个窗口） | HPC/存储 |
| 延迟ACK(8) | 数据包的1/8 | 极低 | 较高 | 大块数据传输 |

在丢包发生时，延迟ACK策略会影响恢复速度。例如，如果接收端采用延迟ACK（4），当第5个包丢失时，接收端需要等到第8个包到达后才能生成NACK（因为前4个包已经触发了一个ACK，第5个包丢失，第6-8个包到达时触发NACK）。这增加了NACK的生成延迟。因此，对于丢包敏感的应用，建议使用立即ACK或降低`ack_freq`到2。

### 15.8 实际网络中的丢包恢复时间分析

我们来计算一个实际的丢包恢复时间。假设RTT=10us，数据包大小=256字节，链路速率=100Gbps。发送端窗口大小=256。当第128个包丢失时，恢复过程如下：

- 接收端检测到丢包：在收到第129个包时发现PSN不连续，生成NACK。延迟约0.1us（包处理时间）。
- NACK传输时间：约0.1us（小包在100Gbps链路上传输时间极短）。
- 发送端收到NACK并开始重传：硬件延迟约0.1us。
- 重传第128-256个包：共129个包，传输时间为129 × 256B × 8 / 100Gbps ≈ 2.64us。
- 接收端收到重传包并恢复：约0.1us。
- 总恢复时间 ≈ 0.1 + 0.1 + 0.1 + 2.64 + 0.1 = 3.04us。

这个时间远小于TCP的毫秒级恢复时间。但如果丢包发生在窗口的末尾，比如第256个包丢失，重传只需要1个包，恢复时间仅约0.4us。最坏情况是丢包发生在窗口的开头，需要重传整个窗口，恢复时间约3us。在10us RTT的链路中，这已经是非常优秀的性能了。

```arch-diagram
┌─────────────────────────────────────────────────────────────┐
│  丢包恢复时间分解（RTT=10us，窗口=256）                    │
├─────────────────────────────────────────────────────────────┤
│  阶段                         时间(us)   占比              │
│  丢包检测与NACK生成           0.2        6.6%              │
│  NACK传输                     0.1        3.3%              │
│  发送端处理与重传开始         0.1        3.3%              │
│  重传数据包传输（129个包）    2.64       86.8%             │
│  总恢复时间                   3.04       100%              │
└─────────────────────────────────────────────────────────────┘
```

表15-2 丢包恢复时间分解

从这个分析可以看出，重传数据包的传输时间占了总恢复时间的绝大部分。因此，减小数据包大小或增大链路带宽可以进一步缩短恢复时间。这也是为什么RoCEv2通常使用256字节或512字节的MTU，而不是标准的1500字节。小MTU可以减少重传的数据量，加快恢复速度。

### 15.9 丢包与拥塞控制的联动机制

丢包检测和拥塞控制是紧密相关的。在DCQCN算法中，当接收端检测到拥塞（通过ECN标记）时，会减少发送速率。但如果发生了真正的丢包，说明拥塞控制可能没有及时生效，需要更激进地降速。在CX-6网卡上，当NACK触发重传时，硬件会自动将当前QP的发送速率降低50%，并在后续的RTT中逐渐恢复。这个机制称为“丢包触发速率衰减”。

具体实现上，每个QP内部有一个`rate_limit`寄存器，初始值为链路速率（100Gbps）。当收到NACK时，硬件将`rate_limit`设置为当前值的一半，然后以每RTT增加5%的速度恢复。这个衰减因子（50%）和恢复速率（5%/RTT）可以通过`mlxconfig`中的`RATE_REDUCTION_FACTOR`和`RATE_RECOVERY_FACTOR`参数调整。在金融场景中，建议将衰减因子设置为25%，恢复速率设置为10%/RTT，以更快地恢复吞吐量。

#### 严重警告

不要禁用丢包触发的速率衰减。如果禁用，一次丢包可能导致连续丢包，最终触发重传超时和连接断开。在CX-5上，禁用该功能的命令是`mlxconfig -d /dev/mst/mt4123_pciconf0 set RATE_REDUCTION_EN=0`，这仅在特定测试场景下使用。

丢包和ECN标记的联动也很重要。如果接收端同时收到ECN标记的包和检测到丢包，它会优先处理丢包（生成NACK），因为丢包对性能的影响更大。在DCQCN的CNP（Congestion Notification Packet）生成逻辑中，如果已经生成了NACK，就不会再生成CNP，避免重复的降速指令。

### 15.10 工程实践：配置与调试丢包恢复参数

在实际部署中，我们需要根据应用场景调整丢包恢复相关的参数。下面是一个典型的配置流程，适用于Mellanox CX-6网卡和Linux 5.4+内核。

第一步：检查当前丢包统计。使用`ethtool -S eth0 | grep -E "nack|retry|timeout"`查看NACK、重传和超时的计数。如果`nack_sent`或`nack_rcv`持续增长，说明网络中存在丢包。正常情况下，这些计数器在数小时内不应有明显增长。

第二步：调整重传缓冲区大小。通过`ibv_devinfo`查看当前QP的最大发送WR数量。如果`max_send_wr`小于512，说明重传缓冲区可能不足。在应用代码中，通过`ibv_create_qp`的`max_send_wr`参数将其增加到1024或2048，但要确保不超过网卡内存限制。

第三步：配置NACK生成延迟。在CX-6上，可以通过`mlxreg`设置`QPC_NACK_DELAY`寄存器，单位是8个时钟周期（40ns）。默认值为0（立即生成）。如果网络中存在短暂的乱序（例如多路径场景），可以设置延迟为1-2，避免因短暂乱序而误触发NACK。

```bash
# 查看当前QP的重传统计
ethtool -S eth0 | grep -E "nack|retry|timeout"

# 设置NACK生成延迟为80ns（2个时钟周期）
mlxreg -d /dev/mst/mt4123_pciconf0 --reg_name QPC --index 0x10 --set "nack_delay=2"

# 调整QP的重传缓冲区大小（通过ibv_create_qp的max_send_wr参数）
# 在应用启动前设置环境变量
export MLX5_SEND_WQE_BB_NUM=1024

# 验证配置是否生效
ibv_devinfo -d mlx5_0 | grep max_send_wr
```

第四步：测试丢包恢复性能。使用`ib_write_bw`工具，在链路中注入丢包（通过tc命令模拟），观察吞吐量下降和恢复时间。例如：`tc qdisc add dev eth0 root netem loss 0.01%`模拟0.01%丢包率，然后运行`ib_write_bw -d mlx5_0 --run_infinitely`，观察吞吐量是否稳定在99%以上。

第五步：监控和告警。在监控系统中添加对`nack_sent`和`retry_exceeded`计数器的告警。如果1分钟内NACK数量超过1000，说明网络存在严重丢包，需要检查PFC配置或链路质量。

#### 章节小结

本章详细讲解了RoCEv2的丢包检测与重传机制。我们从丢包对性能的影响模型入手，定量分析了不同丢包率下的吞吐量衰减。然后深入Go-Back-N重传协议的硬件实现，包括NACK生成逻辑、PSN连续性检查、重传定时器配置。接着讨论了选择性重传的优缺点和重传缓冲区管理。最后，我们通过实际案例分析丢包恢复时间，并介绍了丢包与拥塞控制的联动机制，以及工程实践中的配置调试方法。理解这些机制，是构建可靠无损RoCEv2网络的关键。

#### 本章重点回顾

- 丢包率超过10⁻⁵时，RoCEv2吞吐量开始显著下降，必须通过PFC和DCQCN将丢包率控制在10⁻⁷以下。
- Go-Back-N重传由硬件直接实现，NACK触发重传的延迟在亚微秒级，但会重传大量不必要的包。
- 重传缓冲区大小直接影响可回退深度，在CX-6上每个QP建议配置256KB-512KB，总需求不超过网卡板载内存的80%。
- 丢包触发速率衰减是拥塞控制的重要补充，衰减因子建议设置为50%，恢复速率5%/RTT。
- 工程调试中，通过ethtool监控nack和retry计数器，使用tc命令模拟丢包测试恢复性能。
