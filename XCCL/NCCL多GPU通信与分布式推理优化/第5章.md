# 第5章：集合通信性能基准测试

做分布式训练，最怕的就是“加卡不加性能”。咱们搞NCCL，不能光知道怎么用API，还得知道怎么测。这一章咱们就来讲怎么给NCCL的集合通信做性能基准测试。核心要解决两个问题：一是我的多卡通信到底跑多快？二是瓶颈到底在哪？我们会从最基础的带宽概念讲起，然后手把手带你装好工具、写测试脚本、分析结果。这一章学完，你就能自己动手给集群的通信性能“体检”了。

### 5.1 带宽、延迟与吞吐量：通信性能的三个核心指标

咱们先得搞清楚测什么。在NCCL里，衡量通信性能有三个最基础的指标：带宽、延迟和吞吐量。带宽就像水管子的粗细，单位是GB/s，决定了数据能跑多快。延迟就是水龙头打开到水流出来的时间，单位是微秒（μs），它决定了每次通信的“启动”成本。吞吐量是单位时间内实际完成的数据传输量，它受带宽和延迟共同影响。

在分布式训练里，这几个指标怎么用？举个例子，你的模型是GPT-3那种千亿参数的，每次AllReduce的数据量是几百MB到几个GB。这时候带宽就是瓶颈，因为数据量大，水管子粗不粗直接决定训练快慢。但如果你做的是小模型实时推理，每次通信只有几百KB，那延迟就是关键，因为启动时间占比大。

NCCL官方推荐用`nccl-tests`来测这些指标。这个工具会发不同大小的数据包，然后记录每次通信的耗时。我们来看一个典型的输出结果长什么样。

| 数据大小 (Bytes) | 带宽 (GB/s) | 延迟 (μs) | 总线带宽利用率 (%) |
| --- | --- | --- | --- |
| 256 | 0.02 | 12.3 | 5.2 |
| 4096 | 0.35 | 14.1 | 28.6 |
| 65536 | 4.82 | 18.7 | 68.4 |
| 1048576 | 11.53 | 45.2 | 92.1 |
| 16777216 | 12.81 | 320.6 | 98.7 |

表5-1：NCCL AllReduce性能测试典型输出（8×A100 80GB NVLink）

看到没？数据小的时候（256字节），延迟只有12.3微秒，但带宽只有0.02 GB/s，利用率才5.2%。这是因为大部分时间花在“握手”和“同步”上了。当数据大到16MB时，带宽跑到了12.81 GB/s，利用率接近99%。这就是咱们做分布式训练时追求的状态——让通信“吃饱”。

实际项目里，我们一般关注两个点：一是小数据的延迟（比如128字节到1KB），这决定了通信的“响应速度”；二是大数据的带宽（比如16MB到256MB），这决定了训练吞吐。如果你发现小数据延迟很高（超过20μs），那可能是NVLink或者PCIe链路有问题。如果大数据带宽上不去，那就要看是不是Ring算法或者Tree算法的选择问题了。

#### 信息

NCCL的带宽利用率不是越高越好。如果利用率长期维持在99%以上，说明通信链路已经饱和，再增加通信量只会排队。这时候应该考虑计算和通信的重叠，而不是继续压榨带宽。

### 5.2 安装与编译nccl-tests：你的第一个性能测试工具

讲完理论，咱们动手。nccl-tests是NVIDIA官方提供的NCCL性能测试套件。它不依赖别的库，只需要NCCL本身。咱们来走一遍安装流程。

第一步，确认NCCL已经装好。在终端跑一下`ldconfig -p | grep nccl`，如果看到libnccl.so，说明NCCL已经装好了。版本至少2.12以上，因为新版本对NVLink和IB的支持更好。

第二步，下载nccl-tests。直接git clone：

```bash
git clone https://github.com/NVIDIA/nccl-tests.git
cd nccl-tests
```

第三步，编译。这里有个技巧：如果服务器上有多个GPU，比如4张A100，编译的时候可以指定CUDA路径和NCCL路径。

```bash
make CUDA_HOME=/usr/local/cuda NCCL_HOME=/usr/local/nccl2
```

如果一切顺利，你会看到`build/`目录下生成了一堆可执行文件，比如`all_reduce_perf`、`all_gather_perf`、`reduce_scatter_perf`等。这些就是咱们的“体检工具”。

编译过程中可能遇到几个坑：一是CUDA路径不对，报`cuda_runtime.h: No such file or directory`。这时候检查一下`CUDA_HOME`是不是指向了正确的目录。二是NCCL版本不匹配，比如你装了NCCL 2.16，但nccl-tests是旧的，可能编译不过。建议用最新的nccl-tests，或者直接指定NCCL版本。

装好之后，跑一个最简单的命令验证：

```bash
./build/all_reduce_perf -b 1M -e 256M -f 2 -g 4
```

这个命令的意思是：用4个GPU，从1MB开始测到256MB，每次数据量翻2倍。如果能看到类似表5-1的输出，就说明环境没问题了。

### 5.3 单节点内通信测试：NVLink与PCIe的实战对比

单节点内的通信，主要看GPU之间是怎么连的。A100和H100用的是NVLink，带宽高延迟低；而一些老卡或者低端卡走的是PCIe，带宽差很多。咱们用nccl-tests来实测一下差距。

先跑一个NVLink的测试。假设你有4张A100，它们通过NVLink全互联。

```bash
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 4
# 结果示例（NVLink）：
# 8M: 12.5 GB/s  延迟 640 μs
# 128M: 12.8 GB/s  延迟 10240 μs
```

再跑一个PCIe的测试。如果只有2张卡通过PCIe连接，或者你用`NCCL_NET=Socket`强制走网络模拟PCIe。

```bash
export NCCL_NET=Socket
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 4
# 结果示例（PCIe 4.0 x16）：
# 8M: 3.2 GB/s  延迟 2500 μs
# 128M: 3.5 GB/s  延迟 36500 μs
```

看到差距了吗？NVLink的带宽是PCIe的3.6倍，延迟只有1/4。这就是为什么高端服务器一定要用NVLink。但实际项目里，不是所有GPU都支持NVLink。比如一些云服务器，GPU之间走的是PCIe Switch，那通信性能就会差很多。

这里有个工程经验：如果你的模型并行度很大（比如Tensor Parallelism），通信频繁且数据量大，那NVLink是刚需。如果只是数据并行（Data Parallelism），通信次数少但单次数据量大，PCIe也能凑合。我们之前做过一个项目，在PCIe 4.0的8卡服务器上跑175B模型，AllReduce带宽只有3.2 GB/s，导致训练效率只有NVLink方案的60%。后来换成了NVLink的服务器，直接翻倍。

单节点测试还有一个重要指标：卡间通信的“对称性”。你可以用`all_gather_perf`来测，看每张卡的发送和接收带宽是否一致。如果某张卡明显偏低，那可能是NVLink链路有问题，或者PCIe插槽没插好。

#### 严重警告

在生产环境中，千万别直接用`NCCL_NET=Socket`来模拟PCIe测试，因为Socket走的是CPU内存，带宽和延迟都比真实PCIe差很多。正确的做法是用`NCCL_P2P_DISABLE=1`来禁用GPU直连，强制走PCIe。

### 5.4 多节点通信测试：RDMA与TCP的性能差异

单节点测完了，咱们看多节点。多节点通信的核心是网卡。高端集群用InfiniBand（IB）或者RoCE，走RDMA；低端或者云环境走TCP。咱们用nccl-tests来测一下差异。

首先，确认网卡类型。跑一下`ibstat`（如果有IB卡）或者`ip a`看网络接口。假设你有两台服务器，每台4张A100，通过IB卡互联。测试命令要指定多节点：

```bash
# 在节点1上运行
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 8 --nccl-tests-mpi 2
# 在节点2上运行（需要MPI环境）
mpirun -np 2 -host node1,node2 ./build/all_reduce_perf -b 8M -e 128M -f 2 -g 8
```

测试结果（以InfiniBand HDR100为例）：

| 数据大小 (Bytes) | IB带宽 (GB/s) | TCP带宽 (GB/s) | IB延迟 (μs) | TCP延迟 (μs) |
| --- | --- | --- | --- | --- |
| 8192 | 0.85 | 0.12 | 9.4 | 68.2 |
| 65536 | 6.21 | 0.95 | 10.5 | 72.1 |
| 1048576 | 11.43 | 2.81 | 45.8 | 180.5 |
| 16777216 | 12.51 | 3.12 | 320.1 | 2100.3 |

表5-2：多节点AllReduce性能对比（2节点，每节点4×A100）

IB的优势非常明显：带宽是TCP的4倍，延迟是TCP的1/7。而且数据越大，差距越明显。这是因为RDMA绕过了CPU和内核，直接让GPU和网卡通信。TCP的话，数据要从GPU拷贝到CPU内存，再经过内核协议栈，开销巨大。

实际项目中，如果条件允许，一定要用IB或者RoCE。如果只能用TCP，那就要想办法减少跨节点通信。比如用梯度压缩（Gradient Compression）或者异步通信（Async Communication）。我们之前在一个云集群上做训练，只有25Gbps的TCP网络，AllReduce带宽只有2.5 GB/s。后来我们用了`NCCL_NET_GDR_LEVEL=5`开启了GPU Direct RDMA（虽然只有TCP，但也能绕过一些拷贝），带宽提升到了3.8 GB/s。

### 5.5 不同集合通信操作的性能对比：AllReduce vs AllGather vs ReduceScatter

NCCL支持好几种集合通信操作，每种操作的性能特征不一样。咱们用nccl-tests来对比一下最常用的三种：AllReduce、AllGather和ReduceScatter。

测试命令很简单，换一下可执行文件就行：

```bash
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 8
./build/all_gather_perf -b 8M -e 128M -f 2 -g 8
./build/reduce_scatter_perf -b 8M -e 128M -f 2 -g 8
```

结果对比（8×A100 NVLink，数据大小128MB）：

| 操作类型 | 带宽 (GB/s) | 延迟 (μs) | 算法 |
| --- | --- | --- | --- |
| AllReduce | 12.81 | 10240 | Ring |
| AllGather | 12.53 | 10245 | Ring |
| ReduceScatter | 12.92 | 10235 | Ring |

表5-3：不同集合通信操作的性能对比（8×A100 NVLink，128MB数据）

从表里看，三种操作的带宽和延迟差不多。这是因为在大数据量下，它们都用了Ring算法，通信模式类似。但在小数据量下，差异就出来了。比如128字节的AllReduce延迟是12.3μs，而AllGather是13.1μs，ReduceScatter是11.8μs。

为什么有差异？因为小数据量下，通信的“启动成本”占主导。AllReduce需要两次通信（Reduce + Broadcast），而ReduceScatter只需要一次Reduce，所以延迟更低。AllGather需要一次Gather，但需要所有节点都收到数据，所以延迟略高。

在工程实践中，选择哪种操作取决于你的模型结构。比如在Tensor Parallelism中，我们经常用ReduceScatter来聚合梯度，因为每个GPU只需要一部分梯度。在Data Parallelism中，AllReduce是标准选择。在Pipeline Parallelism中，AllGather用于收集各阶段的输出。所以，了解每种操作的性能特征，能帮你选择最优的通信模式。

#### 设计要点

如果你的模型通信量很大（比如每个batch通信超过100MB），优先用Ring算法，因为它的带宽利用率高。如果通信量小（小于1MB），可以考虑用Tree算法（通过`NCCL_ALGO=Tree`设置），因为它的延迟更低。

### 5.6 算法选择对性能的影响：Ring vs Tree vs CollNet

NCCL支持多种通信算法，最常用的是Ring和Tree，还有最新的CollNet（用于多节点）。算法的选择直接影响性能。咱们来实测一下。

先看Ring算法，这是NCCL的默认算法。它把GPU排成一个环，数据在环上依次传递。好处是带宽利用率高，适合大数据。坏处是延迟随GPU数量线性增长。

```bash
export NCCL_ALGO=Ring
./build/all_reduce_perf -b 1K -e 1M -f 2 -g 8
```

再看Tree算法。它把GPU组织成树形结构，数据从叶子节点汇聚到根节点，再广播回去。好处是延迟低（log(N)级别），适合小数据。坏处是带宽利用率不如Ring。

```bash
export NCCL_ALGO=Tree
./build/all_reduce_perf -b 1K -e 1M -f 2 -g 8
```

对比结果（8×A100，数据大小1KB）：

| 算法 | 延迟 (μs) | 带宽 (GB/s) |
| --- | --- | --- |
| Ring | 18.5 | 0.05 |
| Tree | 12.3 | 0.08 |

表5-4：Ring与Tree算法在小数据下的性能对比（1KB AllReduce）

看到没？小数据下Tree的延迟比Ring低34%。但如果你换成大数据（128MB），Ring的带宽是12.81 GB/s，Tree只有11.2 GB/s。所以，没有绝对的好坏，关键看数据量。

CollNet是NCCL 2.14引入的新算法，专门针对多节点。它利用网卡的硬件多播能力，可以在一个通信步骤内完成AllReduce，延迟更低。但需要网卡支持（比如IB的硬件多播）。实测下来，在4节点、每节点8卡的环境下，CollNet的延迟比Ring低20%，带宽高10%。

实际项目中，我们一般让NCCL自动选择算法。但如果你发现性能不对劲，可以手动指定。比如在训练小模型时（通信量小），强制用Tree；在大模型时，用Ring。多节点场景，优先尝试CollNet。

### 5.7 通信拓扑对性能的影响：环形拓扑与树形拓扑的实测

除了算法，通信拓扑（也就是GPU的物理连接方式）也会影响性能。比如NVLink全互联（每个GPU都直连）和NVLink环形（GPU连成一圈）的带宽不一样。咱们用nccl-tests来测一下。

假设你有4张A100，有两种拓扑：

- **全互联**：每张卡都通过NVLink与其他三张卡直连，带宽最大。
- **环形**：卡0连卡1，卡1连卡2，卡2连卡3，卡3连卡0，带宽受限于单条链路。

在NVIDIA的DGX服务器上，A100是全互联。但在一些定制服务器上，可能是环形。我们可以通过`nvidia-smi topo -m`查看拓扑。

测试命令：

```bash
# 全互联（默认）
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 4
# 模拟环形（通过设置NCCL_TOPO_FILE）
export NCCL_TOPO_FILE=./ring_topo.xml
./build/all_reduce_perf -b 8M -e 128M -f 2 -g 4
```

结果对比：

| 拓扑 | 8M带宽 (GB/s) | 128M带宽 (GB/s) | 延迟 (μs) |
| --- | --- | --- | --- |
| 全互联 | 12.5 | 12.8 | 640 |
| 环形 | 6.2 | 6.4 | 1280 |

表5-5：全互联与环形拓扑的AllReduce性能对比

环形的带宽只有全互联的一半，延迟翻倍。这是因为在环形拓扑下，数据要经过多跳才能到达所有GPU，每条链路的带宽被共享了。

这个结果告诉我们：买服务器的时候，一定要看GPU拓扑。全互联的NVLink是最好的，但价格也贵。如果预算有限，环形也能用，但要做好性能下降的心理准备。另外，在软件层面，我们可以通过调整NCCL的通信顺序来优化。比如用`NCCL_MIN_NCHANNELS`和`NCCL_MAX_NCHANNELS`来调整通信通道数，有时候能缓解环形拓扑的带宽瓶颈。

### 5.8 性能瓶颈定位：如何从测试结果中找出问题

跑完测试，拿到了数据，下一步就是分析。咱们来看几个常见的性能问题，以及怎么从测试结果中定位。

**问题一：带宽上不去。** 比如8张A100 NVLink，理论带宽应该是12.8 GB/s，但你测出来只有6 GB/s。可能的原因：一是NVLink链路降速了（比如某条链路接触不良），二是CPU内存带宽不足（如果用了GDR拷贝）。排查方法：先用`nvidia-smi nvlink -s`查看每条NVLink的速率和状态。如果速率正常，再用`nvidia-smi topo -m`看拓扑，确认是不是环形。如果都没问题，可能是NCCL的通道数不够，可以试试`export NCCL_MIN_NCHANNELS=16`。

**问题二：小数据延迟高。** 比如128字节的AllReduce延迟超过20μs。可能的原因：一是CPU频率低，导致NCCL的同步开销大；二是系统中断太多，影响了通信。排查方法：用`perf stat`看上下文切换次数。如果很高，可以绑核（CPU Affinity）来优化。另一个常见原因是NCCL的`NCCL_IB_TIMEOUT`设置太大，导致超时等待。可以调小到`NCCL_IB_TIMEOUT=10`。

**问题三：多节点通信不稳定。** 比如每次测试的带宽波动超过10%。可能的原因：一是网络拥塞，二是网卡散热导致降频。排查方法：用`ibstat`看网卡速率和错误计数。如果有`link_downed`或者`symbol_error`，说明链路有问题。另外，检查网卡温度，如果超过80°C，可能降频了。

我们之前遇到一个案例：4节点集群，AllReduce带宽只有理论值的60%。排查了三天，最后发现是交换机的一个端口配置成了100Gbps而不是200Gbps。改过来之后，带宽直接翻倍。所以，测试结果一定要和理论值对比，偏差超过20%就要排查。

#### 严重警告

不要只看一次测试结果。NCCL的性能受系统负载影响很大。建议每个配置至少跑5次，取平均值。如果某次结果明显偏离（比如带宽低了50%），那可能是系统有瞬时抖动，需要重跑。

### 5.9 自定义测试脚本：自动化批量测试与结果分析

手动跑nccl-tests太慢了，而且参数多容易出错。咱们写一个自动化脚本，批量测试不同配置，并把结果保存下来。

```bash
#!/bin/bash
# nccl_benchmark.sh - 自动化NCCL性能测试
# 用法: ./nccl_benchmark.sh [gpu_count] [min_size] [max_size]

GPU_COUNT=${1:-8}
MIN_SIZE=${2:-1K}
MAX_SIZE=${3:-256M}
OUTPUT_DIR="nccl_results_$(date +%Y%m%d_%H%M%S)"

mkdir -p $OUTPUT_DIR

# 测试AllReduce
echo "Testing AllReduce with $GPU_COUNT GPUs..."
./build/all_reduce_perf -b $MIN_SIZE -e $MAX_SIZE -f 2 -g $GPU_COUNT \
    > $OUTPUT_DIR/all_reduce_${GPU_COUNT}gpu.txt 2>&1

# 测试AllGather
echo "Testing AllGather..."
./build/all_gather_perf -b $MIN_SIZE -e $MAX_SIZE -f 2 -g $GPU_COUNT \
    > $OUTPUT_DIR/all_gather_${GPU_COUNT}gpu.txt 2>&1

# 测试ReduceScatter
echo "Testing ReduceScatter..."
./build/reduce_scatter_perf -b $MIN_SIZE -e $MAX_SIZE -f 2 -g $GPU_COUNT \
    > $OUTPUT_DIR/reduce_scatter_${GPU_COUNT}gpu.txt 2>&1

# 测试不同算法
for algo in Ring Tree; do
    export NCCL_ALGO=$algo
    echo "Testing AllReduce with $algo algorithm..."
    ./build/all_reduce_perf -b 1K -e 1M -f 2 -g $GPU_COUNT \
        > $OUTPUT_DIR/all_reduce_${algo}_${GPU_COUNT}gpu.txt 2>&1
done

echo "All tests completed. Results saved in $OUTPUT_DIR"
```

这个脚本会测试三种集合操作和两种算法，结果保存到带时间戳的目录。你可以把它加到crontab里，每天凌晨跑一次，监控集群性能变化。

有了原始数据，怎么分析？我一般用Python写个简单的解析脚本，提取关键指标（比如128MB时的带宽和延迟），然后画折线图。这样能直观看到性能趋势。比如，你可以对比不同GPU数量的性能，看扩展性好不好。

```python
# 简单解析nccl-tests输出
import re

def parse_nccl_output(filepath):
    """从nccl-tests输出中提取带宽和延迟"""
    bandwidths = []
    latencies = []
    sizes = []
    with open(filepath, 'r') as f:
        for line in f:
            # 匹配类似 "  1048576   12.81   45.2" 的行
            match = re.search(r'\s+(\d+)\s+([\d.]+)\s+([\d.]+)', line)
            if match:
                sizes.append(int(match.group(1)))
                bandwidths.append(float(match.group(2)))
                latencies.append(float(match.group(3)))
    return sizes, bandwidths, latencies

# 使用示例
sizes, bw, lat = parse_nccl_output('nccl_results/all_reduce_8gpu.txt')
for i in range(len(sizes)):
    print(f"Size: {sizes[i]:>10} bytes, BW: {bw[i]:>6.2f} GB/s, Lat: {lat[i]:>6.1f} μs")
```

这个脚本虽然简单，但能帮你快速从几十个测试文件中提取关键数据。实际项目中，我们还会加上异常检测，比如如果某次测试的带宽低于历史平均值的20%，就自动报警。

#### 设计要点

做性能测试报告时，一定要包含“与理论值的对比”这一项。比如理论NVLink带宽是12.8 GB/s，实测12.5 GB/s，利用率97.7%，这样大家一眼就能看出性能是否正常。如果利用率低于80%，就要写进“待排查问题”列表。

#### 章节小结

这一章咱们从零开始，讲了NCCL性能基准测试的完整流程。从三个核心指标（带宽、延迟、吞吐量）入手，装了nccl-tests，实测了单节点和多节点的性能差异，对比了不同集合操作和算法，还讲了怎么定位瓶颈和自动化测试。核心思想是：性能测试不是跑一次就完事，而是一个持续监控和优化的过程。只有把通信的“体检报告”做扎实了，才能知道分布式训练的瓶颈在哪。

#### 本章重点回顾

- **带宽、延迟、吞吐量**是通信性能的三个核心指标，大数据看带宽，小数据看延迟。
- **nccl-tests**是官方推荐的性能测试工具，编译简单，支持多种集合操作和算法。
- **NVLink vs PCIe**：NVLink带宽是PCIe的3-4倍，延迟是1/4，高端训练场景必须用NVLink。
- **Ring vs Tree**：大数据用Ring，小数据用Tree；多节点优先尝试CollNet。
