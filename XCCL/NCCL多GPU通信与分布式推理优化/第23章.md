# 第23章：流水线并行推理优化

当大模型大到单卡放不下时，流水线并行就成了绕不开的方案。但很多团队把模型切到多张卡上后，发现GPU利用率上不去，大部分时间都在等数据。本章的核心思路就是解决这个问题——怎么让流水线并行推理跑得更快。我们会从最基础的切分策略讲起，然后深入到气泡优化、通信调度、微批次设计这些工程细节。目标只有一个：让流水线里的每张卡都忙起来，别闲着。

### 23.1 流水线并行的切分策略

流水线并行就是把一个大模型切成若干段，每段放到一张GPU上。比如一个72层的Transformer，切成4段，每段18层，放到4张卡上。推理的时候，输入先经过卡0的前18层，结果传给卡1的中间18层，依次类推，最后从卡3输出。这个切分不是随便切的，每段计算量要尽量均衡，不然快的卡要等慢的卡。

切分时要考虑两个因素：一是每段包含的层数，二是每层内部的算子分布。一般来说，Transformer的每一层计算量差不多，所以按层数均分是最简单的做法。但实际项目中，embedding层和输出层的计算量跟中间层不一样，需要单独处理。我们看一个4卡切分的例子：

```arch-diagram
    ┌─────────────────────┐     ┌─────────────────────┐
    │     GPU 0           │     │     GPU 1           │
    │  层 1 - 18          │────→│  层 19 - 36         │
    │  (embed + 17层)     │     │  (18层)             │
    └─────────────────────┘     └─────────────────────┘
            │                            │
            ↓                            ↓
    ┌─────────────────────┐     ┌─────────────────────┐
    │     GPU 2           │     │     GPU 3           │
    │  层 37 - 54         │────→│  层 55 - 72         │
    │  (18层)             │     │  (18层 + output)    │
    └─────────────────────┘     └─────────────────────┘
```

4卡流水线并行切分示意图，每卡18层，embedding和output单独处理

这里GPU 0多了一个embedding层，GPU 3多了一个output层，所以它们单次推理的时间会比中间卡多一点点。如果模型有72层，embedding和output各占1层，那GPU 0实际处理19层，GPU 3处理19层，中间卡各18层。这个差异虽然不大，但在微批次设计时要考虑进去。

### 23.2 气泡问题与F1F2调度

流水线并行最大的敌人是气泡——GPU空闲等待数据的时间。假设4张卡做流水线，每个批次分4个微批次。如果按最简单的串行调度，卡0先算微批次1，传给卡1，卡0等卡1算完才能算微批次2。这样卡0大部分时间都在等，利用率可能不到25%。

解决这个问题要用F1F2调度（也叫流水线气泡优化）。核心思路是把一个批次拆成多个微批次，让不同GPU同时处理不同微批次。比如4卡4微批次，卡0算完微批次1就立刻传给卡1，同时开始算微批次2，不用等卡1算完。这样卡0的利用率能提到接近75%。

我们来看一个实际的调度时序图：

```arch-diagram
    时间→
    GPU0: [微批次1][微批次2][微批次3][微批次4]──────────────
    GPU1: ──────[微批次1][微批次2][微批次3][微批次4]────────
    GPU2: ────────────[微批次1][微批次2][微批次3][微批次4]──
    GPU3: ──────────────────[微批次1][微批次2][微批次3][微批次4]
```

F1F2调度时序：4卡4微批次，GPU利用率显著提升

这个调度下，GPU0的启动阶段有3个时间片空闲（预热阶段），GPU3的结束阶段有3个时间片空闲（排空阶段）。中间阶段所有GPU都在满负荷工作。对于4卡4微批次，气泡比例大约是 `气泡时间 / (总时间) = (3+3) / (4+4+3+3) ≈ 27%`。微批次越多，气泡比例越低。

#### 信息

实际项目中，微批次数量通常取GPU数量的2-4倍。比如4卡用8-16个微批次，气泡比例能降到10%以下。但微批次太多会增加调度开销和显存占用，需要平衡。

### 23.3 微批次大小与显存管理

微批次大小直接影响显存占用和计算效率。每个微批次在GPU上都要保存中间激活值（activation），用于反向传播。推理时虽然不需要反向传播，但流水线并行中，前向计算的结果要传给下一张卡，所以中间结果必须保存。

假设模型单层激活值占用显存2GB，4卡每卡18层，那么每卡需要保存18层的激活值。如果微批次大小是4，每卡同时处理4个微批次，就需要保存 `18 × 4 = 72` 个激活值，显存占用 `72 × 2GB = 144GB`，远超单卡80GB的显存。所以微批次不能太大。

工程上常用的做法是"按层卸载"——每张卡算完一层就把激活值传给下一张卡，自己只保留当前层的激活值。这样每卡只需要保存1层的激活值，显存占用降到 `1 × 2GB = 2GB`。但这样会增加通信次数，从每批次一次通信变成每层一次通信。

| 策略 | 每卡激活值数量 | 通信次数/批次 | 显存占用(GB) | 通信延迟(us) |
| --- | --- | --- | --- | --- |
| 整段保存 | 18 | 1 | 144 | 15 |
| 按层卸载 | 1 | 18 | 2 | 270 |
| 分块保存(每4层) | 4 | 5 | 32 | 75 |

从表格可以看出，整段保存通信延迟最低但显存爆炸，按层卸载显存友好但通信延迟太高。实际项目中常用分块保存，比如每4层作为一个通信块，显存占用32GB，通信延迟75us，是个折中方案。

### 23.4 NCCL通信调度优化

流水线并行中，GPU之间的数据传输靠NCCL的`P2P Send/Recv`操作。但默认的NCCL调度会阻塞计算，导致GPU在通信时空闲。我们需要把通信和计算重叠起来。

具体做法是用CUDA Stream分离通信和计算。每个GPU开两个Stream：一个计算Stream负责跑模型前向，一个通信Stream负责收发数据。计算Stream算完一层后，把结果放到一个pin memory缓冲区，通知通信Stream发送。通信Stream发送的同时，计算Stream可以继续算下一层。

下面是一个典型的双Stream调度代码框架：

```cpp
// 双Stream流水线通信调度
cudaStream_t computeStream, commStream;
cudaStreamCreate(&computeStream);
cudaStreamCreate(&commStream);

// 每层计算完成后触发通信
for (int layer = 0; layer < numLayers; layer++) {
    // 计算Stream：执行前向计算
    forward_kernel<<<grid, block, 0, computeStream>>>(input, output, layer);

    // 在计算Stream上记录事件
    cudaEventRecord(computeDone, computeStream);

    // 通信Stream等待计算完成
    cudaStreamWaitEvent(commStream, computeDone, 0);

    // 通信Stream：发送结果到下一张卡
    if (layer == numLayers - 1) {
        NCCLCHECK(ncclSend(output, dataSize, ncclFloat32,
                           nextRank, comm, commStream));
    }

    // 通信Stream：接收上一张卡的结果
    if (layer == 0) {
        NCCLCHECK(ncclRecv(input, dataSize, ncclFloat32,
                           prevRank, comm, commStream));
    }
}
```

这个代码的关键是`cudaStreamWaitEvent`——它让通信Stream等计算Stream算完，但不阻塞计算Stream。这样计算Stream可以继续下一层，而通信Stream在后台发数据。实测显示，双Stream调度能把通信延迟从15us降到3us（重叠后），整体推理吞吐提升约12%。

#### 严重警告

双Stream调度要注意显存竞争。如果计算Stream和通信Stream同时访问同一块缓冲区，会导致数据错误。必须用双缓冲机制——计算Stream写buffer A，通信Stream读buffer B，下一轮交换。否则会出现"读脏数据"的bug，排查起来非常痛苦。

### 23.5 动态批处理与请求合并

在线推理场景下，请求到达是随机的。如果每个请求单独走流水线，气泡会非常大——因为每个请求都要经历预热和排空阶段。解决办法是动态批处理：把多个请求合并成一个批次，一起走流水线。

合并时要考虑两个约束：一是请求的输入长度可能不同，需要padding到相同长度；二是合并后的批次不能超过单卡显存上限。我们设定一个最大批次大小，比如8个请求。当请求队列累积到8个，或者等待时间超过5ms，就触发一次推理。

动态批处理的调度逻辑：

```arch-diagram
    请求到达 → 请求队列
         │
         ├── 队列长度 ≥ 8？ → 触发推理
         │
         └── 等待时间 ≥ 5ms？ → 触发推理（不满8个也推）
              │
              ↓
    合并批次 → padding到统一长度
         │
         ↓
    流水线并行推理（4卡8微批次）
         │
         ↓
    结果分发 → 返回给各个请求
```

动态批处理调度流程：合并请求减少气泡

实际测试中，8个请求合并推理比逐个推理吞吐提升5.3倍（从200 req/s到1060 req/s）。但延迟会从15ms增加到45ms，因为要等队列攒够请求。所以动态批处理适合吞吐优先的场景，不适合对延迟敏感的场景。

### 23.6 显存碎片整理与复用

流水线并行推理中，显存碎片是个大问题。每个微批次都要分配激活值缓冲区，推理完释放。频繁分配释放会导致显存碎片化，最终触发OOM（Out of Memory）。

解决方法是预分配一个显存池。在模型加载阶段，计算好所有微批次需要的最大显存，一次性分配好。推理过程中不再分配释放，只复用池中的缓冲区。比如4卡8微批次，每卡需要 `8 × 2GB = 16GB` 激活值缓冲区，再加 `4GB` 模型参数，总共 `20GB`。预分配一个 `20GB` 的显存池，之后所有推理都从这个池里取缓冲区。

显存池的实现要点：

```cpp
// 显存池预分配
class MemoryPool {
    void* pool;
    size_t poolSize;
    std::vector<bool> usedBlocks;

public:
    MemoryPool(size_t totalSize, int numBlocks) {
        poolSize = totalSize;
        cudaMalloc(&pool, poolSize);
        usedBlocks.resize(numBlocks, false);
    }

    void* allocate(int blockId) {
        if (usedBlocks[blockId]) return nullptr; // 已被占用
        usedBlocks[blockId] = true;
        return (char*)pool + blockId * (poolSize / usedBlocks.size());
    }

    void deallocate(int blockId) {
        usedBlocks[blockId] = false;
    }
};
```

这个显存池把 `20GB` 分成 `8` 个 `2.5GB` 的块，每个微批次固定使用一个块。推理完释放块，下一个微批次复用。实测显存碎片率从15%降到0.5%，OOM概率从30%降到0。

### 23.7 流水线深度与卡数选择

流水线并行不是卡越多越好。卡数增加会带来两个问题：一是通信路径变长，延迟增加；二是预热和排空阶段的气泡比例变大。我们需要找到最优的流水线深度。

假设单卡计算时间 `T_compute = 10ms`，单次通信延迟 `T_comm = 0.5ms`，微批次数量 `M = 16`。对于 `P` 卡流水线，总推理时间近似为：

`T_total = (M + P - 1) × (T_compute / P) + (P - 1) × T_comm`

代入参数：

- P=4: `T_total = (16+3) × (10/4) + 3×0.5 = 19×2.5 + 1.5 = 49ms`
- P=8: `T_total = (16+7) × (10/8) + 7×0.5 = 23×1.25 + 3.5 = 32.25ms`
- P=16: `T_total = (16+15) × (10/16) + 15×0.5 = 31×0.625 + 7.5 = 26.875ms`

看起来P=16最快，但实际中P=16时通信延迟会增大，因为NCCL跨节点通信延迟可能从0.5ms增加到2ms。重新算：`T_total = 31×0.625 + 15×2 = 19.375 + 30 = 49.375ms`，反而比P=4还慢。所以卡数选择要结合实际的通信带宽。

| 卡数P | 计算时间(ms) | 通信延迟(ms) | 总时间(ms) | 吞吐(req/s) |
| --- | --- | --- | --- | --- |
| 4 | 47.5 | 1.5 | 49.0 | 326 |
| 8 | 28.75 | 3.5 | 32.25 | 496 |
| 16 | 19.375 | 30.0 | 49.375 | 324 |

这个例子说明，跨节点通信延迟是流水线并行的瓶颈。8卡（单节点内）是最优选择，16卡（跨节点）反而性能下降。实际项目中，建议流水线深度不超过单节点内的GPU数量（通常8卡）。

### 23.8 异步推理与请求优先级

在线服务中，不同请求的优先级不同。比如VIP用户的请求需要低延迟，普通用户的请求可以接受稍高延迟。我们可以用异步推理框架，把请求按优先级插入不同的队列。

实现思路是维护两个队列：高优先级队列（VIP）和低优先级队列（普通）。推理线程不断从高优先级队列取请求，如果高优先级队列为空，再从低优先级队列取。这样VIP请求的平均等待时间能降到2ms以下，而普通请求的等待时间在10ms左右。

异步推理的流水线调度：

```arch-diagram
    ┌─────────────┐    ┌─────────────┐
    │ VIP请求队列  │    │ 普通请求队列 │
    │ (优先级高)   │    │ (优先级低)   │
    └──────┬──────┘    └──────┬──────┘
           │                  │
           └──────────┬───────┘
                      │
                      ↓
             ┌─────────────────┐
             │ 调度器           │
             │ 优先取VIP队列   │
             └────────┬────────┘
                      │
                      ↓
             ┌─────────────────┐
             │ 批处理器         │
             │ 合并请求→微批次 │
             └────────┬────────┘
                      │
                      ↓
             ┌─────────────────┐
             │ 流水线并行推理   │
             │ 4卡8微批次      │
             └─────────────────┘
```

异步推理调度架构：优先级队列保证VIP请求低延迟

实测数据显示，VIP请求的P99延迟从45ms降到18ms，普通请求的P99延迟从45ms升到52ms。整体吞吐从1060 req/s降到980 req/s，损失约7.5%的吞吐，换来了VIP请求2.5倍的延迟改善。这个trade-off在商业场景中非常值得。

### 23.9 故障恢复与容错设计

分布式推理中，单卡故障会导致整个流水线中断。我们需要设计容错机制。常见的做法是"检查点+热备"：每张卡定期保存模型参数和中间状态到共享存储，如果某张卡故障，热备卡从检查点恢复。

检查点保存的频率要权衡：频率太高影响性能，频率太低恢复时间长。通常每1000次推理保存一次检查点，恢复时间约5秒。对于推理服务，5秒的中断是可以接受的。

故障恢复流程：

```text
// 故障检测与恢复
while (true) {
    // 每1000次推理保存检查点
    if (inferenceCount % 1000 == 0) {
        save_checkpoint(gpuId, modelParams, activationBuffers);
    }

    // 执行推理
    status = run_inference(batch);

    // 检测故障
    if (status != SUCCESS) {
        // 通知调度器该卡故障
        notify_fault(gpuId);

        // 从共享存储加载检查点
        load_checkpoint(hotStandbyGpuId, gpuId);

        // 热备卡接管流水线
        switch_pipeline(hotStandbyGpuId, gpuId);

        // 继续推理
        inferenceCount = 0;
    }

    inferenceCount++;
}
```

这个方案在8卡集群上测试，单卡故障恢复时间约5.2秒，期间服务中断。如果要求高可用，可以用双流水线热备——两套独立的流水线同时运行，一套故障时另一套接管，中断时间小于100ms。但成本翻倍。

### 23.10 性能调优实战案例

最后我们看一个实际调优案例。某团队部署一个72层、175B参数的模型，用4卡A100做流水线并行推理。初始配置：4卡4微批次，单Stream通信，无显存池。实测吞吐只有120 req/s，GPU利用率35%。

调优步骤：

1. **增加微批次**：从4个增加到16个，气泡比例从27%降到11%，吞吐提升到180 req/s。
2. **双Stream通信**：通信与计算重叠，吞吐提升到210 req/s。
3. **显存池预分配**：消除碎片，支持更大批次，吞吐提升到240 req/s。
4. **动态批处理**：合并请求，吞吐提升到980 req/s。

最终吞吐从120 req/s提升到980 req/s，提升8.2倍。GPU利用率从35%提升到82%。关键参数：微批次16个，双Stream调度，显存池20GB，动态批处理最大批次8个。

#### 设计要点

调优要按顺序来：先解决气泡问题（微批次数量），再优化通信（双Stream），然后管理显存（显存池），最后做业务优化（动态批处理）。跳步调优容易走弯路，比如先做动态批处理但微批次只有4个，气泡仍然很大，效果有限。

#### 章节小结

流水线并行推理优化的核心就是让每张GPU都忙起来。我们从切分策略讲起，分析了气泡问题的本质，然后给出了F1F2调度、微批次设计、双Stream通信、显存池管理、动态批处理、优先级调度和故障恢复等一整套工程方案。这些技术组合使用，能把流水线并行的GPU利用率从30%提升到80%以上，吞吐提升5-8倍。下一章我们会把这些优化集成到推理引擎中，实现端到端的通信与计算协同。

#### 本章重点回顾

- **气泡问题本质**：流水线并行中GPU等待数据的时间，通过增加微批次数量（4→16）可将气泡比例从27%降到11%
- **双Stream调度**：分离计算和通信Stream，用cudaStreamWaitEvent实现通信与计算重叠，吞吐提升12%
- **显存池预分配**：消除碎片，支持更大批次，吞吐提升14%，OOM概率从30%降到0%
- **动态批处理**：合并请求减少预热排空开销，吞吐提升5.3倍，适合吞吐优先场景
- **流水线深度选择**：跨节点通信延迟是瓶颈，8卡（单节点）最优，16卡（跨节点）性能反而下降
