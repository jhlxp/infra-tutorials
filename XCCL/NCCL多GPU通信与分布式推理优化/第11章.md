# 第11章：NCCL集合操作API详解

在分布式训练中，多个GPU之间需要频繁交换数据，比如梯度同步、参数广播、数据分发等。NCCL提供了一套标准的集合通信API，封装了底层复杂的网络和内存管理逻辑。这一章我们就来详细拆解这些API的用法、参数含义和工程中的注意事项。我们会从最基础的`ncclAllReduce`开始，逐步深入到`Broadcast`、`ReduceScatter`、`AllGather`等操作，并配合实际代码和参数配置，让你能直接在项目里用起来。

### 11.1 AllReduce：梯度同步的核心

AllReduce是分布式训练中使用最频繁的集合操作，没有之一。它的作用是把所有GPU上的数据（比如梯度张量）进行归约，然后把结果分发到每个GPU上。在数据并行训练中，每个GPU计算完自己的梯度后，必须调用AllReduce来同步梯度，才能保证所有GPU的模型参数保持一致。

NCCL提供了`ncclAllReduce`函数，原型如下：

```c
ncclResult_t ncclAllReduce(const void* sendbuff, void* recvbuff,
                           size_t count, ncclDataType_t datatype,
                           ncclRedOp_t op, ncclComm_t comm,
                           cudaStream_t stream);
```

参数`sendbuff`是输入数据指针，`recvbuff`是输出数据指针。在实际工程中，这两个指针通常指向同一块显存地址，即原地操作（in-place）。`count`是元素数量，不是字节数。`datatype`指定数据类型，常用的有`ncclFloat32`、`ncclFloat16`等。`op`是归约操作类型，梯度同步一般用`ncclSum`，也就是把所有GPU上的对应元素相加。最后一个参数`stream`是CUDA流，用于异步执行。

我们来看一个实际的调用例子。假设有4个GPU，每个GPU上有一个长度为1024的float32梯度向量，需要做求和归约：

```c
float* d_grad;  // 设备端梯度指针，已在显存中分配
ncclComm_t comm; // 通信域，已在初始化时创建
cudaStream_t stream; // 当前CUDA流

// 执行AllReduce，原地操作
ncclAllReduce(d_grad, d_grad, 1024, ncclFloat32, ncclSum, comm, stream);

// 等待NCCL操作完成
cudaStreamSynchronize(stream);
```

执行完这个调用后，每个GPU上的`d_grad`数组都变成了4个GPU的梯度之和。然后你只需要把每个元素除以GPU数量（4）取平均，就可以得到平均梯度。这里有个工程细节：NCCL的AllReduce默认只做归约，不做平均。平均的操作需要你自己在后续处理，或者使用`ncclAvg`操作（需要NCCL 2.12以上版本）。

在性能方面，AllReduce的通信量跟数据量成正比。对于`N`个GPU，每个GPU有`M`个元素，AllReduce的总通信量约为`2 * (N-1) * M * sizeof(datatype)`字节。以4个GPU、每个GPU 1024个float32元素为例，总通信量大约是`2 * 3 * 1024 * 4 = 24KB`，非常小。但在大模型训练中，梯度动辄几百MB，这时候AllReduce的带宽利用率就成了关键瓶颈。

#### 信息

NCCL的AllReduce内部使用Ring算法或Tree算法。对于小数据量（比如小于128KB），Ring算法延迟更低；对于大数据量，Tree算法带宽利用率更高。NCCL会自动选择最优算法，一般不需要手动干预。

### 11.2 Broadcast：参数初始化与模型分发

Broadcast操作的作用是把一个GPU上的数据广播到所有其他GPU上。在分布式训练中，它常用于模型参数的初始化同步。比如，你只在GPU 0上初始化了模型参数，然后需要把这份参数广播到GPU 1、2、3上，保证所有GPU的起始参数一致。

`ncclBroadcast`的函数原型：

```c
ncclResult_t ncclBroadcast(const void* sendbuff, void* recvbuff,
                           size_t count, ncclDataType_t datatype,
                           int root, ncclComm_t comm,
                           cudaStream_t stream);
```

`root`参数指定了源GPU的rank编号。只有root GPU上的`sendbuff`是有效输入，其他GPU上的`sendbuff`会被忽略。所有GPU（包括root自身）都会把接收到的数据写入`recvbuff`。

来看一个实际用法。假设我们要把GPU 0上的一个大小为4096的float32权重矩阵广播到所有GPU：

```c
float* d_weight; // 每个GPU上都有这个指针，但只有GPU 0上填充了有效数据
int root = 0;

ncclBroadcast(d_weight, d_weight, 4096, ncclFloat32, root, comm, stream);
cudaStreamSynchronize(stream);
```

执行后，所有GPU上的`d_weight`数据都和GPU 0上的完全一致。这里同样支持原地操作，即`sendbuff`和`recvbuff`指向同一地址。

在工程实践中，Broadcast还有一个重要用途：在混合精度训练中，用于同步损失缩放因子（loss scaling factor）。这个因子通常在CPU上计算，然后通过Broadcast分发到所有GPU。由于数据量很小（一般就一个float32值），Broadcast的延迟非常低，几乎可以忽略不计。

Broadcast的通信量是`(N-1) * M * sizeof(datatype)`字节，比AllReduce少一半。但它的使用频率远低于AllReduce，通常只在训练开始或参数恢复时调用一次。

### 11.3 Reduce：单节点收集归约结果

Reduce操作和AllReduce的区别在于：Reduce只把归约结果放在一个指定的GPU上，其他GPU不保留结果。这个操作在分布式推理中比较有用，比如你需要把所有GPU上的推理结果汇总到一台机器上进行后处理。

函数原型：

```c
ncclResult_t ncclReduce(const void* sendbuff, void* recvbuff,
                        size_t count, ncclDataType_t datatype,
                        ncclRedOp_t op, int root,
                        ncclComm_t comm, cudaStream_t stream);
```

`root`参数指定接收结果的GPU。只有root GPU上的`recvbuff`会被写入有效数据，其他GPU上的`recvbuff`不会被修改。

举个例子：假设有4个GPU，每个GPU上有一个长度为512的int32数组，你想把所有数组的元素求和后放到GPU 2上：

```c
int* d_data; // 每个GPU上都有输入数据
int* d_result; // 只在GPU 2上需要分配
int root = 2;

ncclReduce(d_data, d_result, 512, ncclInt32, ncclSum, root, comm, stream);
cudaStreamSynchronize(stream);
```

执行后，GPU 2上的`d_result`数组就是4个GPU数据的逐元素和。其他GPU上的`d_result`内容未定义。

在工程中，Reduce的一个典型场景是分布式评估（evaluation）。在验证集上，每个GPU处理一部分数据，然后通过Reduce把每个batch的损失（loss）汇总到rank 0上，用于计算平均损失。由于只需要汇总一个标量值，通信开销极小。

#### 严重警告

使用Reduce时，非root GPU上的`recvbuff`虽然不会被写入，但NCCL仍然会检查该指针是否有效（即非NULL）。如果你在非root GPU上传递了NULL指针，会导致段错误。建议在所有GPU上都分配相同大小的`recvbuff`，或者至少传入一个合法的设备指针。

### 11.4 AllGather：全量收集与模型并行

AllGather操作会把每个GPU上的数据片段收集起来，拼成一个完整的数据集，然后分发到所有GPU上。在模型并行或张量并行中，这个操作非常常用。比如，你把一个大的权重矩阵按行切分到4个GPU上，前向传播时需要完整的矩阵，就可以用AllGather来收集。

函数原型：

```c
ncclResult_t ncclAllGather(const void* sendbuff, void* recvbuff,
                           size_t sendcount, ncclDataType_t datatype,
                           ncclComm_t comm, cudaStream_t stream);
```

注意这里的参数：`sendcount`是每个GPU发送的元素数量，不是总数量。`recvbuff`的大小必须是`sendcount * nranks`个元素。每个GPU上的`sendbuff`数据会被收集到`recvbuff`中，按照rank顺序排列。

来看一个张量并行中的实际例子。假设有4个GPU，每个GPU上有一个大小为`[256, 128]`的矩阵分片（按行切分），现在需要收集所有分片得到完整的`[1024, 128]`矩阵：

```c
float* d_shard;   // 每个GPU上自己的分片，大小 256*128 = 32768个float
float* d_full;    // 每个GPU上分配的完整矩阵，大小 1024*128 = 131072个float
size_t sendcount = 32768; // 每个GPU发送的元素数

ncclAllGather(d_shard, d_full, sendcount, ncclFloat32, comm, stream);
cudaStreamSynchronize(stream);
```

执行后，每个GPU上的`d_full`数组都包含了所有4个分片，按rank 0、1、2、3的顺序排列。这样每个GPU都有了完整的矩阵数据，可以独立进行矩阵乘法。

AllGather的通信量是`(N-1) * M * sizeof(datatype)`字节，其中`M = sendcount`。注意这里的`M`是每个GPU发送的数据量，总数据量是`N * M`。与AllReduce相比，AllGather的通信量相同，但计算量更少（没有归约操作）。

在工程中，AllGather常用于LayerNorm或Softmax的前向计算，这些操作需要全局的统计量（比如均值、方差），而每个GPU只持有部分数据。

### 11.5 ReduceScatter：梯度压缩的前置操作

ReduceScatter是Reduce和Scatter的组合操作。它先把所有GPU上的数据进行归约，然后把归约结果的不同部分分发到不同的GPU上。每个GPU只得到完整归约结果的一个分片。这个操作在梯度压缩和混合精度训练中特别有用。

函数原型：

```c
ncclResult_t ncclReduceScatter(const void* sendbuff, void* recvbuff,
                               size_t recvcount, ncclDataType_t datatype,
                               ncclRedOp_t op, ncclComm_t comm,
                               cudaStream_t stream);
```

`recvcount`是每个GPU接收的元素数量。`sendbuff`的大小必须是`recvcount * nranks`个元素。每个GPU上的`sendbuff`包含了所有分片的数据，归约后，每个GPU只得到对应分片的结果。

举个例子：4个GPU，每个GPU上有一个长度为4096的float32梯度向量。你想先做AllReduce，但发现通信量太大。改用ReduceScatter，每个GPU只处理1/4的梯度：

```c
float* d_grad;    // 每个GPU上完整的梯度，大小4096
float* d_reduced; // 每个GPU上只接收1024个元素
size_t recvcount = 1024; // 4096 / 4

ncclReduceScatter(d_grad, d_reduced, recvcount, ncclFloat32, ncclSum, comm, stream);
cudaStreamSynchronize(stream);
```

执行后，rank 0上的`d_reduced`包含了所有GPU梯度前1024个元素的和，rank 1上的包含了第1024~2047个元素的和，以此类推。这样每个GPU只需要处理1/4的数据量，后续的梯度压缩或本地更新就可以并行进行。

ReduceScatter的通信量和AllReduce相同，都是`2 * (N-1) * M`字节（这里`M = recvcount`）。但它的优势在于减少了每个GPU后续的计算量和内存访问量。在ZeRO优化器中，ReduceScatter被广泛用于梯度分片。

#### 设计要点

ReduceScatter通常和AllGather配对使用。先做ReduceScatter归约并分片，本地更新梯度后，再用AllGather收集完整的更新后梯度。这个模式在ZeRO-2和ZeRO-3中都有体现。

### 11.6 Send/Recv：点对点通信的基石

虽然NCCL以集合通信闻名，但它也提供了点对点的Send和Recv接口。这些接口在自定义通信模式或实现Pipeline并行时非常有用。比如，在Pipeline并行中，前一个stage的GPU需要把中间激活值发送给下一个stage的GPU，这就是典型的点对点通信。

函数原型：

```c
ncclResult_t ncclSend(const void* sendbuff, size_t count,
                      ncclDataType_t datatype, int peer,
                      ncclComm_t comm, cudaStream_t stream);

ncclResult_t ncclRecv(void* recvbuff, size_t count,
                      ncclDataType_t datatype, int peer,
                      ncclComm_t comm, cudaStream_t stream);
```

`peer`参数指定了对端GPU的rank编号。Send和Recv必须配对使用，否则会导致死锁。NCCL的Send/Recv是异步的，但底层会保证数据的一致性。

来看一个Pipeline并行中的典型用法。假设rank 0需要把大小为`[1024, 512]`的激活矩阵发送给rank 1：

```c
// rank 0: 发送
float* d_activation; // 大小 1024*512 = 524288个float
ncclSend(d_activation, 524288, ncclFloat32, 1, comm, stream);

// rank 1: 接收
float* d_recv; // 同样大小
ncclRecv(d_recv, 524288, ncclFloat32, 0, comm, stream);

// 等待完成
cudaStreamSynchronize(stream);
```

在工程中，使用Send/Recv时要特别注意配对顺序。如果两个GPU同时调用Send（或者同时调用Recv），就会死锁。常见的做法是让rank小的先Send，rank大的先Recv，或者使用`ncclGroupStart/End`来批量提交操作。

Send/Recv的通信量就是数据本身的大小，没有额外的归约开销。在Pipeline并行中，通信量通常远小于计算量，因此延迟不是主要瓶颈，带宽利用率更重要。

### 11.7 集合操作的同步与异步行为

NCCL的所有集合操作都是异步的，这意味着函数调用会立即返回，但实际的通信操作可能还没有开始。异步行为通过CUDA流来管理：NCCL操作会被提交到指定的CUDA流中，与同一个流中的其他CUDA操作（比如kernel计算）保持顺序执行。

理解这一点非常重要。来看一个常见的错误用法：

```c
// 错误示例：没有同步就使用结果
ncclAllReduce(d_grad, d_grad, 1024, ncclFloat32, ncclSum, comm, stream);
// 此时d_grad可能还没有被写入，直接使用会导致错误
launch_kernel(d_grad); // 这个kernel可能读到未更新的数据
```

正确的做法是确保NCCL操作完成后再使用数据。有两种方式：

| 同步方式 | 方法 | 适用场景 | 延迟影响 |
| --- | --- | --- | --- |
| 流同步 | `cudaStreamSynchronize(stream)` | CPU需要等待结果 | 高（阻塞CPU） |
| 事件同步 | `cudaEventRecord(event, stream)` `cudaEventSynchronize(event)` | CPU需要等待，但允许其他流继续 | 中 |
| 依赖同一个流 | 后续操作提交到同一个stream | GPU操作之间依赖 | 低（无CPU等待） |

在分布式训练框架中，通常采用第三种方式：把NCCL操作和后续的计算kernel提交到同一个CUDA流中，这样GPU硬件会自动保证执行顺序，不需要CPU介入同步。这是最高效的做法。

#### 严重警告

不要在不同的CUDA流中同时提交NCCL操作和依赖这些操作结果的kernel，除非你显式使用了事件同步或流间同步机制。否则会导致数据竞争，而且这种错误很难调试，因为结果可能偶尔正确。

### 11.8 分组操作与ncclGroupStart/End

当需要同时发起多个集合操作时（比如多个AllReduce在不同通信域上执行），可以使用`ncclGroupStart`和`ncclGroupEnd`来批量提交。这个机制可以让NCCL内部的调度器更好地优化通信顺序，减少总延迟。

函数原型：

```c
ncclResult_t ncclGroupStart();
ncclResult_t ncclGroupEnd();
```

在`ncclGroupStart`和`ncclGroupEnd`之间，你可以调用任意多个NCCL集合操作。这些操作不会立即执行，而是被缓存起来。当调用`ncclGroupEnd`时，NCCL会一起提交所有操作，并可能进行优化。

来看一个实际场景：在模型并行+数据并行的混合并行中，你可能需要同时更新多个通信域上的梯度。比如，数据并行域上的AllReduce和张量并行域上的AllReduce可以同时发起：

```c
ncclComm_t data_comm;  // 数据并行通信域
ncclComm_t model_comm; // 模型并行通信域

ncclGroupStart();
ncclAllReduce(d_grad_data, d_grad_data, 1024, ncclFloat32, ncclSum, data_comm, stream);
ncclAllReduce(d_grad_model, d_grad_model, 512, ncclFloat32, ncclSum, model_comm, stream);
ncclGroupEnd();
```

使用分组操作的好处：第一，减少API调用开销；第二，NCCL可以重叠两个操作的通信过程，提高带宽利用率；第三，避免多个操作之间的调度冲突。

在工程中，建议把所有不依赖彼此结果的集合操作都放在同一个分组中。但如果操作之间有依赖关系（比如一个操作的输出是另一个操作的输入），则不能放在同一分组中，必须按顺序提交。

### 11.9 错误处理与返回值检查

NCCL API的返回值类型是`ncclResult_t`，它是一个枚举类型，表示操作的结果状态。在实际工程中，绝对不能忽略返回值检查。很多线上故障都是因为NCCL调用失败但没有被捕获，导致后续计算使用了错误的数据。

常见的返回值及其含义：

| 返回值 | 含义 | 常见原因 |
| --- | --- | --- |
| `ncclSuccess` | 操作成功 | — |
| `ncclUnhandledCudaError` | CUDA错误未被处理 | 显存不足、设备重置 |
| `ncclSystemError` | 系统级错误 | 网络断开、PCIe错误 |
| `ncclInternalError` | NCCL内部错误 | 算法选择失败、资源耗尽 |
| `ncclInvalidArgument` | 参数无效 | count为0、datatype不支持 |
| `ncclInvalidUsage` | 使用方式错误 | 在分组外调用、重复初始化 |

推荐的错误处理模式是定义一个宏，在每次NCCL调用后检查返回值：

```c
#define NCCL_CHECK(call) do { \
    ncclResult_t res = call; \
    if (res != ncclSuccess) { \
        fprintf(stderr, "NCCL error %d at %s:%d: %s\n", \
                res, __FILE__, __LINE__, ncclGetErrorString(res)); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

// 使用示例
NCCL_CHECK(ncclAllReduce(d_grad, d_grad, 1024, ncclFloat32, ncclSum, comm, stream));
```

注意，`ncclGetErrorString`函数可以把错误码转换成可读的字符串，方便调试。在线上环境中，建议把错误信息记录到日志中，而不是直接退出。但对于训练任务，NCCL错误通常意味着整个训练无法继续，退出是合理的选择。

### 11.10 性能调优：选择合适的数据类型与操作

NCCL集合操作的性能受多个因素影响，其中数据类型和归约操作的选择是关键。不同的数据类型占用的带宽不同，计算单元的吞吐也不同。

以下是常用数据类型的带宽占用对比（假设总元素数量相同）：

| 数据类型 | 字节数 | 带宽占用 | 适用场景 |
| --- | --- | --- | --- |
| `ncclFloat32` | 4 | 基准 | 标准训练，精度要求高 |
| `ncclFloat16` | 2 | 50% | 混合精度训练，梯度压缩 |
| `ncclBfloat16` | 2 | 50% | BF16训练，动态范围大 |
| `ncclInt32` | 4 | 基准 | 索引、标签同步 |
| `ncclInt8` | 1 | 25% | 量化训练，梯度量化 |

在工程实践中，如果模型使用FP32训练，梯度同步用`ncclFloat32`。如果使用混合精度（AMP），梯度通常也是FP32，所以数据类型不变。但在一些梯度压缩方案中，可以把梯度量化成FP16甚至INT8后再做AllReduce，通信量直接减半或减到1/4。

归约操作的选择也很重要：

- `ncclSum`：最常用，用于梯度求和。
- `ncclAvg`：NCCL 2.12+支持，直接求平均，但性能略低于`ncclSum`（因为内部需要额外除法）。
- `ncclMax`/`ncclMin`：用于取最大值/最小值，在分布式评估中计算准确率时有用。
- `ncclProd`：用于求乘积，使用较少。

性能调优的一个关键点是：尽量让每次集合操作的数据量足够大，以充分利用带宽。如果数据量太小（比如小于1KB），通信延迟会占主导，带宽利用率很低。在工程中，可以把多个小张量合并成一个大的连续张量，再做一次AllReduce，而不是分别做多次小操作。

#### 设计要点

在PyTorch DDP中，梯度桶（gradient bucket）机制就是基于这个原理：把多个小梯度合并成一个大的连续缓冲区，然后一次性调用AllReduce。这可以显著提高通信效率，减少API调用次数。

#### 章节小结

这一章我们详细拆解了NCCL的集合操作API，从最核心的AllReduce开始，到Broadcast、Reduce、AllGather、ReduceScatter，再到点对点的Send/Recv。每个API都给出了实际代码示例和参数说明，并重点强调了工程中的注意事项，比如原地操作、同步方式、错误处理、分组提交等。理解这些API的底层行为，是写出高效、稳定的分布式训练代码的基础。

#### 本章重点回顾

- AllReduce是梯度同步的核心操作，使用`ncclSum`进行求和归约，平均操作需要手动处理或使用`ncclAvg`。
- Broadcast用于参数初始化同步，Reduce用于单节点收集结果，AllGather用于全量数据收集。
- ReduceScatter是梯度分片的前置操作，通常与AllGather配对使用，实现ZeRO优化器。
- Send/Recv提供点对点通信能力，必须配对使用，避免死锁。
- 所有NCCL操作都是异步的，通过CUDA流管理执行顺序，推荐使用同一流来避免显式同步。
- 使用`ncclGroupStart/End`批量提交独立操作，可以优化通信调度，提高带宽利用率。
- 必须检查NCCL API的返回值，使用宏封装可以简化错误处理。
- 选择合适的数据类型和归约操作，合并小张量成大缓冲区，可以显著提升通信性能。
