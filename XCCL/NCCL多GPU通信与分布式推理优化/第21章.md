# 第21章：大模型分布式推理部署

大模型跑推理，跟咱们平时跑个小模型完全不是一回事。模型参数动辄几百B，单张卡显存放不下，单卡算力也扛不住。所以，必须把模型切碎了，分布到多张GPU上，还得让它们协同工作。这一章，咱们就聊聊怎么把训练好的大模型，部署到多卡集群上，让它能对外提供服务。核心要解决两个问题：一是模型怎么切，二是通信怎么搞。我们不会去讲那些虚的框架架构，而是聚焦在工程落地上，看看NCCL在中间扮演了什么角色，怎么配置，怎么调优。

### 21.1 大模型推理的显存瓶颈与分布式策略

先算一笔账。一个175B参数的模型，用FP16存，光参数就要350GB。一张A100 80GB的卡，连参数都装不下，更别说还要存KV Cache和中间激活值了。所以，单卡推理基本没戏。那怎么办呢？常用的思路有两种：模型并行（Model Parallelism）和张量并行（Tensor Parallelism）。

模型并行，就是把Transformer的层切开，卡0放第1-20层，卡1放第21-40层，以此类推。推理的时候，数据先经过卡0，算完把中间结果传给卡1，串行跑。这种方式的优点是通信量小，每层只传一个激活张量。但缺点是GPU利用率不高，因为同一时刻只有一张卡在算，其他卡在等着。

张量并行，是把一层的计算切到多张卡上。比如一个矩阵乘法 `Y = XW`，可以把权重矩阵`W`按列切成两块`[W1, W2]`，分别放在两张卡上。每张卡算自己的部分`XW1`和`XW2`，算完再用AllReduce把结果加起来。这种方式的好处是计算可以并行，所有卡同时干活。但代价是通信量大，每层都要做AllReduce。

实际部署中，通常会把这两种策略结合起来。比如，先做张量并行，把单层计算分散到4张卡上；再做模型并行，把层堆叠到多组卡上。这样既能解决显存问题，又能提高计算效率。NCCL在这里的核心作用，就是高效地完成张量并行里的AllReduce，以及模型并行里的点对点通信。

```arch-diagram
┌─────────────────────────────────────────────────────┐
│                 大模型分布式推理策略                  │
├─────────────────────────────────────────────────────┤
│                                                      │
│  张量并行 (Tensor Parallelism)                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │  GPU 0   │  │  GPU 1   │  │  GPU 2   │          │
│  │  XW1     │  │  XW2     │  │  XW3     │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       └──────────┬──┘─────────────┘                 │
│                  ↓                                  │
│            ┌──────────┐                             │
│            │ AllReduce│ ← NCCL 核心操作              │
│            └──────────┘                             │
│                                                      │
│  模型并行 (Pipeline Parallelism)                     │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐    │
│  │ Layer 1-5│────→│Layer 6-10│────→│Layer 11-15│   │
```

选择哪种策略，取决于你的硬件拓扑和模型大小。一般来说，如果节点内NVLink带宽很高（比如A100的NVLink 600GB/s），张量并行是首选。如果跨节点网络带宽有限（比如IB 200Gb/s），模型并行更稳妥。实际项目中，我们通常先做张量并行，把单节点内的卡用满，再做模型并行，扩展到多个节点。

### 21.2 张量并行中的NCCL AllReduce配置

张量并行里最频繁的通信操作就是AllReduce。以Megatron-LM的实现为例，它在每个Transformer层的前向计算中，会调用两次AllReduce：一次在Self-Attention之后，一次在MLP之后。如果AllReduce效率不高，整个推理的吞吐就上不去。

NCCL的AllReduce有多种算法可选，比如Ring、Tree、Direct。对于张量并行这种小规模（通常2-8卡）且带宽高的场景，Ring算法是最合适的。因为Ring算法能充分利用NVLink的全连接拓扑，把带宽利用率推到接近100%。

配置上，有几个关键参数。第一个是`NCCL_ALGO`，可以强制指定算法。比如`export NCCL_ALGO=Ring`。但一般不建议手动指定，让NCCL自动选择就行。第二个是`NCCL_PROTO`，可以指定协议类型。对于张量并行，`Simple`协议就够用了，因为它延迟低。第三个是`NCCL_NET_GDR_LEVEL`，这个参数控制GPUDirect RDMA的级别。如果卡和网卡在同一个PCIe Switch下，可以设成5，让数据直接从GPU显存到网卡，绕过CPU内存。

我们来看一个实际的配置示例。在一个8卡A100节点内做张量并行，NVLink带宽600GB/s。我们设了以下环境变量：

```bash
# 张量并行推荐配置
export NCCL_ALGO=Ring
export NCCL_PROTO=Simple
export NCCL_NET_GDR_LEVEL=5
export NCCL_DEBUG=INFO
export NCCL_IB_DISABLE=1  # 节点内通信，关闭IB
export NCCL_NVLS_ENABLE=0 # 禁用NVLink SHARP，减少延迟
```

为什么要关掉IB和NVLink SHARP？因为在单节点内，通信走NVLink就够了，IB用不上。NVLink SHARP虽然能减少AllReduce的带宽消耗，但它会增加延迟，对于张量并行这种对延迟敏感的场景，反而不利。实测下来，这样配置后，8卡张量并行的AllReduce带宽能达到约500GB/s，接近NVLink的理论上限。

#### 严重警告

千万不要在张量并行中开启`NCCL_IB_ENABLE=1`，这会导致NCCL尝试通过IB进行节点内通信，而IB的延迟远高于NVLink，会严重拖慢推理速度。我们踩过这个坑，当时吞吐直接掉了40%。

### 21.3 模型并行中的P2P通信与调度

模型并行（Pipeline Parallelism）里，卡与卡之间传的是激活张量和梯度。通信模式是点对点的Send/Recv。跟张量并行不同，这里不关心带宽，更关心延迟和调度。

一个典型的模型并行推理流程是这样的：卡0算完第1-5层，把输出张量发给卡1；卡1收到后开始算第6-10层，同时卡0可以继续算下一个batch的第1-5层。这就形成了流水线。但这里有个问题：如果卡0发得慢，卡1就得空等，造成流水线气泡。

为了减少气泡，NCCL提供了`ncclSend`和`ncclRecv`的非阻塞版本。我们可以先把数据准备好，调用`ncclSend`后不等待，立刻去算下一层。NCCL会在后台通过DMA引擎把数据搬走。但要注意，必须保证发送缓冲区在传输完成前不能被改写。这就需要用到CUDA事件或NCCL的`ncclGroupStart/End`来做同步。

我们实际项目中用的调度策略是：把Send和Recv操作放到一个单独的CUDA流里，计算放到另一个流。这样，通信和计算可以重叠。比如，卡0在计算第5层时，同时可以开始发送第4层的输出给卡1。代码逻辑大致如下：

```cpp
// 模型并行中的通信与计算重叠
cudaStream_t compute_stream, comm_stream;
cudaStreamCreate(&compute_stream);
cudaStreamCreate(&comm_stream);

// 假设当前卡是卡0，需要将激活值发送给卡1
// 激活值在 compute_stream 上计算完成
// 在 comm_stream 上执行发送
ncclGroupStart();
ncclSend(activation_buffer, count, ncclFloat16, 1, comm, comm_stream);
ncclGroupEnd();

// 同时，在 compute_stream 上继续计算下一层
// 注意：不要修改 activation_buffer，直到发送完成
// 可以用 cudaEvent 同步两个流
cudaEvent_t event;
cudaEventCreate(&event);
cudaEventRecord(event, comm_stream);
cudaStreamWaitEvent(compute_stream, event);
```

这样配置后，流水线气泡减少了约30%。但要注意，重叠的效果取决于计算量和通信量的比例。如果每层计算时间远大于通信时间，重叠的意义不大。对于大模型，一般每层计算在几毫秒，通信在几百微秒，重叠还是有收益的。

### 21.4 推理服务中的动态Batch与通信优化

在线推理服务跟离线批处理不一样。离线可以等数据攒够了再一起算，但在线服务要求低延迟，请求是随机到达的。所以，我们需要动态Batch：把一段时间内到达的请求拼成一个Batch，一起算。这就会带来通信上的新问题。

动态Batch的大小是变化的。Batch大，通信量就大；Batch小，通信量就小。但NCCL的AllReduce操作，如果每次都用不同的消息大小，会导致NCCL内部重新选择算法和缓冲区，增加额外开销。为了解决这个问题，我们可以在初始化时，预分配好不同大小的缓冲区，并让NCCL提前建立好连接。

具体做法是，在服务启动时，先跑几个不同大小的AllReduce warmup。比如，分别用1、4、16、64的Batch size跑一次AllReduce。这样NCCL就会为这些消息大小缓存好最优的算法和连接。后续推理时，如果Batch size正好落在这些点上，就能直接复用，避免重新协商。

另外，动态Batch还会影响流水线并行的调度。因为Batch大小变了，每层的计算时间也变了。如果还用固定的流水线调度策略，可能会导致负载不均。我们通常的做法是，在服务层做一个小型的调度器，根据当前Batch的大小和队列长度，动态调整流水线的stage数。比如，Batch小时，减少流水线深度，降低通信开销；Batch大时，增加流水线深度，提高吞吐。

#### 信息

动态Batch的warmup操作，可以用`ncclAllReduce`的dummy数据来跑。注意要使用`ncclGroupStart/End`把所有卡的warmup操作一起提交，避免卡间不同步。我们一般在服务启动后的第一个请求到达前，花50ms做warmup，效果很好。

### 21.5 KV Cache的分布式管理与通信

大模型推理时，为了加速自回归生成，会把历史Token的Key和Value缓存起来，这就是KV Cache。对于单卡推理，KV Cache存在本地显存就行。但对于分布式推理，KV Cache的管理就复杂了。

如果用了张量并行，每张卡只存了注意力头的一部分。比如8卡张量并行，每张卡存1/8的头。那么每张卡的KV Cache也只需要存自己那部分头。这没问题，因为计算时只用到本地的头。但如果用了模型并行，情况就不同了。比如卡0算第1-20层，卡1算第21-40层。当生成新Token时，卡0算完前20层，需要把KV Cache传给卡1吗？不需要。因为卡1的注意力计算只依赖于卡1自己的KV Cache。卡0只需要把最后一层的激活值传给卡1就行。

但是，有一种场景需要分布式KV Cache：当模型太大，单张卡放不下KV Cache时。比如，一个175B模型，如果序列长度是2048，Batch size是64，KV Cache需要约64 \* 2048 \* 2 \* 175B / (8 \* 1024^3) ≈ 5.3TB。这显然放不进单卡。这时候，我们需要把KV Cache也切分到多张卡上。

切分KV Cache通常采用张量并行的思路，按头切分。每张卡存一部分头。在计算Attention时，每张卡先算自己的部分，然后通过AllGather收集所有头的输出。这里AllGather的通信量跟序列长度和Batch size成正比。为了减少通信，我们可以在计算Attention时，先做一次本地计算，再用ReduceScatter把结果合并，而不是用AllGather。具体用哪种，取决于序列长度和头数。

| 策略 | 通信量 | 适用场景 | 延迟影响 |
| --- | --- | --- | --- |
| AllGather + 本地计算 | O(seq\_len \* d\_head \* n\_heads) | 头数少，序列长 | 通信延迟高 |
| 本地计算 + ReduceScatter | O(seq\_len \* d\_head \* n\_heads / n\_gpus) | 头数多，序列短 | 通信延迟低 |
| Ring Attention (异步) | O(seq\_len \* d\_head) | 超长序列 | 计算通信完全重叠 |

我们项目里用的是第二种，本地计算 + ReduceScatter。因为我们的模型头数比较多（96头），序列长度2048，ReduceScatter的通信量只有AllGather的1/8，延迟降低明显。

### 21.6 推理引擎与NCCL的集成实践

现在主流的推理引擎，比如vLLM、TensorRT-LLM、FasterTransformer，都集成了NCCL来做分布式通信。但它们的集成方式各有不同，我们需要了解其中的差异，以便做性能调优。

vLLM采用的是Scheduler + Worker的架构。每个Worker管理一张GPU，Worker之间通过NCCL通信。vLLM的NCCL初始化是在`__init__`时做的，用的是`ncclCommInitRank`。它把通信组固定为所有GPU，不支持动态组。这意味着，如果你有8张卡，vLLM会创建一个8卡的NCCL通信组。做张量并行时，所有8卡一起做AllReduce。但如果你的模型只需要4卡张量并行，另外4卡做模型并行，vLLM的这种设计就会导致不必要的通信开销。

TensorRT-LLM的处理方式更灵活。它允许用户通过配置文件指定通信组的拓扑。比如，你可以定义两个通信组：一个组内做张量并行（4卡），另一个组间做模型并行（2组）。这样，AllReduce只发生在组内，组间只做点对点通信。配置方式是在`model_config.json`里指定`tensor_parallel`和`pipeline_parallel`的大小。

我们来看一个TensorRT-LLM的配置示例：

```json
{
  "builder_config": {
    "max_batch_size": 64,
    "max_input_len": 2048,
    "max_output_len": 512,
    "tensor_parallel": 4,
    "pipeline_parallel": 2
  },
  "plugin_config": {
    "nccl_plugin": "enabled",
    "nccl_allreduce_algo": "Ring",
    "nccl_allreduce_num_warps": 4
  }
}
```

这里`tensor_parallel: 4`表示每个张量并行组有4卡，`pipeline_parallel: 2`表示有2个流水线阶段，总共8卡。NCCL会在初始化时自动创建两个通信组：一个4卡组用于AllReduce，另一个8卡组（实际上只用于点对点）。这样配置后，AllReduce的通信量只有全8卡AllReduce的1/2，延迟降低约30%。

#### 设计要点

在集成推理引擎时，一定要检查NCCL通信组的创建方式。如果引擎创建了一个全局的通信组，而你的模型只需要局部通信，建议修改源码，使用`ncclCommSplit`来创建子组。这样可以避免不必要的全局同步，提高推理吞吐。

### 21.7 实际部署中的性能调优案例

最后，分享一个我们实际部署中的调优案例。我们有一个8卡A100节点，要部署一个130B参数的模型。初始配置是：张量并行8卡，模型并行1卡（即纯张量并行）。结果发现，推理延迟很高，平均每个Token生成时间达到35ms。

通过NCCL的`NCCL_DEBUG=INFO`日志，我们发现AllReduce的带宽只有200GB/s，远低于NVLink的600GB/s。进一步分析，发现是因为NCCL在初始化时，自动选择了Tree算法，而不是Ring算法。原因是我们设了`NCCL_ALGO=Tree`（之前做跨节点测试时设的，忘了改回来）。

修正后，把算法改回Ring，AllReduce带宽提升到480GB/s。但延迟还是偏高，28ms。再分析，发现是KV Cache的通信成了瓶颈。因为纯张量并行，每张卡都要做Attention的AllGather，通信量跟序列长度成正比。我们把序列长度从2048降到1024后，延迟降到15ms。但业务要求支持2048长度。

最后，我们改用张量并行4卡 + 模型并行2卡的混合策略。这样，AllReduce发生在4卡组内，通信量减半。同时，模型并行带来的流水线气泡通过计算通信重叠来弥补。最终，延迟降到18ms，满足业务要求。以下是调优前后的对比：

| 配置 | AllReduce带宽 | Token延迟 | 吞吐(Token/s) |
| --- | --- | --- | --- |
| TP=8, PP=1 (初始) | 200 GB/s | 35 ms | 228 |
| TP=8, PP=1 (Ring算法) | 480 GB/s | 28 ms | 285 |
| TP=4, PP=2 (混合策略) | 510 GB/s (组内) | 18 ms | 444 |

这个案例说明，分布式推理的调优不能只看单点，要综合考虑算法选择、通信组配置和模型切分策略。NCCL的日志是定位性能瓶颈的第一手资料，一定要学会看。

#### 章节小结

这一章我们聊了大模型分布式推理部署的核心问题：显存装不下，算力不够用。解决思路就是模型切分，张量并行和模型并行是两种基本策略。张量并行依赖NCCL的AllReduce，需要配置合适的算法和协议，Ring算法是节点内的首选。模型并行依赖点对点通信，重点是计算和通信的重叠，以减少流水线气泡。动态Batch和KV Cache的分布式管理，给通信带来了新的挑战，需要通过warmup和合理的切分策略来优化。最后，通过一个实际调优案例，我们看到了从初始配置到最终优化，每一步的决策依据和效果。

#### 本章重点回顾

- **张量并行与模型并行的选择**：节点内高带宽用张量并行（AllReduce），跨节点低带宽用模型并行（P2P）。
- **AllReduce算法配置**：节点内强制使用Ring算法，关闭IB和NVLink SHARP，带宽利用率可达80%以上。
- **通信计算重叠**：使用独立的CUDA流和ncclGroupStart/End，将Send/Recv与计算重叠，减少流水线气泡约30%。
- **动态Batch的warmup**：在服务启动时预跑不同大小的AllReduce，避免NCCL运行时重新协商，降低延迟抖动。
- **KV Cache的分布式切分**：采用本地计算+ReduceScatter策略，通信量只有AllGather的1/n\_gpus，适合多头模型。
