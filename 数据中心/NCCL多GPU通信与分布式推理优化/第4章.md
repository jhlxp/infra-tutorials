# 第4章：NVLink与PCIe带宽对比

分布式训练中，GPU之间的数据传输速度直接决定了整体效率。很多同学刚开始接触多卡通信时，最常问的一个问题就是：NVLink和PCIe到底差多少？为什么大家都在追求NVLink？这一章我们就来彻底搞清楚这两个通信通道的工作原理、带宽差异和实际使用场景。我们会从物理层开始讲起，逐步深入到协议栈，最后给出实际工程中的选型建议。通过这一章的学习，你会明白在什么场景下用NVLink，什么场景下用PCIe，以及如何让它们协同工作。

### 4.1 物理层对比：铜线 vs 差分对

我们先从最基础的物理层看起。PCIe和NVLink都是高速串行总线，但它们的物理实现差别很大。PCIe Gen4/Gen5使用的是标准的差分信号对，每一条lane由一对TX和一对RX组成，工作频率在16GHz到32GHz之间。NVLink则使用更短距离的铜线连接，通常直接焊在PCB板上，或者通过背板连接器，传输距离不超过几十厘米。

来看一下具体的参数对比：

| 参数 | PCIe Gen4 x16 | PCIe Gen5 x16 | NVLink 2.0 (V100) | NVLink 3.0 (A100) |
| --- | --- | --- | --- | --- |
| 单lane速率 | 16 GT/s | 32 GT/s | 20 GT/s | 50 GT/s |
| 总lane数 | 16 | 16 | 6 | 12 |
| 单向带宽 | 31.5 GB/s | 63 GB/s | 150 GB/s | 600 GB/s |
| 双向带宽 | 63 GB/s | 126 GB/s | 300 GB/s | 1200 GB/s |
| 传输距离 | 可达30cm | 可达30cm | 5-10cm | 5-10cm |
| 功耗 | 约10W | 约15W | 约8W | 约12W |

从表格里可以看到，NVLink 3.0的单向带宽达到了600GB/s，是PCIe Gen5 x16的将近10倍。这个差距是怎么来的？一方面是NVLink的物理层优化，它使用了更短的传输路径和更低的信号衰减；另一方面是NVLink的lane数更多，而且每个lane的速率也更高。

```arch-diagram

│                    GPU到GPU的物理连接                        │
│                                                             │
│  ┌─────────┐    PCIe Gen5 x16    ┌─────────┐               │
│  │  GPU 0  │◄───────────────────►│  GPU 1  │               │
│  │         │    (63 GB/s)        │         │               │
│  └────┬────┘                     └────┬────┘               │
│       │                               │                    │
│       │   NVLink 3.0 (600 GB/s)       │                    │
│       └───────────────────────────────┘                    │
│                                                             │
│  注意：NVLink是直连，PCIe需要经过CPU/芯片组转发              │
└─────────────────────────────────────────────────────────────┘
```

图4-1：NVLink与PCIe的物理连接对比

在实际的服务器里，NVLink的走线非常短，通常就在同一个GPU模组内部。比如NVIDIA DGX A100服务器，8张A100 GPU通过NVSwitch全互联，每个GPU有12条NVLink lane，总带宽达到600GB/s。而PCIe需要经过CPU的PCIe控制器，再通过主板走线到另一个GPU，路径长，延迟大。

#### 信息

在NVIDIA的HGX基板上，NVLink走线长度控制在5cm以内，信号完整性非常好。而PCIe走线通常要经过主板、背板、甚至线缆，长度可能达到30cm以上，信号衰减严重，这是NVLink能跑更高频率的物理基础。

### 4.2 协议栈差异：轻量级 vs 通用型

物理层之外，协议栈的差异更大。PCIe是一个通用总线协议，它要兼容各种设备——显卡、网卡、SSD、采集卡等等。所以PCIe协议栈非常复杂，有事务层、数据链路层、物理层三层结构，每层都有复杂的错误检测、重传、流量控制机制。NVLink是专门为GPU到GPU通信设计的，协议栈非常精简。

我们来看一下NVLink的协议栈结构：

```arch-diagram
┌─────────────────────────────────────────────────────────────┐
│                   NVLink 协议栈                              │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  应用层：NCCL / CUDA Direct                        │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  传输层：原子操作 + 直接内存访问                    │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  链路层：简单的CRC校验 + 重传                      │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  物理层：差分信号 + 时钟数据恢复                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  对比PCIe：NVLink去掉了事务层的复杂状态机，减少了30%的       │
│  协议开销，延迟更低。                                       │
└─────────────────────────────────────────────────────────────┘
```

图4-2：NVLink协议栈结构

PCIe的事务层要处理各种类型的TLP（事务层包），包括内存读写、IO读写、配置读写、消息信号中断等等。NVLink不需要这些，它只做一件事：GPU之间的内存直接访问。NVLink支持三种基本操作：读取、写入、原子操作。原子操作特别重要，因为NCCL在做AllReduce时，需要用到原子加来合并梯度。

NVLink的链路层也很简单。它使用了一种轻量级的CRC校验，而不是PCIe那种复杂的LCRC（链路层CRC）+ ECRC（端到端CRC）双重校验。NVLink的重传机制是硬件自动完成的，延迟非常低，通常在几十纳秒级别。PCIe的重传需要软件参与，延迟会高一个数量级。

#### 设计要点

NVLink的原子操作是硬件实现的，不需要CPU参与。这意味着你可以直接在GPU上发起一个原子加操作到另一个GPU的显存，延迟只有几百纳秒。而PCIe要实现同样的功能，需要先通过CPU转发，延迟会到微秒级。

### 4.3 带宽实测方法：用NCCL测试工具

理论带宽是一回事，实际能跑多少是另一回事。NCCL自带了一个测试工具叫`nccl-tests`，可以用来测量实际的通信带宽。我们来看一下怎么用。

首先需要编译nccl-tests：

make CUDA_HOME=/usr/local/cuda NCCL_HOME=/usr/local/nccl

### 4.4 NVLink拓扑与带宽瓶颈

NVLink的拓扑结构直接影响实际带宽。在DGX系列服务器中，GPU通过NVSwitch全互联，任意两个GPU之间都有6条或12条NVLink lane。但在一些双卡或四卡工作站上，NVLink的拓扑可能不是全互联的。

拓扑不同，NCCL的通信算法也要调整。在全互联拓扑中，NCCL可以使用Ring算法，每个GPU只和相邻的两个GPU通信，数据在环上流转。在非全互联拓扑中，NCCL会使用Tree算法，或者结合PCIe做混合通信。

#### 设计要点

在DGX A100上，NCCL默认使用NVLink Ring算法，带宽利用率可以达到90%以上。但如果你的GPU之间只有部分NVLink连接，NCCL会自动降级到Tree算法，带宽利用率会降到60%左右。所以，了解你的拓扑结构很重要。

### 4.5 PCIe带宽的实际限制

PCIe的理论带宽听起来不错，但实际能跑到的带宽往往只有理论值的60%-70%。原因有几个：首先是协议开销，PCIe每个数据包都有头部开销，加上CRC校验和流量控制，实际有效数据比例大概在80%左右。其次是路径延迟，PCIe要经过CPU的根复合体（Root Complex），这会引入额外的延迟。

我们来看一下实际测试结果：

| 测试场景 | 理论带宽 | 实测带宽 | 利用率 |
| --- | --- | --- | --- |
| PCIe Gen4 x16 单向 | 31.5 GB/s | 22.8 GB/s | 72.4% |
| PCIe Gen4 x16 双向 | 63 GB/s | 41.2 GB/s | 65.4% |
| PCIe Gen5 x16 单向 | 63 GB/s | 45.1 GB/s | 71.6% |
| PCIe Gen5 x16 双向 | 126 GB/s | 82.3 GB/s | 65.3% |
| NVLink 3.0 单向 | 600 GB/s | 548 GB/s | 91.3% |
| NVLink 3.0 双向 | 1200 GB/s | 1120 GB/s | 93.3% |

从表格可以看到，NVLink的利用率明显高于PCIe。这是因为NVLink的协议开销小，而且不需要经过CPU转发。另外，PCIe的带宽是共享的，如果多个GPU同时通过PCIe通信，它们会竞争带宽。NVLink是点对点直连，每个链路都是独立的。

在实际的8卡服务器中，如果所有GPU都通过PCIe通信，每个GPU能分到的带宽只有理论值的1/8左右。比如PCIe Gen4 x16双向63GB/s，8个GPU共享，每个GPU只能拿到不到8GB/s。而NVLink每个GPU独享600GB/s，差距非常明显。

#### 严重警告

千万不要以为PCIe Gen5 x16的126GB/s双向带宽可以满足8卡通信需求。在实际的AllReduce操作中，如果使用PCIe，8个GPU的通信带宽会受限于PCIe交换机的总带宽，通常只有10-15GB/s。而NVLink可以达到500GB/s以上。这就是为什么大模型训练必须用NVLink。

### 4.6 混合通信策略：NVLink + PCIe协同工作

虽然NVLink很快，但并不是所有GPU之间都有NVLink连接。在一些大型集群中，GPU可能分布在不同的节点上，节点之间只能用InfiniBand或以太网。即使在同一节点内，如果GPU数量超过8个，也可能出现部分GPU之间没有NVLink的情况。

NCCL支持混合通信策略，它可以根据GPU之间的连接拓扑，自动选择最优的通信路径。来看一下NCCL的通信路径选择逻辑：

```arch-diagram
┌─────────────────────────────────────────────────────────────┐
│               NCCL 通信路径选择                              │
│                                                             │
│  发起通信请求                                               │
│       │                                                    │
│       ↓                                                    │
│  检查GPU之间是否有NVLink直连                                │
│       │                                                    │
│       ├── 有NVLink ──► 使用NVLink Ring算法                  │
│       │                                                    │
│       └── 无NVLink ──► 检查是否在同一PCIe交换机下           │
│                        │                                   │
│                        ├── 同一交换机 ──► 使用PCIe P2P     │
│                        │                                   │
│                        └── 不同交换机 ──► 使用CPU中转       │
│                                                             │
│  最终：NCCL会为每对GPU选择最优路径                           │
└─────────────────────────────────────────────────────────────┘
```

图4-5：NCCL通信路径选择流程

在代码层面，NCCL通过`ncclTopo`模块来获取GPU拓扑信息。这个模块会读取系统文件`/sys/class/nvlink/`和`/sys/bus/pci/devices/`，构建出一张完整的拓扑图。然后NCCL根据这张图，为每个通信操作选择最优路径。

一个典型的混合通信场景是这样的：8个GPU在一个节点内，通过NVSwitch全互联。节点之间通过InfiniBand连接。NCCL会在节点内部使用NVLink Ring算法，在节点之间使用InfiniBand Ring算法。这样既利用了NVLink的高带宽，又利用了InfiniBand的长距离传输能力。

#### 设计要点

在编写分布式训练代码时，可以通过设置环境变量`NCCL_NET_GDR_LEVEL`来控制是否使用GPU Direct RDMA。如果设置为5，NCCL会优先使用NVLink进行节点内通信，用InfiniBand进行节点间通信。这样可以最大化利用带宽。

### 4.7 延迟对比：微秒级 vs 纳秒级

除了带宽，延迟也是关键指标。NVLink的延迟非常低，因为它不需要经过CPU，也不需要复杂的协议栈。PCIe的延迟要高得多，因为数据要经过PCIe控制器、CPU内存控制器、系统总线等多个环节。

来看一下实测延迟数据：

| 操作类型 | NVLink 3.0 | PCIe Gen5 x16 | InfiniBand HDR |
| --- | --- | --- | --- |
| 小数据包（4字节） | 0.2 us | 1.8 us | 2.5 us |
| 中等数据包（1KB） | 0.5 us | 3.2 us | 4.1 us |
| 大数据包（1MB） | 2.1 us | 12.5 us | 15.3 us |
| 原子操作 | 0.3 us | 2.5 us | 不支持 |

从表格可以看出，NVLink的延迟比PCIe低了一个数量级。这对于小数据包的通信特别重要。在分布式训练中，虽然大部分通信是大数据包的梯度同步，但也有一些控制消息是小数据包。比如NCCL的同步屏障（Barrier）操作，就是小数据包通信。NVLink的低延迟可以显著加快同步速度。

另外，NVLink的原子操作延迟只有0.3微秒，这对于实现高效的梯度合并非常关键。NCCL的AllReduce算法中，有一个步骤是使用原子操作来合并部分结果。如果使用PCIe，原子操作的延迟会高很多，导致整体性能下降。

#### 信息

在实际测试中，NVLink的延迟主要由物理传输时间和链路层处理时间组成。物理传输时间取决于距离，NVLink的走线很短，所以这部分时间可以忽略。链路层处理时间主要是CRC校验和重传，NVLink的硬件实现非常高效，处理时间在100纳秒以内。

### 4.8 实际训练中的带宽选择建议

了解了NVLink和PCIe的差异之后，我们来看看在实际训练中怎么选择。这里给出几条实用的建议：

**第一条：优先使用NVLink**。只要GPU之间有NVLink连接，就一定要用NVLink。NCCL默认会优先使用NVLink，但如果你手动设置了`NCCL_ALGO`环境变量，可能会覆盖默认行为。建议不要手动设置，让NCCL自动选择。

**第二条：注意PCIe带宽共享**。如果某些GPU之间没有NVLink，需要通过PCIe通信，要特别注意PCIe带宽的共享问题。在一个PCIe交换机下挂载多个GPU时，这些GPU共享PCIe交换机的上行带宽。建议将通信频繁的GPU放在同一个NVLink域内。

**第三条：使用GPU Direct RDMA**。如果节点之间需要通过InfiniBand通信，一定要开启GPU Direct RDMA。这样数据可以直接从GPU显存传输到网卡，不需要经过CPU内存，可以节省一次PCIe传输。

来看一下不同场景下的推荐配置：

| 训练场景 | GPU数量 | 推荐通信方案 | 预期带宽 |
| --- | --- | --- | --- |
| 单机双卡 | 2 | NVLink直连 | 300 GB/s |
| 单机四卡 | 4 | NVLink全互联 | 600 GB/s |
| 单机八卡 | 8 | NVSwitch | 600 GB/s |
| 多机八卡 | 16 | NVSwitch + InfiniBand | 600 GB/s (节点内) + 200 Gb/s (节点间) |
| 无NVLink双卡 | 2 | PCIe P2P | 22 GB/s |

#### 严重警告

在购买服务器时，一定要确认GPU之间的NVLink拓扑。有些厂商为了降低成本，会在8卡服务器中只提供4条NVLink lane，而不是12条。这种配置下，NVLink的带宽只有200GB/s，比标准的600GB/s差很多。购买前一定要用`nvidia-smi topo -m`检查。

### 4.9 NCCL的自动路径选择机制

NCCL在初始化时会自动探测GPU之间的连接拓扑，然后为每个通信操作选择最优路径。这个机制是通过`ncclTopo`模块实现的。我们来看一下NCCL的拓扑探测过程：

```text
// NCCL拓扑探测的核心逻辑
void ncclTopoGetSystem() {
    // 1. 遍历所有GPU
    for (int i = 0; i < nGpus; i++) {
        // 2. 读取NVLink信息
        int nvlinkCount = getNvlinkCount(gpu[i]);
        // 3. 读取PCIe信息
        int pcieGen = getPcieGeneration(gpu[i]);
        int pcieLanes = getPcieLanes(gpu[i]);
        // 4. 记录到拓扑结构
        topo->gpus[i].nvlinkCount = nvlinkCount;
        topo->gpus[i].pcieGen = pcieGen;
        topo->gpus[i].pcieLanes = pcieLanes;
    }
    // 5. 构建连接矩阵
    for (int i = 0; i < nGpus; i++) {
        for (int j = i+1; j < nGpus; j++) {
            // 检查是否有NVLink直连
            if (hasNvlinkDirect(gpu[i], gpu[j])) {
                topo->links[i][j] = NVLINK;
            } else if (hasPcieP2P(gpu[i], gpu[j])) {
                topo->links[i][j] = PCIE_P2P;
            } else {
                topo->links[i][j] = CPU_ACCESS;
            }
        }
    }
}
```

NCCL根据拓扑信息，为不同的通信模式选择不同的算法：

- **Ring算法**：适用于NVLink全互联拓扑，每个GPU只和相邻两个GPU通信，带宽利用率高。
- **Tree算法**：适用于部分NVLink连接或PCIe通信，数据从根节点广播到所有子节点。
- **Direct算法**：适用于小数据包，直接点对点通信，不需要经过中间节点。

NCCL还会根据数据包大小动态调整算法。对于小数据包（小于256KB），使用Direct算法，因为Ring算法的启动开销太大。对于大数据包，使用Ring算法，因为带宽利用率更高。

#### 设计要点

你可以通过设置环境变量`NCCL_ALGO=Ring`来强制NCCL使用Ring算法，或者`NCCL_ALGO=Tree`强制使用Tree算法。但建议不要这样做，让NCCL自动选择效果最好。只有在调试性能问题时，才手动设置来对比不同算法的效果。

### 4.10 性能调优实战：从PCIe到NVLink的迁移

最后，我们来看一个实际的性能调优案例。假设你有一台服务器，原本使用PCIe进行GPU通信，现在升级到了NVLink。我们来看一下性能提升了多少。

先看PCIe下的性能：

```bash
# 使用PCIe通信的测试结果
./build/all_reduce_perf -b 128M -e 128M -g 8

# 输出：
#     size       time    algbw    busbw
#      (B)     (usec)  (GB/s)  (GB/s)
# 134217728   11250.0    11.9     8.9
```

可以看到，8个GPU通过PCIe做AllReduce，带宽只有8.9GB/s。这是因为8个GPU共享PCIe带宽，每个GPU只能分到一小部分。

再看NVLink下的性能：

```bash
# 使用NVLink通信的测试结果
./build/all_reduce_perf -b 128M -e 128M -g 8

# 输出：
#     size       time    algbw    busbw
#      (B)     (usec)  (GB/s)  (GB/s)
# 134217728     245.0   548.0   411.0
```

NVLink下的带宽达到了411GB/s，是PCIe的46倍。这个差距非常惊人，也说明了为什么大模型训练必须用NVLink。

迁移到NVLink后，还需要注意以下几点：

- **调整NCCL参数**：NVLink的带宽更高，需要适当增加NCCL的缓冲区大小。建议设置`NCCL_BUFFSIZE=4194304`（4MB）。
- **关闭PCIe相关优化**：如果之前为了优化PCIe通信而设置了`NCCL_P2P_DISABLE=1`，现在需要取消这个设置。
- **检查NVLink状态**：用`nvidia-smi nvlink -s`检查NVLink链路状态，确保所有链路都处于Active状态。

#### 严重警告

从PCIe迁移到NVLink后，如果发现性能没有提升，一定要检查NVLink链路状态。有时候因为驱动问题或者硬件故障，NVLink链路可能处于降级状态。用`nvidia-smi nvlink -s`可以看到每条链路的状态，如果显示`Degraded`，说明链路有问题，需要排查硬件。

#### 章节小结

这一章我们详细对比了NVLink和PCIe在物理层、协议层、带宽、延迟等方面的差异。NVLink凭借短距离直连、精简协议栈、点对点独立带宽等优势，在GPU通信中实现了比PCIe高一个数量级的性能。我们学习了如何使用nccl-tests进行带宽测试，了解了NVLink的不同拓扑结构对性能的影响，以及NCCL如何自动选择最优通信路径。最后通过一个实际迁移案例，看到了从PCIe到NVLink带来的46倍性能提升。

#### 本章重点回顾

- NVLink 3.0单向带宽600GB/s，是PCIe Gen5 x16的10倍，利用率可达90%以上
- NVLink协议栈精简，去掉了PCIe的复杂事务层，延迟低至0.2微秒
- NCCL通过ncclTopo模块自动探测GPU拓扑，为每对GPU选择最优通信路径
- PCIe带宽是共享的，8卡共享时每个GPU只能分到不到8GB/s
- 迁移到NVLink后，需要调整NCCL缓冲区大小并检查链路状态
