# 第9章：模型并行通信模式

模型并行是分布式训练中应对超大模型的关键手段。当单个GPU显存放不下整个模型时，我们需要把模型的不同层或不同参数切分到多个GPU上。这一章我们聚焦在模型并行场景下的通信模式。跟数据并行不同，模型并行中每个GPU只持有模型的一部分，前向和反向传播时需要在GPU之间传递激活值和梯度。我们会从最基础的层间流水线通信讲起，再到层内张量切分的All-Reduce变体，最后聊一聊通信调度和带宽利用率的工程优化。这一章的目标是让你理解不同模型并行策略下通信模式的选择依据，以及如何用NCCL原语高效实现这些模式。

### 9.1 层间流水线通信：点对点传输的基石

在流水线并行（Pipeline Parallelism）中，模型按层切分到多个GPU上，每个GPU负责一个stage。前向计算时，数据从第一个stage流到最后一个stage；反向计算时，梯度从最后一个stage流回第一个stage。这个过程中，相邻stage之间需要传递激活值和梯度，本质上是点对点通信。NCCL提供了`ncclSend`和`ncclRecv`原语来实现这种通信。

我们来看一个4-stage流水线的例子。假设模型有16层，每4层放在一个GPU上。GPU0持有第1-4层，GPU1持有第5-8层，依此类推。前向时，GPU0计算完第4层的输出后，需要把这个输出张量发送给GPU1。反向时，GPU1计算完第5层的梯度后，需要把梯度发送给GPU0。这种通信模式的特点是：通信只在相邻GPU之间发生，通信量等于一个micro-batch的激活值大小。

```python
import torch
import torch.distributed as dist

# 假设rank0持有stage0，rank1持有stage1
# 前向传播：rank0发送激活值给rank1
if rank == 0:
    # 计算完第4层输出
    activation = torch.randn(4, 1024, device='cuda')  # batch=4, hidden=1024
    dist.send(activation, dst=1)
elif rank == 1:
    activation = torch.empty(4, 1024, device='cuda')
    dist.recv(activation, src=0)
    # 继续计算第5-8层

# 反向传播：rank1发送梯度给rank0
if rank == 1:
    grad = torch.randn(4, 1024, device='cuda')
    dist.send(grad, dst=0)
elif rank == 0:
    grad = torch.empty(4, 1024, device='cuda')
    dist.recv(grad, src=1)
```

这里有一个工程经验：点对点通信的延迟跟消息大小和硬件拓扑有关。在NVLink连接的8卡机器上，4KB以下的消息延迟主要由软件开销决定，大约在5-10微秒；4MB以上的消息带宽可以接近NVLink的理论峰值（如每链路150GB/s）。对于流水线并行，激活值通常比较大（比如hidden\_size=1024, batch=4时，单层激活值约16KB，4层约64KB），所以带宽利用率是关键。

为了隐藏通信延迟，我们可以让计算和通信重叠。比如在GPU0发送激活值后，不要立即等待GPU1接收完成，而是继续做下一批数据的预处理。NCCL的`ncclSend`默认是异步的，只要用`ncclGroupStart/End`包裹，就可以实现通信与计算的重叠。下面是一个实际项目中的调度方式：

| 流水线阶段 | 前向通信操作 | 反向通信操作 | 典型消息大小 | NVLink延迟(μs) |
| --- | --- | --- | --- | --- |
| stage0→stage1 | send激活值 | recv梯度 | 64KB | 12 |
| stage1→stage2 | send激活值 | recv梯度 | 64KB | 12 |
| stage2→stage3 | send激活值 | recv梯度 | 64KB | 12 |

```arch-diagram
┌──────────┐    send激活值    ┌──────────┐    send激活值    ┌──────────┐
│  GPU0    │─────────────────→│  GPU1    │─────────────────→│  GPU2    │
│ stage0   │                  │ stage1   │                  │ stage2   │
│ 层1-4    │←─────────────────│ 层5-8    │←─────────────────│ 层9-12   │
└──────────┘    recv梯度      └──────────┘    recv梯度      └──────────┘
```

### 9.2 张量并行中的通信：All-Reduce的变体

张量并行（Tensor Parallelism）跟流水线并行不同，它是在一层内部做切分。比如一个Transformer的FFN层，权重矩阵是`[hidden, 4*hidden]`，我们可以把它按列切分到2个GPU上，每个GPU持有`[hidden, 2*hidden]`。计算时，每个GPU做一部分矩阵乘法，然后通过All-Reduce把结果合并。这种通信模式的特点是：通信发生在同一层的所有GPU之间，通信量等于激活值大小乘以GPU数量。

具体来说，在张量并行中，前向计算需要一次All-Reduce来合并各个GPU的部分结果；反向计算需要一次All-Reduce来同步梯度。这里的All-Reduce跟数据并行中的All-Reduce有区别：数据并行中All-Reduce的是整个梯度的平均值，而张量并行中All-Reduce的是部分求和的结果。NCCL的`ncclAllReduce`可以配置不同的规约操作，比如`ncclSum`用于求和，`ncclAvg`用于平均。

我们来看一个2-way张量并行的例子。假设hidden\_size=4096，FFN中间层大小=16384。每个GPU持有`[4096, 8192]`的权重。输入x是`[batch, 4096]`，每个GPU计算`x @ W_i`得到`[batch, 8192]`。然后两个GPU通过All-Reduce把结果拼接成`[batch, 16384]`。这里All-Reduce的通信量是`batch * 8192 * 4 bytes`（FP32）。

```python
import torch
import torch.distributed as dist

# 张量并行：2个GPU，按列切分FFN权重
world_size = 2
rank = dist.get_rank()

# 每个GPU持有部分权重
W_local = torch.randn(4096, 8192, device='cuda')  # 原始是[4096, 16384]

# 前向计算
x = torch.randn(4, 4096, device='cuda')  # batch=4
local_output = torch.mm(x, W_local)  # [4, 8192]

# All-Reduce求和，得到完整输出
dist.all_reduce(local_output, op=dist.ReduceOp.SUM)
# 现在local_output是[4, 16384]的完整结果
```

这里有个关键参数：通信量跟batch size和hidden size成正比。在GPT-3 175B训练中，hidden\_size=12288，batch\_size=1536，单层张量并行的All-Reduce通信量大约是`1536 * 12288 * 4 = 75MB`。如果使用8-way张量并行，通信量会更大。所以张量并行通常只在单机内使用（通过NVLink），避免跨机通信带来的高延迟。

#### 信息

在NVIDIA A100 80GB机器上，8卡NVLink的聚合带宽是600GB/s（单向）。对于75MB的消息，理论传输时间约125μs。但实际工程中，加上NCCL的软件开销和同步等待，实测延迟在200-300μs。这个延迟相对于FFN层的计算时间（约1-2ms）是可以接受的。

### 9.3 序列并行中的通信模式

序列并行（Sequence Parallelism）是最近大模型训练中常用的一种并行策略。它把输入序列的长度维度切分到多个GPU上。比如一个序列长度是2048，我们可以把它切成4段，每段512，分别放到4个GPU上。每个GPU只计算自己那一段的注意力，但注意力计算需要全局的key和value，所以GPU之间需要通信来交换key和value。

具体来说，在序列并行中，每个GPU计算完自己的query、key、value后，需要把所有GPU的key和value收集到一起，才能计算完整的注意力分数。这通常使用All-Gather操作。NCCL的`ncclAllGather`可以把每个GPU上的数据收集到所有GPU上。通信量等于key和value的总大小乘以GPU数量。

我们来看一个4-way序列并行的例子。假设序列长度=2048，hidden\_size=4096，head=32，每个head的维度=128。每个GPU持有`[512, 4096]`的输入。计算得到key和value的维度是`[512, 32, 128]`。All-Gather后，每个GPU得到`[2048, 32, 128]`的完整key和value。

```python
import torch
import torch.distributed as dist

# 序列并行：4个GPU，切分序列长度
world_size = 4
rank = dist.get_rank()
seq_len_per_gpu = 512
total_seq_len = 2048
hidden_size = 4096
num_heads = 32
head_dim = 128

# 每个GPU计算自己的key和value
key_local = torch.randn(seq_len_per_gpu, num_heads, head_dim, device='cuda')
value_local = torch.randn(seq_len_per_gpu, num_heads, head_dim, device='cuda')

# All-Gather收集所有key和value
key_full = torch.empty(total_seq_len, num_heads, head_dim, device='cuda')
value_full = torch.empty(total_seq_len, num_heads, head_dim, device='cuda')

dist.all_gather(list(key_full.chunk(world_size, dim=0)), key_local)
dist.all_gather(list(value_full.chunk(world_size, dim=0)), value_local)

# 现在可以计算完整注意力
# key_full: [2048, 32, 128]
```

序列并行的通信量跟序列长度和hidden size成正比。对于长序列（比如16K或32K），通信量会非常大。实际工程中，序列并行通常跟张量并行和流水线并行组合使用。比如在Megatron-LM框架中，序列并行是在张量并行的基础上实现的，它把All-Gather操作跟矩阵乘法做了融合，减少了通信次数。

### 9.4 通信调度：同步与异步的选择

在模型并行中，通信操作的调度方式直接影响训练吞吐。同步通信要求所有GPU都到达同一个通信点后才能开始传输，而异步通信允许GPU在等待通信完成的同时做其他计算。NCCL提供了两种方式：阻塞式调用和非阻塞式调用。阻塞式调用（如`torch.distributed.all_reduce`的默认行为）会等待通信完成才返回；非阻塞式调用（使用`async_op=True`）立即返回一个`Work`对象，可以后续通过`wait()`同步。

在流水线并行中，我们通常使用异步通信来隐藏延迟。比如在GPU0发送激活值给GPU1后，GPU0可以立即开始计算下一批数据，而不需要等待GPU1确认接收。NCCL的`ncclSend`和`ncclRecv`在默认情况下就是非阻塞的，但需要配合`ncclGroupStart/End`来确保所有操作被正确提交。

在张量并行中，情况稍有不同。All-Reduce操作通常需要等待所有GPU都到达同步点，因为每个GPU都需要其他GPU的数据才能继续计算。这种情况下，同步通信是必要的。但我们可以通过将All-Reduce跟计算重叠来减少等待时间。比如在FFN层中，第一个矩阵乘法完成后，可以立即启动All-Reduce，然后开始计算第二个矩阵乘法的部分结果。

| 并行策略 | 通信原语 | 推荐调度方式 | 延迟隐藏方法 | 典型场景 |
| --- | --- | --- | --- | --- |
| 流水线并行 | Send/Recv | 异步非阻塞 | 计算与通信重叠 | 跨stage传递激活值 |
| 张量并行 | All-Reduce | 同步+计算重叠 | 将All-Reduce与GEMM重叠 | 层内部分结果合并 |
| 序列并行 | All-Gather | 同步 | 将All-Gather与注意力计算重叠 | 长序列注意力计算 |

工程中有一个常见问题：同步通信会导致GPU空闲等待，降低利用率。比如在8-way张量并行中，如果某个GPU计算速度稍慢，其他7个GPU都会在All-Reduce处等待。解决方法是使用NCCL的`ncclBroadcast`和`ncclReduce`的异步版本，并结合CUDA streams来实现细粒度的调度。下面是一个使用CUDA stream实现计算通信重叠的例子：

```python
import torch
import torch.distributed as dist

# 使用两个CUDA stream：一个计算，一个通信
compute_stream = torch.cuda.Stream()
comm_stream = torch.cuda.Stream()

# 前向计算：第一个矩阵乘法
with torch.cuda.stream(compute_stream):
    local_output = torch.mm(x, W_local)

# 在通信stream上启动All-Reduce
with torch.cuda.stream(comm_stream):
    # 等待计算完成
    comm_stream.wait_stream(compute_stream)
    dist.all_reduce(local_output, op=dist.ReduceOp.SUM, async_op=True)

# 主stream继续做其他计算
# 比如计算第二个矩阵乘法的部分结果
with torch.cuda.stream(compute_stream):
    # 这里可以做一些不依赖local_output的计算
    intermediate = torch.mm(x, W_local2)

# 同步所有stream
torch.cuda.synchronize()
```

#### 严重警告

使用多stream重叠计算和通信时，一定要注意数据依赖关系。如果后续计算依赖All-Reduce的结果，必须在通信完成后才能开始。上述例子中，如果`intermediate`的计算依赖`local_output`，就会产生数据竞争。实际项目中建议用`torch.cuda.Stream.wait_event`或`torch.cuda.Event`来精确控制同步点。

### 9.5 通信量分析与带宽利用率

模型并行中的通信量是影响训练效率的关键因素。我们需要精确计算每个并行策略的通信量，才能做出合理的设计决策。通信量通常用每个训练step传输的字节数来衡量。对于流水线并行，每个micro-batch的通信量是`2 * activation_size`（前向一次，反向一次）。对于张量并行，每个transformer层的通信量是`2 * hidden_size * batch_size * 4 bytes`（FP32）。

我们以GPT-3 175B模型为例，计算不同并行策略的通信量。假设hidden\_size=12288，batch\_size=1536，序列长度=2048，使用FP16训练（每个元素2字节）。每个transformer层的激活值大小是`batch * seq_len * hidden_size * 2 = 1536 * 2048 * 12288 * 2 ≈ 77GB`。这显然太大了，所以实际训练中会使用梯度检查点（activation checkpointing）来减少存储。

在8-way张量并行下，每个GPU只需要存储1/8的激活值，但通信量是`2 * 77GB / 8 = 19.25GB`（因为All-Reduce需要传输完整激活值）。如果NVLink带宽是600GB/s，通信时间约32ms。而一个transformer层的计算时间大约在50-100ms，所以通信占比在30%-60%之间。

带宽利用率是另一个关键指标。NCCL的All-Reduce在消息大小超过1MB时，带宽利用率可以达到90%以上。但对于小消息（比如流水线并行中的激活值，可能只有64KB），带宽利用率会下降到50%以下。这是因为小消息的通信延迟主要由软件开销和握手协议决定。工程中可以通过合并多个小消息来提升带宽利用率，比如把多个层的激活值打包在一起发送。

### 9.6 混合并行中的通信模式组合

在实际的大模型训练中，我们很少单独使用一种并行策略。通常是把数据并行、流水线并行、张量并行和序列并行组合在一起。这种混合并行策略的通信模式非常复杂，需要精心设计通信拓扑和调度顺序。以Megatron-LM框架为例，它使用了3D并行：数据并行（DP）+ 张量并行（TP）+ 流水线并行（PP）。

在3D并行中，通信发生在三个维度上：数据并行维度做All-Reduce梯度同步，张量并行维度做All-Reduce部分结果合并，流水线并行维度做点对点激活值传递。这三个维度的通信是相互独立的，但共享同一组GPU和NVLink带宽。如果调度不当，不同维度的通信会互相干扰，导致带宽争用和延迟增加。

NCCL提供了`ncclCommSplit`功能，可以把一个通信域拆分成多个子通信域，每个子通信域负责一个维度的通信。这样可以避免不同通信模式之间的干扰。下面是一个3D并行中通信域划分的例子：

```python
import torch.distributed as dist

# 假设总共有64个GPU，配置为：DP=4, TP=4, PP=4
# 总GPU数 = DP * TP * PP = 4 * 4 * 4 = 64

# 创建通信组
# 数据并行组：相同TP和PP的GPU组成一个DP组
dp_ranks = [0, 16, 32, 48]  # 示例：rank0,16,32,48在同一个DP组
dp_group = dist.new_group(ranks=dp_ranks)

# 张量并行组：相同DP和PP的GPU组成一个TP组
tp_ranks = [0, 1, 2, 3]  # 示例：rank0-3在同一个TP组
tp_group = dist.new_group(ranks=tp_ranks)

# 流水线并行组：相同DP和TP的GPU组成一个PP组
pp_ranks = [0, 4, 8, 12]  # 示例：rank0,4,8,12在同一个PP组
pp_group = dist.new_group(ranks=pp_ranks)

# 在TP组内做张量并行通信
dist.all_reduce(tensor, op=dist.ReduceOp.SUM, group=tp_group)

# 在PP组内做流水线并行通信
dist.send(activation, dst=next_rank, group=pp_group)

# 在DP组内做梯度同步
dist.all_reduce(grad, op=dist.ReduceOp.AVG, group=dp_group)
```

这里有一个工程经验：不同维度的通信优先级不同。张量并行通信通常是同步的，延迟敏感，应该分配最高的优先级；流水线并行通信是异步的，可以延迟一些；数据并行通信的梯度同步可以在计算的同时进行，优先级最低。NCCL的`ncclCommSplit`支持设置每个子通信域的优先级，通过`ncclConfig`结构体中的`priority`字段。

| 并行维度 | 通信原语 | 优先级 | 延迟要求 | 典型消息大小 |
| --- | --- | --- | --- | --- |
| 张量并行 (TP) | All-Reduce | 高 | < 500μs | 10-100MB |
| 流水线并行 (PP) | Send/Recv | 中 | < 2ms | 64KB-1MB |
| 数据并行 (DP) | All-Reduce | 低 | < 10ms | 100-500MB |

### 9.7 通信拓扑感知的优化策略

NCCL在初始化时会探测GPU之间的物理拓扑，包括NVLink连接、PCIe层次和跨机网络。模型并行的通信模式应该跟硬件拓扑匹配，才能获得最佳性能。比如在DGX A100机器上，8个GPU通过NVSwitch全互联，任意两个GPU之间的带宽相同。但在一些定制化服务器上，GPU之间可能只有部分NVLink连接，形成环形或混合立方体拓扑。

对于张量并行，我们希望通信量最大的GPU对之间具有高带宽连接。比如在8卡机器上，如果使用2-way张量并行，应该把GPU0和GPU3配对（它们之间有NVLink直连），而不是GPU0和GPU7（可能只有PCIe连接）。NCCL的`ncclTopoGetSystem`函数可以获取拓扑信息，我们可以根据拓扑来分配GPU rank。

下面是一个拓扑感知的rank分配示例：

```python
import torch
import torch.distributed as dist
from torch.cuda import nccl

# 获取当前机器的GPU拓扑
topo = nccl.ncclTopoGetSystem()
# topo包含了GPU之间的连接类型和带宽

# 假设我们想要2-way张量并行
# 找到NVLink直连的GPU对
pairs = []
for i in range(8):
    for j in range(i+1, 8):
        link_type = nccl.ncclTopoGetLinkType(topo, i, j)
        if link_type == nccl.NCCL_TOPO_LINK_NVLINK:
            pairs.append((i, j))

# 选择带宽最高的对
# 实际项目中需要更复杂的匹配算法
best_pair = pairs[0]  # 简化示例
print(f"推荐张量并行GPU对: {best_pair}")
```

对于流水线并行，通信只发生在相邻stage之间，所以应该把相邻stage放在物理上相邻的GPU上。比如在NVLink环形拓扑中，把stage0放在GPU0，stage1放在GPU1，这样send/recv走NVLink，延迟最低。如果跨机流水线并行，需要把每个stage内部的所有GPU放在同一台机器上，避免跨机通信成为瓶颈。

### 9.8 通信故障处理与容错

在分布式训练中，通信故障是不可避免的。NCCL提供了超时机制和错误处理API。当通信操作超时或失败时，NCCL会返回错误码，我们可以根据错误类型做相应的处理。常见的通信故障包括：GPU掉线、网络中断、带宽降级等。

对于模型并行，通信故障的影响范围取决于并行策略。如果张量并行中的一个GPU掉线，整个训练任务都会失败，因为每个GPU都依赖其他GPU的数据。而流水线并行中，如果某个stage的GPU掉线，只会影响该stage的通信，其他stage可能可以继续运行（但最终会因为数据不完整而失败）。

NCCL 2.12及以上版本支持`ncclCommAbort`和`ncclCommDestroy`，可以在通信失败后清理资源并重新初始化。下面是一个简单的容错处理示例：

```python
import torch
import torch.distributed as dist

max_retries = 3
for attempt in range(max_retries):
    try:
        # 执行All-Reduce
        dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
        break  # 成功则退出重试循环
    except RuntimeError as e:
        print(f"通信失败，第{attempt+1}次重试: {e}")
        if attempt == max_retries - 1:
            raise  # 最后一次重试仍然失败，抛出异常
        # 销毁当前通信组并重新初始化
        dist.destroy_process_group()
        dist.init_process_group(backend='nccl')
        # 重新创建通信组
        # ...
```

在实际工程中，更常见的做法是使用检查点（checkpoint）来恢复训练。每训练一定步数就保存模型状态，当通信故障发生时，从最近的检查点恢复。这样即使通信失败导致训练中断，也不会丢失太多进度。对于模型并行，检查点需要保存所有GPU上的模型参数和优化器状态，恢复时需要确保所有GPU都加载到相同的状态。

#### 章节小结

这一章我们深入讲解了模型并行中的通信模式。从流水线并行的点对点通信开始，到张量并行的All-Reduce变体，再到序列并行的All-Gather模式。我们分析了不同并行策略的通信量、延迟和带宽利用率，并讨论了通信调度和拓扑感知的优化方法。最后介绍了混合并行中的通信域划分和故障处理。理解这些通信模式是设计高效分布式训练系统的基础。

#### 本章重点回顾

- 流水线并行使用点对点Send/Recv通信，通信量等于激活值大小，适合跨stage传输
- 张量并行使用All-Reduce合并部分结果，通信量随GPU数量线性增长，适合单机内使用
- 序列并行使用All-Gather收集全局key和value，通信量跟序列长度成正比，适合长序列场景
- 混合并行中不同维度的通信需要划分独立的通信域，并设置优先级避免带宽争用
- 通信拓扑感知的rank分配可以最大化利用NVLink带宽，减少跨机通信延迟
