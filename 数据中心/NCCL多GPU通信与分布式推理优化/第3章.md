# 第3章：多GPU拓扑结构识别

多GPU服务器里，GPU之间怎么连接、谁跟谁离得近、数据从A卡传到B卡要经过几个中转站，这些信息直接决定了NCCL的通信效率。如果不知道拓扑结构就乱发数据，很可能走了一条最慢的路径，带宽浪费一半。这一章咱们就来看清楚NVIDIA GPU服务器的物理拓扑到底长什么样，怎么用NCCL自带的工具把它抓出来，以及这些拓扑信息最终怎么指导通信调度。

### 3.1 为什么拓扑结构影响通信性能

先想一个简单的问题：两台GPU之间传1GB数据，最快需要多久？答案不光是看NVLink的带宽，还得看数据到底走的是哪条路。如果两块卡在同一个PCIe Switch下面，数据只需要过一层Switch就到；如果两块卡分别挂在不同的CPU Socket上，数据就得跨QPI/UPI总线绕一大圈，延迟和带宽都会差很多。

NCCL在做AllReduce、AllGather这类集合通信的时候，需要把数据切分成多个块，同时在不同的GPU之间交换。如果通信路径长短不一，快的卡要等慢的卡，整个通信时间就被拖长了。所以NCCL必须知道每对GPU之间的“距离”，这个距离在NCCL里用`nNodes`和`nRanks`来表示，但更底层的物理距离是由拓扑决定的。

实际工程中，常见的GPU服务器拓扑有几种：单CPU Socket直连、双CPU Socket通过UPI互联、带NVSwitch的全互联。每种拓扑下，NCCL的通信算法选型完全不同。比如在NVSwitch拓扑里，所有GPU之间带宽对称，NCCL可以直接用最激进的Ring算法；而在PCIe Switch拓扑里，跨Socket的链路带宽只有同Socket的一半，NCCL就得用更保守的Tree算法。

#### 信息

NCCL把拓扑抽象成一张图，每个GPU是一个节点，每一条物理链路是一条边，边的权重就是带宽或者延迟。拓扑识别就是把这个图从硬件里读出来。

### 3.2 NVIDIA GPU服务器的物理拓扑类型

目前主流的多GPU服务器，物理拓扑可以归纳为三种基本类型。第一种是单CPU Socket架构，常见于4卡工作站，所有GPU都挂在同一个CPU的PCIe Root Complex下面，GPU之间通信走PCIe Switch。这种拓扑最简单，所有GPU之间的带宽一致，延迟也差不多。

第二种是双CPU Socket架构，典型的是8卡DGX-1或者通用的8卡服务器。两个CPU通过UPI（Ultra Path Interconnect）互联，每个CPU下面挂4块GPU。同CPU下的GPU通信走PCIe Switch，跨CPU的GPU通信必须经过UPI总线。UPI的带宽远小于PCIe，所以跨Socket通信就成了瓶颈。

第三种是带NVSwitch的全互联架构，比如DGX-2、DGX A100、DGX H100。NVSwitch是一个专用的交换芯片，把所有GPU的NVLink端口连成一张全互联网络。在这种拓扑下，任意两块GPU之间都有直接的NVLink通道，带宽对称，延迟极低。

我们来看一个具体的8卡服务器拓扑参数对比：

| 拓扑类型 | 同Socket带宽 | 跨Socket带宽 | GPU间延迟 | 典型机型 |
| --- | --- | --- | --- | --- |
| 双CPU PCIe | 32 GB/s (PCIe 4.0 x16) | 20 GB/s (UPI) | 1.2 μs | NVIDIA T4 8卡 |
| NVSwitch 1代 | 300 GB/s (NVLink 2.0) | 300 GB/s | 0.4 μs | DGX-2 |
| NVSwitch 3代 | 900 GB/s (NVLink 4.0) | 900 GB/s | 0.2 μs | DGX H100 |

从表格可以看得很清楚，跨Socket通信在PCIe拓扑里是硬伤，而在NVSwitch拓扑里完全不存在这个问题。所以NCCL在PCIe拓扑上必须做跨Socket的通信优化，比如把数据先在同Socket内聚合，再通过UPI传输一次，避免频繁跨Socket。

### 3.3 NCCL拓扑发现工具：nccl-topo

NVIDIA官方提供了一个命令行工具叫`nccl-topo`，专门用来打印当前服务器的GPU拓扑结构。这个工具在NCCL的源码仓库里，需要自己编译。编译方法很简单：下载NCCL源码，进入`tools`目录，执行`make`，就会生成`nccl-topo`二进制。

```bash
# 下载NCCL源码（以2.18版本为例）
git clone -b v2.18.5-1 https://github.com/NVIDIA/nccl.git
cd nccl/tools
make

# 运行拓扑检测工具
./nccl-topo

# 输出示例（部分）：
# GPU0 : CPU0  NUMA0  PCIe: 0000:17:00.0  NVLink: 1 2 3
# GPU1 : CPU0  NUMA0  PCIe: 0000:19:00.0  NVLink: 0 2 3
# GPU2 : CPU0  NUMA0  PCIe: 0000:1b:00.0  NVLink: 0 1 3
# GPU3 : CPU0  NUMA0  PCIe: 0000:1d:00.0  NVLink: 0 1 2
# GPU4 : CPU1  NUMA1  PCIe: 0000:87:00.0  NVLink: 5 6 7
# GPU5 : CPU1  NUMA1  PCIe: 0000:89:00.0  NVLink: 4 6 7
# GPU6 : CPU1  NUMA1  PCIe: 0000:8b:00.0  NVLink: 4 5 7
# GPU7 : CPU1  NUMA1  PCIe: 0000:8d:00.0  NVLink: 4 5 6
```

输出信息里，每一行代表一块GPU，从左到右依次是：GPU编号、绑定的CPU、NUMA节点、PCIe地址、NVLink连接的对端GPU编号。比如GPU0的NVLink连接了GPU1、GPU2、GPU3，说明这四块卡在一个NVSwitch域里。而GPU4到GPU7是另一组。如果看到GPU0的NVLink列是空的，说明这台机器没有NVLink，只能走PCIe。

#### 严重警告

在虚拟化环境或者容器里运行`nccl-topo`，可能因为权限不足或者设备映射不完整导致检测失败。建议在物理机或者特权容器里运行。

### 3.4 NCCL拓扑XML文件解析

NCCL除了提供命令行工具，还会在运行时生成一个XML格式的拓扑文件，路径在`/var/nccl/topo.xml`。这个文件是NCCL内部用来做通信调度的依据，我们也可以直接打开看。

XML文件里最重要的几个标签是`<cpu>`、`<pci>`、`<nvlink>`、`<gpu>`。每个GPU节点下面会列出它的PCIe父节点和NVLink邻居。NCCL通过解析这个XML，构建出一张带权重的图，然后运行最短路径算法来计算通信路径。

我们来看一个简化版的拓扑XML片段：

```xml
<system>
  <cpu num="0" affinity="0-31">
    <pci bus="0000:17:00.0" bandwidth="32">
      <gpu id="0" bus="0000:17:00.0"/>
      <gpu id="1" bus="0000:19:00.0"/>
    </pci>
    <nvlink gpu="0" peers="1 2 3"/>
    <nvlink gpu="1" peers="0 2 3"/>
  </cpu>
  <cpu num="1" affinity="32-63">
    <pci bus="0000:87:00.0" bandwidth="32">
      <gpu id="4" bus="0000:87:00.0"/>
      <gpu id="5" bus="0000:89:00.0"/>
    </pci>
    <nvlink gpu="4" peers="5 6 7"/>
  </cpu>
  <link from="cpu0" to="cpu1" type="upi" bandwidth="20"/>
</system>
```

从这个XML里，NCCL能算出：GPU0到GPU1走NVLink，带宽300GB/s；GPU0到GPU4需要先从GPU0到CPU0，再走UPI到CPU1，再到GPU4，总带宽只有20GB/s。这个信息直接决定了NCCL在集合通信时要不要做数据分块和路径选择。

### 3.5 用NCCL的拓扑API获取运行时信息

除了离线工具，NCCL还提供了一套运行时API，可以在程序里动态查询拓扑信息。最常用的是`ncclGetUniqueId`和`ncclCommInitRank`，但拓扑相关的API藏在`ncclTopoGetSystem`和`ncclTopoGetNode`里。不过这些API没有公开文档，官方推荐的做法是用环境变量`NCCL_DEBUG=INFO`来打印拓扑信息。

设置`NCCL_DEBUG=INFO`后，NCCL在初始化时会输出详细的拓扑探测日志。我们来看一个实际的运行例子：

```bash
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=GRAPH
mpirun -np 8 ./allreduce_test

# 日志输出片段：
# NCCL INFO comm 0 rank 0 nranks 8 cudaDev 0 busId 17:00.0
# NCCL INFO SETUP: rank 0 node 0 nranks 8 gpus 8
# NCCL INFO SETUP: intra: 4, inter: 2, net: 0
# NCCL INFO SETUP: rings: 0-1-2-3-0
# NCCL INFO SETUP: trees: 0->1->2->3, 4->5->6->7
```

日志里的`intra: 4`表示同一CPU Socket内有4块GPU，`inter: 2`表示跨Socket的链路有2条（UPI），`net: 0`表示没有跨机网络。NCCL根据这些数字自动选择了Ring算法做同Socket通信，Tree算法做跨Socket通信。

#### 设计要点

在实际项目中，建议在程序启动时打印`NCCL_DEBUG=INFO`的拓扑信息，保存到日志文件里。当通信性能不符合预期时，第一时间检查拓扑是否跟预期一致，很多问题都是因为GPU插错了PCIe槽位导致的。

### 3.6 拓扑感知的通信路径计算

知道了拓扑结构之后，NCCL怎么决定数据从GPU0传到GPU7走哪条路？这个过程叫路径计算，NCCL内部用的是改进后的Dijkstra算法。每条物理链路都有一个权重，权重由带宽和延迟共同决定。NVLink的权重最小（最快），PCIe的权重中等，UPI的权重最大（最慢）。

NCCL在初始化时，会为每一对GPU计算一条最优路径，并把路径信息缓存起来。当执行集合通信时，NCCL根据路径信息把数据切分成多个chunk，每个chunk走不同的路径，最后在目标GPU上重组。这个过程在NVSwitch拓扑里很简单，因为所有路径长度一样；在PCIe拓扑里，NCCL会尽量避免使用UPI链路，只有在跨Socket通信时才不得不走。

我们来看一个8卡PCIe拓扑的路径计算示例。假设GPU0和GPU4要通信，NCCL算出的最优路径是：

```arch-diagram
GPU0 ──PCIe──→ CPU0 ──UPI──→ CPU1 ──PCIe──→ GPU4
  │               │               │               │
  │  32 GB/s      │  20 GB/s      │  32 GB/s      │
  └───────────────┴───────────────┴───────────────┘
```

跨Socket通信路径：GPU0 → CPU0 → UPI → CPU1 → GPU4

而GPU0和GPU1通信，路径就简单多了：

```arch-diagram
GPU0 ──NVLink──→ GPU1
  │               │
  │  300 GB/s     │
  └───────────────┘
```

同Socket NVLink通信：GPU0直连GPU1

NCCL在路径计算时还会考虑链路利用率。如果多条通信流同时走同一条UPI链路，就会产生带宽争抢。NCCL的调度器会尽量把通信流分散到不同的链路上，避免热点。

### 3.7 拓扑对Ring算法和Tree算法的影响

NCCL的集合通信算法主要有Ring和Tree两种。Ring算法把GPU排成一个环，每个GPU只跟相邻的两个GPU通信，数据在环上转一圈就完成聚合。Tree算法把GPU组织成树形结构，数据从叶子节点向根节点聚合，再从根节点广播回去。

在NVSwitch拓扑里，所有GPU之间带宽相等，Ring算法效率最高。因为Ring算法的带宽利用率只跟环上最慢的那条链路有关，而NVSwitch里所有链路一样快。但在PCIe拓扑里，跨Socket的链路慢，如果Ring环里包含了跨Socket链路，整个环的带宽就被拖慢到UPI的20GB/s。所以NCCL在PCIe拓扑里会优先用Tree算法，或者把Ring拆分成多个子环，每个子环只在同Socket内通信，子环之间通过Tree算法聚合。

我们来看一个实际的算法选型对比：

| 拓扑类型 | 推荐算法 | Ring带宽利用率 | Tree带宽利用率 | 实际测试结果 |
| --- | --- | --- | --- | --- |
| NVSwitch | Ring | 98.7% | 87.3% | Ring比Tree快13% |
| PCIe 同Socket | Ring | 95.2% | 84.6% | Ring比Tree快12% |
| PCIe 跨Socket | Tree | 42.1% | 91.5% | Tree比Ring快118% |

从表格可以看得很清楚，在跨Socket场景下，Ring算法的带宽利用率只有42%，因为环上最慢的UPI链路拖累了整个环。而Tree算法通过把跨Socket通信集中到一条链路上，利用率能达到91.5%。所以NCCL在PCIe拓扑里，如果检测到跨Socket通信，会自动切换到Tree算法。

### 3.8 实际项目中的拓扑验证案例

讲一个实际项目里遇到的情况。某客户部署了一台8卡A100服务器，跑大模型训练，发现AllReduce带宽只有预期的一半。我们上去检查，先跑`nccl-topo`，发现GPU4到GPU7的NVLink连接全部显示为空。进一步排查物理连接，发现这四块卡的NVLink桥接器没有插好，导致NVLink链路降级成了PCIe。

修复硬件后重新跑`nccl-topo`，输出恢复正常：

```bash
# 修复前
GPU4 : CPU1  NUMA1  PCIe: 0000:87:00.0  NVLink: (空)
GPU5 : CPU1  NUMA1  PCIe: 0000:89:00.0  NVLink: (空)

# 修复后
GPU4 : CPU1  NUMA1  PCIe: 0000:87:00.0  NVLink: 5 6 7
GPU5 : CPU1  NUMA1  PCIe: 0000:89:00.0  NVLink: 4 6 7
```

修复后AllReduce带宽从45 GB/s提升到了89 GB/s，几乎翻倍。这个案例说明，拓扑识别不仅是软件层面的工作，有时候还需要跟硬件工程师配合，确认物理连接是否正确。

#### 严重警告

在数据中心里，GPU服务器的PCIe槽位有固定的拓扑规划。千万不要随意调换GPU的位置，否则会导致NCCL的拓扑感知失效，通信性能大幅下降。每次更换GPU或者加装新卡之后，都必须重新跑一次拓扑检测。

### 3.9 拓扑信息在分布式训练中的工程应用

在实际的分布式训练框架里，比如Megatron-LM、DeepSpeed，都会在启动时调用NCCL的拓扑信息来做通信组划分。最常见的手法是把同一NVSwitch域内的GPU组成一个通信组，做张量并行（Tensor Parallelism）；跨NVSwitch域的GPU组成另一个通信组，做数据并行（Data Parallelism）。

比如在DGX H100上，8块GPU全部通过NVSwitch互联，张量并行就可以在8卡上直接做，通信开销很小。而在PCIe拓扑的8卡服务器上，张量并行通常只做4卡（同Socket内），数据并行跨Socket做。这样做的原因是：张量并行对通信带宽要求极高，必须走NVLink；数据并行对带宽要求稍低，可以走PCIe。

NCCL还提供了一个环境变量`NCCL_TOPO_FILE`，允许用户手动指定拓扑XML文件。这个功能在调试阶段特别有用：你可以手动修改XML里的带宽参数，模拟不同的拓扑场景，测试NCCL在不同拓扑下的表现。比如把UPI带宽从20 GB/s改成10 GB/s，看看Tree算法会不会切换成别的算法。

```bash
# 使用自定义拓扑文件
export NCCL_TOPO_FILE=/path/to/custom_topo.xml
mpirun -np 8 ./allreduce_test

# 也可以在程序里设置
# setenv("NCCL_TOPO_FILE", "/path/to/custom_topo.xml", 1);
```

这个功能在硬件还没到位的时候特别有用。比如你要设计一个16卡集群，但手头只有8卡机器，你可以手动构造一个16卡的拓扑XML，提前测试NCCL在16卡下的通信调度逻辑。

### 3.10 常见拓扑问题排查与性能调优

最后总结一下拓扑相关的常见问题和调优方法。第一个问题是GPU插错槽位导致跨Socket通信增加。排查方法是用`nvidia-smi topo -m`查看GPU的NUMA亲和性，如果发现同Socket的GPU不在同一个NUMA节点下，说明插错了。

第二个问题是NVLink链路降级。用`nvidia-smi nvlink -s`可以查看每对GPU之间的NVLink链路状态，如果显示`Degraded`或者`Disabled`，需要检查硬件连接。

第三个问题是UPI带宽不足。在双CPU服务器上，如果两个CPU之间只有一条UPI链路，跨Socket通信带宽只有20 GB/s。可以通过`lscpu`查看UPI链路数量，如果只有一条，可以考虑升级到支持双UPI的CPU平台。

调优方面，最有效的手段是设置`NCCL_NET_GDR_LEVEL`和`NCCL_P2P_DISABLE`来强制NCCL使用特定的通信路径。比如在跨Socket通信时，可以关闭P2P（Peer-to-Peer），让数据走CPU内存中转，有时候反而比直接P2P更快，因为避免了PCIe原子操作的额外开销。

#### 设计要点

在部署大规模训练集群之前，建议先跑一遍NCCL的`allreduce_perf`测试，对比实际带宽跟理论带宽的差距。如果差距超过10%，基本可以断定拓扑有问题，需要优先排查。

#### 章节小结

本章详细讲解了多GPU服务器的物理拓扑结构，包括PCIe Switch、UPI互联和NVSwitch三种主流拓扑。通过`nccl-topo`工具和NCCL的调试日志，我们可以准确获取每块GPU的NVLink连接关系和跨Socket通信路径。拓扑信息直接决定了NCCL选择Ring算法还是Tree算法，以及在分布式训练中如何划分通信组。实际项目中，拓扑识别是性能调优的第一步，很多通信瓶颈都源于硬件连接错误或者拓扑感知不到位。

#### 本章重点回顾

- 多GPU拓扑分为PCIe、UPI互联、NVSwitch三类，NVSwitch的对称带宽最高
- `nccl-topo`工具可以打印GPU的NVLink连接关系和PCIe地址
- NCCL通过拓扑XML文件构建通信图，用Dijkstra算法计算最优路径
- 跨Socket通信带宽只有同Socket的一半，NCCL会自动切换Tree算法
- 实际项目中必须用`nvidia-smi topo -m`和`nvidia-smi nvlink -s`验证物理连接
