# 第10章：流水线并行通信设计

大规模Transformer模型的训练和推理，单卡显存放不下，多卡数据并行又因为模型太大导致通信量爆炸。流水线并行把模型切成多个阶段，每个GPU只负责一个阶段，显存压力瞬间降下来。但问题来了：前一个阶段算完的结果怎么高效传给下一个阶段？这就是本章要解决的核心问题。我们会从最基础的F-t-S（前向-张量-后向）通信模式讲起，深入到点对点通信的异步优化、调度策略、以及NCCL原语在流水线场景下的具体用法。目标很明确：让你理解流水线并行的通信瓶颈在哪里，以及如何用NCCL的API把通信开销压到最低。

### 10.1 流水线并行的基本通信模式：F-t-S

先看一个最简单的流水线场景：4个GPU，模型切成4段，每个GPU算一段。数据从GPU0流到GPU3，这叫前向传播（Forward）。前向结束后，梯度从GPU3反向流回GPU0，这叫后向传播（Backward）。所以流水线并行的通信模式就是前向传数据、后向传梯度，中间每个阶段还要做张量（Tensor）的收发。这种模式业界叫F-t-S（Forward-tensor-Backward）。

每个GPU在流水线里只跟相邻的GPU通信：前一个GPU发数据给它，它处理后发给下一个GPU。这种点对点通信（P2P）跟集合通信（AllReduce）不一样，它不需要全局同步，但要求收发必须配对。如果发送端发了数据，接收端没及时收，就会导致显存堆积，甚至死锁。

我们来看一个典型的F-t-S通信时序图：

```arch-diagram
时间轴↓
        GPU0          GPU1          GPU2          GPU3
         │             │             │             │
F0───────┼────────────→│             │             │
         │             │             │             │
F1───────┼───────────────────────────→│             │
         │             │             │             │
F2───────┼─────────────────────────────────────────→│
         │             │             │             │
B2←──────┼──────────────────────────────────────────│
         │             │             │             │
B1←──────┼───────────────────────────│             │
         │             │             │             │
B0←──────┼────────────│             │             │
         │             │             │             │
```

图10-1：4级流水线的F-t-S通信时序（F=前向，B=后向，数字代表微批次编号）

实际工程中，F-t-S模式必须配合微批次（Micro-batch）才能把GPU利用率提上去。比如一次送4个微批次进流水线，前向传播时GPU0算完微批次0就传给GPU1，然后立刻开始算微批次1，这样GPU1不用等GPU0算完所有批次再开始。这种调度策略叫“异步流水线”，后面我们会详细讲。

#### 信息

在NCCL中，F-t-S模式对应的API是`ncclSend`和`ncclRecv`。这两个函数是点对点通信的基础，但必须成对调用，否则会导致通信卡死。实际项目中，我们通常把收发操作放到单独的CUDA流里，避免阻塞计算流。

### 10.2 点对点通信原语：ncclSend与ncclRecv

NCCL提供了两个最基本的点对点通信原语：`ncclSend`和`ncclRecv`。它们的函数签名如下：

```c
ncclResult_t ncclSend(const void* sendbuff, size_t count,
                       ncclDataType_t datatype, int peer,
                       ncclComm_t comm, cudaStream_t stream);

ncclResult_t ncclRecv(void* recvbuff, size_t count,
                       ncclDataType_t datatype, int peer,
                       ncclComm_t comm, cudaStream_t stream);
```

参数说明：`sendbuff`是发送缓冲区指针，`recvbuff`是接收缓冲区指针；`count`是元素个数；`datatype`是数据类型（比如`ncclFloat32`）；`peer`是目标/源GPU的rank号；`comm`是NCCL通信器；`stream`是CUDA流。

这里有个关键点：`ncclSend`和`ncclRecv`都是异步操作，它们会立即返回，但实际数据传输在指定的CUDA流中执行。所以你必须确保在访问接收缓冲区之前，调用`cudaStreamSynchronize`或者用事件同步。否则你会读到未完成的数据。

在流水线并行中，我们通常把收发操作放到一个单独的通信流（communication stream）中，计算操作放到计算流（compute stream）中。这样通信和计算可以重叠，提高整体吞吐量。来看一个实际的代码片段：

```c
// 假设有4个GPU，rank=1的GPU负责流水线的第2阶段
// 它需要从rank=0接收输入，计算后发给rank=2

cudaStream_t computeStream, commStream;
cudaStreamCreate(&computeStream);
cudaStreamCreate(&commStream);

// 接收前一个阶段的数据（异步）
ncclRecv(inputBuffer, batchSize, ncclFloat32, 0, comm, commStream);

// 在计算流中等待接收完成，然后做前向计算
cudaEvent_t recvDone;
cudaEventCreate(&recvDone);
cudaEventRecord(recvDone, commStream);
cudaStreamWaitEvent(computeStream, recvDone);

// 前向计算（在computeStream中）
forward_kernel<<<grid, block, 0, computeStream>>>(inputBuffer, outputBuffer);

// 发送结果到下一个阶段（在commStream中）
// 先等计算完成
cudaEvent_t computeDone;
cudaEventCreate(&computeDone);
cudaEventRecord(computeDone, computeStream);
cudaStreamWaitEvent(commStream, computeDone);

ncclSend(outputBuffer, batchSize, ncclFloat32, 2, comm, commStream);
```

这个模式在GPT-3、LLaMA等大模型的流水线并行训练中被广泛使用。通过事件（Event）来同步计算流和通信流，避免了全局同步带来的性能损失。

#### 严重警告

千万不要在同一个CUDA流里同时做计算和通信！这会导致计算等待通信完成，或者通信等待计算完成，GPU利用率直接掉到50%以下。必须用两个独立的流，并通过事件来协调。

### 10.3 通信与计算重叠：双缓冲机制

上一节我们用两个流实现了通信和计算的重叠，但还有一个问题：当GPU在做前向计算时，它不能同时发送上一个微批次的结果吗？答案是：可以，但需要双缓冲（Double Buffering）。

双缓冲的核心思想是：准备两块缓冲区，一块用于当前计算，另一块用于通信。当GPU在计算缓冲区A中的数据时，通信引擎可以把缓冲区B中的结果发送给下一个GPU。等计算和通信都完成后，交换缓冲区角色。

来看一个双缓冲的流水线设计图：

```arch-diagram
        ┌─────────────────────────────────────────────────────┐
        │               GPU k (流水线第k阶段)                 │
        │                                                     │
        │  ┌──────────────┐          ┌──────────────┐        │
        │  │  缓冲A       │          │  缓冲B       │        │
        │  │ (计算中)     │          │ (通信中)     │        │
        │  │ 输入: MB_i   │          │ 输入: MB_j   │        │
        │  │ 输出: 结果_i │          │ 输出: 结果_j │        │
        │  └──────────────┘          └──────────────┘        │
        │         │                         │                 │
        │         │ 计算流                  │ 通信流          │
        │         ↓                         ↓                 │
        │  ┌──────────────┐          ┌──────────────┐        │
        │  │ 前向计算核    │          │ ncclSend     │        │
        │  │ (kernel)     │          │ (发送结果_j) │        │
        │  └──────────────┘          └──────────────┘        │
        └─────────────────────────────────────────────────────┘
```

图10-2：双缓冲机制下计算流与通信流的重叠

实现双缓冲的关键是：在代码中维护两个缓冲区指针，并用一个计数器来切换。下面是一个简化的实现：

```c
// 双缓冲流水线阶段代码（伪代码）
float* buffers[2];  // 两个缓冲区
int current = 0;    // 当前使用的缓冲区索引

for (int mb = 0; mb < numMicroBatches; mb++) {
    int next = 1 - current;  // 另一个缓冲区

    // 1. 从上一个阶段接收数据到next缓冲区（异步）
    ncclRecv(buffers[next], batchSize, ncclFloat32, prevRank, comm, commStream);

    // 2. 在计算流中等待接收完成
    cudaStreamWaitEvent(computeStream, recvEvent[next]);

    // 3. 在计算流中执行前向计算，结果写入current缓冲区
    forward_kernel<<<..., computeStream>>>(buffers[next], buffers[current]);

    // 4. 在通信流中等待计算完成
    cudaStreamWaitEvent(commStream, computeEvent[current]);

    // 5. 发送current缓冲区的结果到下一个阶段
    ncclSend(buffers[current], batchSize, ncclFloat32, nextRank, comm, commStream);

    // 6. 交换缓冲区
    current = next;
}
```

这个模式可以让通信和计算完全重叠。在理想情况下，通信开销被“隐藏”在计算时间里，流水线的吞吐量只受限于最慢的那个阶段。实际测试中，使用双缓冲后，4级流水线的吞吐量提升了约30%-40%（在batch size=8、模型参数量1.5B的场景下）。

### 10.4 调度策略：1F1B vs 非对称调度

流水线并行的调度策略直接决定了通信模式和显存占用。目前最主流的两种策略是1F1B（一个前向跟一个后向）和非对称调度（比如PipeDream的调度）。

1F1B调度：每个GPU交替执行一个前向传播和一个后向传播。这样做的优点是显存占用低——你只需要保存当前微批次的中间激活值。但缺点是通信模式变成了“前向-后向交错”，导致通信带宽利用率不高，因为前向和后向的通信方向相反。

非对称调度：允许一个GPU连续执行多个前向传播，然后再连续执行多个后向传播。这样可以批量发送数据，提高通信带宽利用率。但代价是显存占用高，因为你需要保存多个微批次的中间激活值。

我们用一个表格来对比这两种调度策略：

| 调度策略 | 通信模式 | 显存占用 | 带宽利用率 | 适用场景 |
| --- | --- | --- | --- | --- |
| 1F1B | 前向-后向交错 | 低（1个微批次） | 中（约60%） | 显存受限的模型（如LLaMA-65B） |
| 非对称调度 | 批量前向→批量后向 | 高（N个微批次） | 高（约85%） | 带宽受限的模型（如ViT-22B） |
| PipeDream-2BW | 双权重缓冲 | 中（2个权重副本） | 高（约80%） | 需要高吞吐的推理场景 |

在实际项目中，选择哪种调度策略取决于你的瓶颈是显存还是带宽。如果模型大到连一个微批次的激活值都放不下，那就只能用1F1B。如果显存够用但通信带宽是瓶颈，非对称调度更合适。

NCCL本身不提供调度策略的实现，但你可以通过控制`ncclSend`和`ncclRecv`的调用顺序来实现任意调度策略。比如在1F1B中，你需要在每个微批次的前向之后立即安排后向的接收操作。

### 10.5 流水线气泡与通信优化

流水线并行最大的性能杀手是“气泡”（Bubble）。简单来说，气泡就是GPU在等待数据时处于空闲状态的时间。比如流水线刚开始时，GPU3要等GPU0、1、2都算完才能开始工作，这段时间GPU3就是空闲的。

气泡的大小跟微批次数量（m）和流水线深度（p）有关。理论上，气泡占比 = (p-1)/(m+p-1)。当m远大于p时，气泡可以忽略不计。但在实际推理场景中，batch size往往很小（比如在线推理一次只送一个请求），m=1时气泡占比高达(p-1)/p，即4级流水线有75%的气泡！

为了减少气泡，我们可以从通信层面做优化：

- **减少通信延迟**：使用NVLink而不是PCIe，延迟可以从10μs降到1μs。
- **增大通信粒度**：把多个小张量合并成一个大张量一次发送，减少通信次数。
- **预取数据**：在计算当前微批次的同时，提前发起下一个微批次的接收操作。

来看一个实际的通信优化案例：在8卡A100（NVLink 3.0）上跑GPT-3 175B的流水线并行推理，微批次大小=1。优化前，每层的前向计算耗时2ms，但通信耗时0.8ms，气泡占比约28%。优化后，通过合并通信（把多个层的输出合并成一个张量发送）和预取数据，通信耗时降到0.3ms，气泡占比降到13%。

#### 设计要点

在流水线并行中，通信优化的目标是让通信时间小于计算时间，这样通信就可以被计算完全掩盖。如果通信时间大于计算时间，你的流水线就会变成“通信瓶颈”，GPU利用率会急剧下降。此时应该考虑减少流水线深度，或者使用更大的微批次。

### 10.6 异步流水线与NCCL分组通信

前面我们讲的双缓冲和事件同步，本质上都是在实现异步流水线。但NCCL还提供了一个更高级的特性：分组通信（Group Communication）。通过`ncclGroupStart`和`ncclGroupEnd`，你可以把多个NCCL操作打包成一个组，NCCL会尝试在内部优化这些操作的执行顺序。

在流水线并行中，分组通信的一个典型应用是：同时发起前向发送和后向接收。比如，一个GPU在完成前向计算后，需要把结果发给下一个GPU，同时它还要从上一个GPU接收梯度。这两个操作可以放到一个组里，NCCL会尽量让它们并行执行。

```c
// 使用NCCL分组通信同时发起发送和接收
ncclGroupStart();
// 发送前向结果给下一个GPU
ncclSend(outputBuffer, batchSize, ncclFloat32, nextRank, comm, commStream);
// 从上一个GPU接收梯度
ncclRecv(gradBuffer, batchSize, ncclFloat32, prevRank, comm, commStream);
ncclGroupEnd();

// 等待两个操作都完成
cudaStreamSynchronize(commStream);
```

分组通信的好处是：NCCL可以同时调度多个DMA引擎，在NVLink带宽充足的情况下，发送和接收可以真正并行。实测在A100 8卡环境中，使用分组通信比分别调用`ncclSend`和`ncclRecv`，通信吞吐量提升了约15%。

但要注意：分组通信并不是万能的。如果发送和接收都依赖同一个缓冲区（比如你正在发送当前批次的结果，同时又要接收下一个批次的输入），那就不能放到一个组里，因为缓冲区有数据依赖。这时候还是得用事件同步来保证顺序。

### 10.7 梯度累积与通信合并

在流水线并行的后向传播中，每个微批次都会产生一份梯度。如果每个微批次的梯度都单独发送，通信次数会非常多。比如微批次数量m=32，流水线深度p=8，那么每个GPU需要发送32次梯度，每次发送的数据量可能只有几百KB。小数据量的通信效率很低，因为通信的启动延迟（latency）占了主导。

解决方案是梯度累积：在本地累积多个微批次的梯度，然后一次发送。比如累积4个微批次的梯度后再发送，通信次数就从32次降到了8次，通信效率大幅提升。

来看一个梯度累积的通信参数对比：

| 累积步数 | 通信次数 | 单次数据量 | 总通信时间（μs） | 带宽利用率 |
| --- | --- | --- | --- | --- |
| 1（无累积） | 32 | 256 KB | 3200 | 25% |
| 4 | 8 | 1 MB | 1200 | 68% |
| 8 | 4 | 2 MB | 900 | 85% |

从表格可以看出，累积步数从1增加到4，总通信时间从3200μs降到了1200μs，降幅超过60%。但继续增加到8步，收益就变小了，因为单次数据量增大后，带宽利用率已经接近上限。

在NCCL中实现梯度累积很简单：在本地维护一个梯度累积缓冲区，每次后向计算后把梯度累加进去，当累积步数达到阈值时，调用`ncclSend`发送累积后的梯度，然后清零缓冲区。

### 10.8 实际案例：8卡流水线并行推理优化

最后我们用一个完整的案例来总结本章的内容。假设我们要在8张A100 GPU上部署一个LLaMA-65B模型做在线推理。模型有60层Transformer，我们切成8段，每段7-8层。推理的batch size=1（在线请求）。

初始配置：使用1F1B调度，单缓冲，无梯度累积（推理不需要梯度）。实测吞吐量只有120 tokens/s，GPU利用率45%。瓶颈分析发现，气泡占比高达70%，因为batch size=1，微批次数量m=1，气泡占比 = (8-1)/(1+8-1) = 87.5%。

优化步骤：

1. **改用非对称调度**：允许每个GPU连续处理多个请求（比如4个请求），然后批量发送结果。这样m=4，气泡占比降到 (8-1)/(4+8-1)=63%。吞吐量提升到180 tokens/s。
2. **引入双缓冲**：让通信和计算重叠。在连续处理4个请求的场景下，双缓冲让通信时间被计算掩盖。吞吐量提升到220 tokens/s。
3. **使用分组通信**：把前向发送和后向接收（虽然推理没有后向，但这里指接收下一个请求的输入）打包成组。吞吐量提升到240 tokens/s。
4. **合并通信**：把每层之间的输出合并成一个张量发送。原来每段有7层，需要7次通信，合并后只需要1次。通信次数从7次降到1次，通信延迟从0.5ms降到0.1ms。最终吞吐量达到270 tokens/s。

这个案例说明，流水线并行的通信优化是一个系统工程，需要从调度策略、缓冲机制、分组通信、数据合并等多个维度入手。每个优化点都能带来10%-30%的提升，叠加起来效果非常可观。

#### 章节小结

本章深入讲解了流水线并行中的通信设计，从F-t-S通信模式出发，详细介绍了ncclSend/ncclRecv原语、双缓冲机制、1F1B与非对称调度策略、气泡优化、分组通信和梯度累积等关键技术。核心思路是通过异步化和合并通信来隐藏通信延迟，让GPU尽可能保持忙碌。实际案例表明，通过综合运用这些优化手段，流水线并行的推理吞吐量可以提升2倍以上。

#### 本章重点回顾

- **F-t-S通信模式**：前向传数据、后向传梯度，是流水线并行的通信基础，必须成对使用ncclSend/ncclRecv。
- **双缓冲机制**：用两个缓冲区交替进行计算和通信，让通信时间被计算时间掩盖，是提升吞吐量的核心手段。
- **调度策略选择**：1F1B显存友好但带宽利用率低，非对称调度带宽利用率高但显存占用大，需根据瓶颈选择。
- **气泡优化**：通过增大微批次数量、合并通信、预取数据来减少GPU空闲时间。
- **分组通信**：使用ncclGroupStart/End同时发起多个通信操作，让DMA引擎并行工作，提升带宽利用率。
