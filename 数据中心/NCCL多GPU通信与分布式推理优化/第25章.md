# 第25章：混合并行通信方案设计

在大规模分布式训练中，单一的数据并行或模型并行往往无法满足极致效率的需求。混合并行通过组合多种并行策略，让通信模式变得复杂多变。本章的核心思路是：如何根据模型结构、集群拓扑和带宽约束，设计一套高效的混合并行通信方案。我们将从通信拓扑的编排入手，讨论如何将数据并行、张量并行和流水线并行的通信流量在同一个集群中合理调度，避免通信冲突和带宽争抢。重点会放在NCCL通信域的划分、通信流量的优先级管理，以及如何通过异步通信和计算-通信重叠来隐藏通信延迟。这些技术在实际部署千亿参数模型时是必须掌握的。

### 25.1 混合并行中的通信域划分

混合并行的第一步，就是要把GPU集群划分成不同的通信域。每个并行策略对应一个独立的通信域，比如数据并行域、张量并行域和流水线并行域。在NCCL里，每个通信域就是一个独立的NCCL通信器（communicator）。划分的关键在于：同一个GPU可能要同时属于多个通信域，这就要求我们在创建通信器时，给每个GPU分配不同的rank映射关系。

来看一个典型的4节点8卡（共32卡）的集群配置。假设我们要训练一个Transformer模型，采用混合并行：张量并行度为4（跨4卡），流水线并行度为4（跨4个stage），数据并行度为2（复制2份模型副本）。那么每个GPU需要加入3个通信域。我们通过设置NCCL的`NCCL_COMM_ID`和`CUDA_VISIBLE_DEVICES`来创建不同的通信组。实际工程中，我们一般用框架（如Megatron-LM）自动生成这些配置，但理解底层的rank映射关系很重要。

| 并行策略 | 通信域大小 | 通信域数量 | 通信操作 | 典型延迟要求 |
| --- | --- | --- | --- | --- |
| 数据并行 | 16 | 2 | AllReduce | < 50us |
| 张量并行 | 4 | 8 | AllGather / ReduceScatter | < 10us |
| 流水线并行 | 4 | 8 | P2P Send/Recv | < 100us |

上表给出了一个32卡集群的通信域划分示例。数据并行域最大，因为要同步16个GPU的梯度；张量并行域最小但数量最多，因为每个Transformer层内部的张量并行操作都需要快速完成；流水线并行域则负责stage之间的点对点通信。在NCCL中，我们通过`ncclCommInitRank`来初始化这些通信器，每个通信器内部使用独立的环（ring）或树（tree）拓扑。

#### 设计要点

划分通信域时，要优先保证张量并行域的通信延迟最低。因为张量并行是计算密集型的，通信延迟会直接影响单层计算时间。通常把张量并行域限制在同一个NVSwitch域内，利用NVLink的高带宽（单向600GB/s）和低延迟（~1us）。数据并行域可以跨节点，但要注意网卡带宽瓶颈。

### 25.2 通信流量建模与冲突分析

有了通信域的划分，下一步就是分析不同通信流之间是否存在冲突。在混合并行中，多个通信域可能同时发起通信操作，这些操作会争抢同一个NVLink端口或同一个网卡。如果不做控制，就会导致带宽分配不均，通信延迟剧烈抖动。

我们用一个实际案例来说明。在32卡集群上运行混合并行训练，每个GPU同时参与：数据并行的AllReduce（跨节点）、张量并行的AllGather（节点内）和流水线并行的P2P Send（跨节点）。如果不对这些通信流进行编排，就会出现如下冲突：数据并行的AllReduce流量和张量并行的AllGather流量同时使用NVLink，导致张量并行通信延迟从3us飙升到15us，单层计算时间增加30%。

解决思路是给通信流分配优先级。NCCL提供了`ncclCommSplit`接口，可以将一个大的通信器拆分成多个优先级不同的子通信器。高优先级的通信（如张量并行）先占用带宽，低优先级的通信（如数据并行）在空闲时再进行。我们还可以通过`NCCL_LAUNCH_MODE`和`NCCL_MAX_CTAS_PER_DEVICE`来调整通信kernel的调度策略。

```arch-diagram
        时间轴 ────────────────────────────────────────→

        无优先级编排：
        TP AllGather:    ┌──────┐    ┌──────┐    ┌──────┐
                         │ 3us  │    │ 15us │    │ 3us  │
                         └──────┘    └──────┘    └──────┘
        DP AllReduce:         ┌──────────────┐     ┌──────────────┐
                             │  20us        │     │  20us        │
                             └──────────────┘     └──────────────┘
                         (冲突导致TP延迟变大)

        有优先级编排：
        TP AllGather:    ┌──────┐┌──────┐┌──────┐┌──────┐
                         │ 3us  ││ 3us  ││ 3us  ││ 3us  │
                         └──────┘└──────┘└──────┘└──────┘
        DP AllReduce:              ┌──────────────┐
                                  │  20us        │
                                  └──────────────┘
                         (TP优先，DP在空闲窗口执行)
```

图25-1：通信流优先级编排前后的延迟对比

从图中可以清楚看到，没有优先级编排时，TP通信被DP通信打断，延迟增加了5倍。通过优先级编排，TP通信可以连续执行，DP通信被推迟到TP空闲的时间窗口。在工程实现上，我们通过`ncclCommSplit`创建两个通信器：`comm_high`和`comm_low`，然后在代码中先调用高优先级通信器的操作，再调用低优先级的。

### 25.3 计算-通信重叠的工程实现

混合并行中，通信操作通常不会占满整个时间轴，我们可以利用计算和通信的并行执行来隐藏通信延迟。NCCL的通信操作是异步的，调用`ncclAllReduce`后，CPU会立即返回，GPU会继续执行后续的CUDA kernel。但要注意，如果后续的kernel需要用到通信结果，就必须显式同步。

在混合并行中，我们利用这个特性来实现计算-通信重叠。以流水线并行为例，每个stage在计算完当前micro batch的forward后，立即发起P2P Send将激活值传给下一个stage，同时继续计算下一个micro batch。这样，通信和计算就在时间上重叠了。NCCL的P2P操作使用GPU的RDMA引擎，不占用计算资源，所以重叠效果很好。

下面是一个使用CUDA事件和NCCL通信重叠的代码示例。这段代码展示了如何在计算完一部分数据后，立即启动通信，而不等待所有计算完成。

```cpp
// 混合并行中的计算-通信重叠示例
// 假设有4个micro batch，每个batch计算后立即发送

cudaStream_t compute_stream, comm_stream;
cudaStreamCreate(&compute_stream);
cudaStreamCreate(&comm_stream);

ncclComm_t comm;
// ... 初始化ncclComm ...

for (int i = 0; i < 4; i++) {
    // 在计算流上执行forward计算
    forward_kernel<<<grid, block, 0, compute_stream>>>(input[i], output[i]);

    // 在计算流上记录事件
    cudaEvent_t event;
    cudaEventCreate(&event);
    cudaEventRecord(event, compute_stream);

    // 在通信流上等待计算完成
    cudaStreamWaitEvent(comm_stream, event, 0);

    // 在通信流上发起异步发送
    ncclSend(output[i], size, ncclFloat32, peer_rank, comm, comm_stream);

    // 计算流继续计算下一个batch
    // 不需要等待ncclSend完成
}

// 最后同步所有流
cudaStreamSynchronize(compute_stream);
cudaStreamSynchronize(comm_stream);
```

这段代码的关键在于：计算流和通信流是独立的，通过CUDA事件来同步。计算流完成forward后，通信流立即接管发送操作，而计算流可以继续下一个batch的计算。这样，通信延迟就被隐藏在了计算过程中。在实际项目中，我们通常使用`cudaMemcpyAsync`和`ncclSend`的组合，让数据在GPU之间直接传输，不经过CPU。

#### 严重警告

不要在所有计算完成后再统一通信。那样做会让通信变成串行的，导致GPU空闲等待。正确的做法是：每计算完一个micro batch，就立即发起通信，让通信和计算流水线化。否则，通信延迟会直接叠加到训练时间上，降低吞吐量。

### 25.4 张量并行与数据并行的融合通信

在混合并行中，张量并行和数据并行经常同时存在。张量并行需要对每个Transformer层的输出做AllGather，数据并行需要对梯度做AllReduce。如果这两个操作分开做，就会产生两次全局同步，浪费带宽。一个优化技巧是将它们融合成一个操作。

具体来说，张量并行的AllGather会产生一个完整的张量，这个张量在数据并行域中需要做AllReduce。我们可以将AllGather和AllReduce合并成一个ReduceScatter + AllGather的组合操作，或者直接使用NCCL的`ncclAllReduce`在更大的通信域上完成。但这样会破坏并行粒度，需要权衡。

在实际的Megatron-LM框架中，他们采用了一种叫"分布式优化器"的技术：每个GPU只维护一部分优化器状态，通过ReduceScatter将梯度分散到各个GPU，然后每个GPU独立更新自己的参数分片，最后通过AllGather收集完整参数。这样就把数据并行的AllReduce和张量并行的AllGather融合在了一起，减少了通信量。

下面是一个融合通信的示意图，展示了数据流的变化。

```arch-diagram
        传统方式：
        ┌──────────┐     ┌──────────┐     ┌──────────┐
        │ TP计算   │────→│ AllGather│────→│ AllReduce│────→ 完整参数
        └──────────┘     └──────────┘     └──────────┘
                         (张量并行域)     (数据并行域)

        融合方式：
        ┌──────────┐     ┌──────────────┐     ┌──────────┐
        │ TP计算   │────→│ ReduceScatter│────→│ AllGather│────→ 完整参数
        └──────────┘     └──────────────┘     └──────────┘
                         (跨两个域)           (跨两个域)
```

图25-2：传统方式与融合通信方式的数据流对比

融合通信的核心思想是：在同一个通信操作中完成多个并行策略的同步需求。这要求我们重新设计通信域的拓扑，让一个通信器同时覆盖张量并行和数据并行的所有GPU。在NCCL中，我们可以创建一个全局通信器，然后通过`ncclCommSplit`划分出子组，但融合操作需要自定义kernel来实现。

### 25.5 流水线并行中的异步通信调度

流水线并行是混合并行中最容易产生通信空闲的环节。因为每个stage只能等前一个stage的数据到了才能开始计算，如果通信延迟大，就会导致流水线气泡。解决方法是使用异步通信调度，让每个stage提前预取下一份数据。

在NCCL中，P2P的Send/Recv操作是异步的。我们可以让每个stage在计算当前micro batch的同时，提前发起下一个micro batch的Recv操作。这样，当当前batch计算完成时，下一个batch的数据已经传输完毕，可以直接开始计算。这个技巧在GPipe和PipeDream的实践中都有应用。

具体实现时，我们使用双缓冲（double buffering）技术：分配两个缓冲区，一个用于当前计算，一个用于预取通信。计算和通信交替使用这两个缓冲区，实现流水线化。下面是一个双缓冲的配置示例。

| 时间步 | Stage 1 操作 | Stage 2 操作 | 通信操作 | 缓冲区状态 |
| --- | --- | --- | --- | --- |
| t0 | 计算 batch0 | 空闲 | 无 | buf0: 计算中 |
| t1 | 计算 batch1 | 接收 batch0 | Send(batch0) | buf0: 发送中 buf1: 计算中 |
| t2 | 计算 batch2 | 计算 batch0 | Send(batch1) | buf0: 空闲 buf1: 计算中 |
| t3 | 计算 batch3 | 计算 batch1 | Send(batch2) | buf0: 计算中 buf1: 发送中 |

上表展示了双缓冲的工作流程。Stage 1在t1时刻计算batch1的同时，Stage 2正在接收batch0。这样，Stage 2在t2时刻就可以直接开始计算batch0，不需要等待通信。通过这种方式，流水线的气泡被压缩到了最小。在NCCL中，我们通过`ncclRecv`的异步调用和CUDA事件的同步来实现这个调度。

### 25.6 通信拓扑的自适应选择

混合并行中，不同并行策略对通信拓扑的偏好不同。张量并行喜欢低延迟的环拓扑，数据并行喜欢高带宽的树拓扑。NCCL支持多种拓扑算法，包括Ring、Tree、CollNet等。在混合并行场景下，我们不应该对所有通信域使用同一种拓扑，而应该根据通信特征自适应选择。

NCCL提供了`NCCL_ALGO`和`NCCL_PROTO`环境变量来指定算法和协议。但更精细的控制是通过`ncclCommSplit`的`ncclConfig_t`结构体，可以为每个子通信器设置不同的算法偏好。例如，张量并行域设置`NCCL_ALGO_RING`，数据并行域设置`NCCL_ALGO_TREE`。

在实际项目中，我们通过性能测试来确定最优拓扑。在训练开始前，先运行一个简短的profiling阶段，测试不同拓扑下的通信延迟和带宽，然后选择最优配置。这个profiling过程可以利用NCCL的`ncclGroupStart`和`ncclGroupEnd`来批量测试多个配置。

#### 信息

在NVLink全互联的节点内，Ring拓扑的延迟最低（因为跳数少）；但在跨节点的InfiniBand网络中，Tree拓扑的带宽利用率更高。混合并行中，张量并行通常限制在节点内，所以用Ring；数据并行跨节点，用Tree。这个配置可以显著提升整体吞吐量。

### 25.7 多流通信的带宽分配策略

当多个通信流同时存在时，如何公平分配带宽是一个工程难题。NCCL内部使用一种基于credit的流控机制，每个通信流有固定的credit数量，用完就必须等待。但在混合并行中，不同通信流的优先级不同，我们需要手动调整credit分配。

我们可以通过`NCCL_BUFFSIZE`和`NCCL_NTHREADS`来间接控制带宽分配。增大`NCCL_BUFFSIZE`可以让单个通信操作占用更多带宽，但会阻塞其他操作。更精细的控制是通过`ncclConfig_t`中的`minCTAs`和`maxCTAs`参数，限制每个通信操作使用的SM数量。

下面是一个实际调优案例。在一个8卡节点上运行混合并行，发现数据并行的AllReduce占用了60%的NVLink带宽，导致张量并行的AllGather延迟从2us飙升到8us。通过将数据并行的`maxCTAs`从8降低到4，限制了它的带宽占用，张量并行延迟恢复到3us，整体吞吐量提升了15%。

```arch-diagram

        │  DP AllReduce: ████████████████████ 60%        │
        │  TP AllGather: ██████████ 30%                  │
        │  PP P2P:       ████ 10%                        │
        └────────────────────────────────────────────────┘

        带宽分配比例（优化后）：
        ┌────────────────────────────────────────────────┐
        │  DP AllReduce: ██████████ 30%                  │

        └────────────────────────────────────────────────┘
```

从图中可以看出，通过限制数据并行的带宽，我们把更多资源让给了延迟敏感的张量并行。这个调整虽然降低了数据并行的通信速度，但因为张量并行是计算的关键路径，整体训练速度反而提升了。这就是混合并行中带宽分配的权衡艺术。

### 25.8 混合并行通信的Profiling与调优工具

设计完通信方案后，必须通过profiling来验证效果。NCCL提供了`NCCL_DEBUG`和`NCCL_DEBUG_SUBSYS`环境变量，可以输出详细的通信日志。但更专业的工具是NVIDIA的Nsight Systems和Nsight Compute，它们可以可视化GPU kernel的执行时间线和通信操作。

在混合并行场景下，我们关注三个关键指标：通信延迟、带宽利用率和计算-通信重叠度。Nsight Systems可以显示每个GPU上计算流和通信流的时间线，让我们直观地看到是否有空闲气泡。如果发现通信操作之间有空隙，说明重叠做得不够好，需要调整micro batch的大小或通信调度顺序。

还有一个实用技巧是使用NCCL的`ncclCommGetAsyncError`来检测通信错误。在混合并行中，由于通信域复杂，很容易出现资源竞争导致的超时错误。通过定期检查异步错误，可以提前发现通信瓶颈。下面是一个profiling脚本的示例，用于收集每个通信域的延迟数据。

```bash
# 混合并行通信profiling脚本
# 设置NCCL调试级别
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=INIT,GRAPH,ENV

# 设置使用Ring算法
export NCCL_ALGO=RING

# 运行训练任务，捕获日志
mpirun -np 32 ./train_mixed_parallel.sh 2>&1 | tee nccl_profile.log

# 从日志中提取通信延迟
grep "ncclAllReduce" nccl_profile.log | awk '{print $5}' > allreduce_latency.txt
grep "ncclAllGather" nccl_profile.log | awk '{print $5}' > allgather_latency.txt

# 计算平均延迟
echo "Average AllReduce latency:"
awk '{sum+=$1; count++} END {print sum/count " us"}' allreduce_latency.txt

echo "Average AllGather latency:"
awk '{sum+=$1; count++} END {print sum/count " us"}' allgather_latency.txt
```

这个脚本可以帮助我们快速获取通信延迟数据。在实际调优中，我们通常会在不同配置下运行这个脚本，然后对比延迟数据，选择最优方案。记住，混合并行的调优是一个迭代过程，不要期望一次就能找到最佳配置。

### 25.9 实际项目中的混合并行部署案例

最后，我们来看一个实际项目的部署案例。某公司训练一个175B参数的GPT模型，使用512张A100 GPU，采用混合并行：张量并行度8，流水线并行度8，数据并行度8。集群拓扑是8节点，每节点8卡，节点内NVLink全互联，节点间用4条200Gbps InfiniBand连接。

初始配置下，训练吞吐量只有120 TFLOPS/GPU，远低于理论峰值312 TFLOPS。通过profiling发现，问题出在通信域划分不合理：张量并行域跨了节点，导致NVLink和InfiniBand混合使用，延迟从2us飙升到20us。解决方案是将张量并行限制在节点内，只使用NVLink通信。调整后，张量并行延迟降到3us，吞吐量提升到210 TFLOPS/GPU。

进一步优化发现，数据并行的AllReduce占用了大量InfiniBand带宽，导致流水线并行的P2P通信出现排队。通过给流水线并行分配更高的优先级（使用`ncclCommSplit`），并限制数据并行的带宽（设置`NCCL_BUFFSIZE=128K`），最终吞吐量达到265 TFLOPS/GPU，接近理论峰值的85%。

#### 设计要点

在部署大规模混合并行时，一定要先做拓扑映射：把张量并行放在节点内，流水线并行尽量放在同一机架内，数据并行可以跨机架。这个映射关系决定了通信延迟的底线。另外，每个节点的GPU数量最好是2的幂次，方便划分通信域。

### 25.10 混合并行通信的未来趋势

随着模型规模继续增长，混合并行的通信方案也在不断演进。一个趋势是使用全对全（All-to-All）通信来替代传统的AllGather和ReduceScatter，因为All-to-All在特定拓扑下可以更好地利用带宽。NCCL正在开发新的All-to-All算法，预计在未来的版本中会支持。

另一个趋势是引入可编程的通信调度器，让用户自定义通信流的优先级和带宽分配。NCCL的`ncclConfig_t`结构体已经预留了扩展字段，未来可能会支持更细粒度的QoS控制。对于工程师来说，理解这些底层机制，才能在未来的框架升级中快速适应。

混合并行通信方案设计是分布式训练中最具挑战性的环节之一。本章从通信域划分入手，讨论了如何根据并行策略创建独立的NCCL通信器，并分析了通信流量冲突的解决方法。我们深入探讨了计算-通信重叠的工程实现，给出了双缓冲和异步调度的具体代码。通过实际案例，展示了如何通过调整带宽分配和拓扑选择来优化通信性能。最后，介绍了profiling工具的使用和未来发展趋势。掌握这些技术，你就能在实际项目中设计出高效的混合并行通信方案。

#### 本章重点回顾

- **通信域划分**：每个并行策略对应一个独立的NCCL通信器，通过`ncclCommInitRank`创建，张量并行域优先使用NVLink。
- **通信流优先级**：使用`ncclCommSplit`创建高/低优先级通信器，确保延迟敏感的张量并行优先占用带宽。
- **计算-通信重叠**：通过双缓冲和异步通信，让通信延迟隐藏在计算过程中，减少GPU空闲时间。
- **融合通信**：将AllGather和AllReduce合并为ReduceScatter+AllGather，减少全局同步次数。
- **拓扑自适应**：张量并行用Ring拓扑（低延迟），数据并行用Tree拓扑（高带宽），通过`NCCL_ALGO`配置。
