# 第20章：多节点RDMA网络配置

多节点分布式训练的核心瓶颈往往不在GPU算力，而在节点间的网络通信。RDMA（远程直接内存访问）技术允许GPU的数据绕过CPU和操作系统内核，直接从一台机器的显存传输到另一台机器的显存，延迟从毫秒级降到微秒级。本章我们聚焦于RDMA网络的工程配置，包括网卡选型、驱动安装、链路协商、RoCE与InfiniBand的差异、以及NCCL对RDMA的适配参数调优。我们会用实际集群的配置案例，带你一步步搭建一个可用的多节点RDMA通信环境，并验证带宽是否达标。

### 20.1 RDMA网卡选型与硬件环境准备

在开始配置之前，先搞清楚硬件选型。目前主流的多节点GPU训练集群，RDMA网络主要分两条路线：InfiniBand和RoCE（RDMA over Converged Ethernet）。InfiniBand走的是专用网络，典型代表是Mellanox（现NVIDIA）的ConnectX系列网卡和Quantum交换机，端到端延迟可以做到1微秒以内，带宽从100Gbps到400Gbps。RoCE则跑在标准以太网上，成本更低，但需要交换机支持PFC（优先级流控制）和ECN（显式拥塞通知）才能保证无损。

我们以一个4节点、每节点8卡A100的集群为例。每个节点配一块Mellanox ConnectX-6 HDR网卡，单端口200Gbps。注意，如果节点内GPU通过NVSwitch互联，跨节点必须走RDMA，那么网卡的PCIe带宽不能成为瓶颈。ConnectX-6使用PCIe 4.0 x16，理论带宽约32GB/s，跟200Gbps（约25GB/s）匹配。选型时还要看网卡是否支持GPUDirect RDMA——这个功能允许GPU显存直接与网卡交换数据，不需要经过CPU内存中转。NCCL的通信效率很大程度上依赖这个特性。

硬件安装方面，网卡插槽要选直连CPU的PCIe插槽，不要走PCH桥接，否则延迟会增加。光纤线缆使用QSFP56模块，注意区分单模和多模。同一集群内，所有节点的网卡固件版本必须一致，否则链路协商可能失败。我们遇到过因为固件版本差一个补丁号，导致200G链路只能跑到40G的情况。

| 参数项 | InfiniBand HDR | RoCE v2 (100G) |
| --- | --- | --- |
| 典型网卡 | ConnectX-6 HDR | ConnectX-5 100G |
| 单端口带宽 | 200 Gbps | 100 Gbps |
| 端到端延迟 | 0.7 us | 1.5 us |
| 交换机类型 | 专用IB交换机 | 支持PFC/ECN的以太网交换机 |
| GPUDirect支持 | 原生支持 | 需配置 |
| 每端口成本（含光模块） | 约 $2000 | 约 $800 |

选型没有绝对的好坏。如果团队已经有成熟的以太网运维经验，RoCE可以省成本；如果追求极致性能和稳定性（比如训练千亿参数大模型），InfiniBand是更稳妥的选择。我们下面的配置以InfiniBand为例，RoCE的配置差异会在20.4节单独说明。

### 20.2 驱动安装与固件升级

网卡插好后，第一步是安装驱动。Mellanox网卡的驱动叫MLNX\_OFED（OpenFabrics Enterprise Distribution），可以从NVIDIA官网下载对应操作系统版本的包。安装前确认内核版本，因为OFED会编译内核模块。我们用的Ubuntu 22.04，内核5.15，下载MLNX\_OFED\_LINUX-5.8-x86\_64.tgz。

安装命令很简单：

```bash
# 解压
tar xzf MLNX_OFED_LINUX-5.8-ubuntu22.04-x86_64.tgz
cd MLNX_OFED_LINUX-5.8-ubuntu22.04-x86_64

# 运行安装脚本
sudo ./mlnxofedinstall --all --force

# 安装完成后重启
sudo reboot
```

安装过程中会编译内核模块，如果报错缺少依赖，按提示安装对应包就行。重启后，用`ibstat`命令检查网卡状态：

```bash
ibstat

CA 'mlx5_0'
    CA type: MT4125
    Number of ports: 1
    Firmware version: 20.32.1010
    Node guid: 0x248a070300a1b2c3
    System image GUID: 0x248a070300a1b2c3
    Port 1:
        State: Active
        Physical state: LinkUp
        Rate: 200
        Base lid: 1
        LMC: 0
        SM lid: 1
        Capability mask: 0x2651e84a
```

关键看`State: Active`和`Rate: 200`（表示200Gbps）。如果State显示`Down`，检查光模块是否插紧、交换机端口是否启用。固件版本建议升级到厂商推荐的最新稳定版。固件升级工具在OFED包里：

```bash
# 查看当前固件版本
flint -d /dev/mst/mt4125_pciconf0 query

# 升级固件
flint -d /dev/mst/mt4125_pciconf0 -i fw-ConnectX6HDR-20_32_1010.bin burn
```

#### 严重警告

固件升级过程中绝对不能断电或重启，否则网卡可能变砖。建议在维护窗口操作，并且先备份当前固件。

驱动和固件就绪后，我们验证一下GPUDirect是否生效。运行`nvidia-smi topo -m`，看GPU和网卡的PCIe拓扑关系，确保网卡和GPU在同一个NUMA节点下。

### 20.3 子网管理器（SM）配置

InfiniBand网络需要子网管理器（Subnet Manager，SM）来负责路由计算和链路管理。每个IB子网必须至少有一个SM运行。通常，我们在其中一个节点上运行OpenSM服务，其他节点作为普通端节点。如果SM挂了，整个IB网络会瘫痪，所以生产环境建议部署主备SM。

安装OpenSM（OFED自带）：

```bash
sudo apt install opensm
# 启动服务
sudo systemctl enable opensm
sudo systemctl start opensm

# 查看SM状态
ibsmutil -s

# 输出示例
SM state: MASTER
SM priority: 15
SM guid: 0x248a070300a1b2c3
SM port: 1
SM LID: 1
```

SM启动后，它会自动发现所有端节点，分配LID（本地标识符），并计算最优路径。你可以用`ibnetdiscover`命令查看整个IB网络的拓扑：

```bash
ibnetdiscover

# 输出示例（简化）
Switch 0x248a070300d4e5f6 "HDR Switch"
    LID: 2
    Ports: 36
    [1] -> HCA 0x248a070300a1b2c3 "node1" port 1 (LID 1)
    [2] -> HCA 0x248a070300a1b2d4 "node2" port 1 (LID 3)
    [3] -> HCA 0x248a070300a1b2e5 "node3" port 1 (LID 4)
    [4] -> HCA 0x248a070300a1b2f6 "node4" port 1 (LID 5)
```

如果拓扑里看不到某个节点，检查那个节点的网卡是否Active，或者SM的轮询周期还没完成。SM默认每5秒扫描一次，可以修改`/etc/opensm/opensm.conf`里的`polling_time`参数。

对于多交换机的大型集群，SM的路由算法可以选`minhop`（最小跳数）或`updn`（避免环路）。我们4节点单交换机，用默认的minhop就行。

### 20.4 RoCE v2 无损网络配置要点

如果用的是RoCE而不是InfiniBand，配置重点就变成了保证网络“无损”。RoCE依赖标准以太网，但标准以太网会丢包，而RDMA一旦丢包，性能会断崖式下降。所以必须开启交换机和网卡的PFC（优先级流控制）和ECN（显式拥塞通知）。

PFC的作用是：当接收端缓冲区快满时，发送端暂停发送，防止丢包。配置PFC需要在交换机的每个端口上，为RDMA流量（通常用优先级3）启用流控。网卡侧也要对应设置：

```bash
# 设置网卡优先级3启用PFC
mlnx_qos -i ens1f0 -p 3,0,0,0,0,0,0,0

# 查看PFC配置
mlnx_qos -i ens1f0 --show

# 输出示例
PFC Configuration:
    Priority 0: Disabled
    Priority 1: Disabled
    Priority 2: Disabled
    Priority 3: Enabled
    Priority 4: Disabled
    ...
```

ECN则是在交换机检测到拥塞时，在IP头中打标记，通知发送端主动降速。配置ECN需要交换机支持DCBX协议，并且网卡开启DCTCP（数据中心TCP）模式。具体参数因交换机品牌而异，以Cisco Nexus 9000为例：

```bash
# 交换机配置示例（Cisco NX-OS）
interface ethernet 1/1
    priority-flow-control mode on
    priority-flow-control priority 3
    no shutdown

# 启用ECN
class-map type qos match-all RDMA
    match dscp 26
policy-map type qos ECN
    class RDMA
        set qos-group 1
        random-detect ecn
    class class-default
        set qos-group 0
```

RoCE的配置比InfiniBand繁琐得多，而且一旦PFC或ECN参数不对，训练时会出现间歇性性能抖动。我们建议：如果预算允许，直接上InfiniBand，省下的运维时间远比硬件差价值钱。

### 20.5 NCCL的RDMA适配参数调优

硬件和网络就绪后，NCCL需要一些环境变量来充分利用RDMA。最关键的参数是`NCCL_IB_HCA`，它指定NCCL使用哪块IB网卡。如果节点有多块网卡，必须明确指定，否则NCCL可能选错：

```bash
# 指定使用mlx5_0网卡
export NCCL_IB_HCA=mlx5_0

# 如果有多块，可以用逗号分隔，或者用通配符
export NCCL_IB_HCA=mlx5_0,mlx5_1

# 禁用RoCE（如果同时有IB和RoCE）
export NCCL_IB_DISABLE=0
export NCCL_NET_GDR_LEVEL=5   # 启用GPUDirect RDMA
```

`NCCL_NET_GDR_LEVEL`控制GPUDirect RDMA的启用级别。值从0到5，5表示完全启用。如果设置后通信报错，降级到4或3，逐步排查。另一个重要参数是`NCCL_IB_TIMEOUT`，它控制RDMA操作的超时时间，单位是毫秒。默认值可能偏小，在大规模集群下容易超时：

```bash
# 设置超时时间为20秒
export NCCL_IB_TIMEOUT=20

# 设置重试次数
export NCCL_IB_RETRY_CNT=7

# 设置每个连接的QP数量（影响带宽）
export NCCL_IB_QPS_PER_CONNECTION=8
```

我们用一个实际测试来验证参数效果。在4节点上运行NCCL的`all_reduce_perf`基准测试：

```bash
# 在4个节点上分别运行（假设hostfile已配置）
mpirun -np 32 -hostfile hosts -x NCCL_IB_HCA=mlx5_0 \
    -x NCCL_NET_GDR_LEVEL=5 \
    -x NCCL_IB_TIMEOUT=20 \
    -x NCCL_IB_QPS_PER_CONNECTION=8 \
    ./build/all_reduce_perf -b 1M -e 1G -f 2

# 输出示例（128M消息）
#   size       time   algbw   busbw
#   128M       156us  820.5   1538.5 GB/s
```

注意看`busbw`，它表示总线带宽。4节点32卡，理论总带宽是32 × 200Gbps = 6.4Tbps，但实际受限于PCIe和NVSwitch，128M消息的busbw能到1.5TB/s已经不错了。如果远低于这个值，比如只有200GB/s，说明RDMA配置有问题，需要检查链路速率或GPUDirect是否生效。

### 20.6 链路聚合与多网卡绑定

单节点单块200G网卡，对8卡A100来说可能不够。A100的NVSwitch带宽是600GB/s，跨节点通信如果只有25GB/s（200Gbps），会成为瓶颈。解决方案是每个节点插两块或四块网卡，做链路聚合。NCCL支持多网卡并行通信，通过`NCCL_IB_HCA`指定多块网卡即可：

```bash
# 使用两块网卡
export NCCL_IB_HCA=mlx5_0,mlx5_1

# 或者使用通配符匹配所有mlx5设备
export NCCL_IB_HCA=^mlx5_
```

NCCL会自动在多个网卡之间做负载均衡，不需要额外的bonding配置。但要注意，多网卡必须连接到不同的交换机上行，否则单交换机故障会导致整个集群通信中断。网络拓扑设计上，我们推荐使用Fat-Tree结构，每个节点的每块网卡连接到不同的叶子交换机，叶子交换机再通过脊交换机互联。

下面是一个4节点、每节点2块网卡的拓扑示意：

```arch-diagram
                    ┌─────────────────┐
                    │   Spine Switch  │
                    │   (2台冗余)      │
                    └────────┬────────┘
                             │
          ┌──────────────────┼──────────────────┐
          │                  │                  │
   ┌──────┴──────┐   ┌──────┴──────┐   ┌──────┴──────┐
   │ Leaf Sw 1   │   │ Leaf Sw 2   │   │ Leaf Sw 3   │
   │ (32口 HDR)  │   │ (32口 HDR)  │   │ (32口 HDR)  │
   └──────┬──────┘   └──────┬──────┘   └──────┬──────┘
          │                  │                  │
    ┌─────┼─────┐      ┌─────┼─────┐      ┌─────┼─────┐
    │     │     │      │     │     │      │     │     │
  Node1  Node2 Node3  Node1  Node2 Node3  Node1  Node2 Node3
  port1  port1 port1  port2  port2 port2  port1  port1 port1
```

图：4节点双网卡Fat-Tree拓扑，每块网卡连不同Leaf交换机

配置多网卡后，用`ibstat`检查每块网卡的状态，确认它们都处于Active且Rate为200。然后用NCCL测试看聚合带宽是否线性增长。如果两块网卡，busbw应该接近单网卡的两倍。

### 20.7 常见问题排查与性能验证

配置完成后，我们整理几个最常遇到的坑和排查方法。

**问题1：ibstat显示LinkUp，但NCCL通信失败。**
检查防火墙是否阻断了IB通信。IB使用IPoIB（IP over InfiniBand）时，也需要配置IP地址。但NCCL走的是原生IB协议，不依赖IP，所以IP配置不是必须的。不过，如果`ib0`接口没有IP，某些管理工具会报错。建议给每个节点的IB接口配一个内部IP，方便ssh和调试。

**问题2：GPUDirect RDMA不生效。**
运行`nvidia-smi topo -m`，确认GPU和网卡在同一PCIe根复合体下。如果GPU在CPU0的PCIe域，网卡在CPU1的域，跨NUMA访问会导致GPUDirect失效。这种情况下，要么换插槽，要么设置`NCCL_NET_GDR_LEVEL=4`降级使用。

**问题3：多网卡负载不均。**
用`perfquery`或`ibdiagnet`查看每条链路的流量。如果某块网卡流量明显偏低，检查SM的路由策略是否把流量都分配到了同一路径。可以手动调整SM的路由表，或者升级到支持自适应路由的交换机。

最后，我们做一个完整的端到端带宽验证。用`ib_write_bw`工具测试两台机器间的点对点RDMA带宽：

```bash
# 在服务端（node1）
ib_write_bw -d mlx5_0 -p 18515

# 在客户端（node2）
ib_write_bw -d mlx5_0 -p 18515 node1

# 输出示例
#       bytes     #iterations    BW peak[MB/sec]    BW average[MB/sec]
#       65536     5000           23750.12            23680.45
```

单流200Gbps的理论极限是25000 MB/s，实测23680 MB/s，效率约94.7%，符合预期。如果低于20000 MB/s，检查光模块、线缆、PCIe速率（`lspci -vvv`看LinkSta）。

#### 章节小结

多节点RDMA网络配置是分布式训练的基础设施，直接影响NCCL的跨节点通信带宽。本章从硬件选型出发，详细讲解了InfiniBand和RoCE的驱动安装、子网管理器配置、NCCL参数调优、多网卡聚合以及常见问题排查。核心要点是：确保网卡固件一致、SM稳定运行、GPUDirect RDMA生效，并通过`ib_write_bw`和NCCL基准测试验证实际带宽。只有网络配置到位，后续的大模型分布式推理部署才能发挥出硬件性能。

#### 本章重点回顾

- InfiniBand vs RoCE：IB配置简单性能稳定，RoCE需PFC/ECN保证无损，运维成本更高。
- GPUDirect RDMA是NCCL高效通信的前提，必须通过PCIe拓扑验证GPU与网卡在同一NUMA节点。
- NCCL关键环境变量：`NCCL_IB_HCA`指定网卡，`NCCL_NET_GDR_LEVEL=5`启用GPUDirect，`NCCL_IB_TIMEOUT`防止大规模集群超时。
- 多网卡聚合通过`NCCL_IB_HCA`指定多设备实现，无需系统级bonding，但网络拓扑需Fat-Tree避免单点故障。
- 性能验证两步走：先用`ib_write_bw`测点对点带宽，再用NCCL `all_reduce_perf`测实际通信效率。
