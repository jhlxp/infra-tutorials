# 第20章：故障诊断与快速恢复

RoCEv2网络跑在数据中心里，最怕的就是出问题。链路闪断、交换机缓存溢出、PFC死锁、网卡丢包，任何一个故障都可能让分布式训练任务中断，或者存储集群的IO性能掉到地板。这一章我们不讲虚的，直接聚焦在实际运维中怎么快速定位RoCEv2网络的故障根因，以及如何通过配置和机制实现秒级的快速恢复。我们会从最基础的链路层排查讲起，深入到PFC死锁的检测、ECN水线误配置导致的问题，最后给出一个完整的故障恢复方案和参数配置模板。

### 20.1 链路层故障：物理层与CRC错误排查

我们先从最底层开始。RoCEv2的报文本质上是跑在以太网物理链路上的，如果光纤、光模块或者网口本身有问题，上层协议再优化也没用。常见的链路层故障包括：光模块光功率过低、光纤弯曲半径过小导致信号衰减、网口协商速率不一致、以及最让人头疼的CRC错误。CRC错误意味着报文在传输过程中发生了比特翻转，这种错误一旦出现，交换机或者网卡会直接丢弃报文，导致TCP/IP协议栈重传，但RoCEv2的RDMA传输层（RC/UC）也会触发重传，严重时会导致QoS队列阻塞。

排查链路层故障，第一步是看端口计数器。在交换机和网卡上，都有专门的统计项记录CRC错误、对齐错误、FCS错误等。我们以Mellanox ConnectX-6网卡为例，用`mlxreg_ext`命令或者`ethtool -S`可以查看到详细的错误计数。正常运行的链路，CRC错误计数应该始终为0。如果发现持续增长，说明这条链路的光模块或者线缆需要更换。

```bash
# 查看网卡端口统计，重点关注CRC错误
ethtool -S ens1f0 | grep -E "crc|fcs|align_error"

# 输出示例
#     crc_errors: 0
#     fcs_errors: 0
#     align_errors: 0
#     symbol_errors: 0

# 如果发现CRC错误非零，进一步查看光模块光功率
ethtool -m ens1f0 | grep -E "Laser|Power|Temperature"

# 输出示例
#     Laser bias current: 8.50 mA
#     Laser output power: 0.52 mW / -2.84 dBm
#     Receiver signal average optical power: 0.31 mW / -5.09 dBm
#     Module temperature: 42.5 Celsius
```

光模块的接收光功率如果低于-10dBm，误码率会急剧上升。我们建议接收光功率保持在-3dBm到-7dBm之间。如果发现光功率异常，优先检查光纤连接器的清洁度，用光纤清洁笔擦拭端面。另外，交换机的端口错误计数同样重要，登录到交换机（比如NVIDIA Spectrum交换机）使用`show interface counters errors`查看。

#### 严重警告

不要忽略单个端口的CRC错误。在RoCEv2网络中，哪怕只有0.01%的丢包率，也会导致RDMA传输性能下降50%以上。因为RoCEv2的Go-Back-N重传机制会重传丢失报文之后的所有报文，浪费大量带宽。所以，链路层必须做到零CRC错误。

排查完物理层，接下来我们看PFC流控层面的问题。CRC错误会导致报文被丢弃，而PFC流控本身如果配置不当，也会引发连锁反应。

### 20.2 PFC死锁检测与自动恢复

PFC（Priority Flow Control）是RoCEv2网络实现无损传输的关键。它基于每个优先级独立做XON/XOFF流控。但PFC有一个著名的副作用：死锁。死锁的发生场景通常是这样的：交换机A的端口P1向交换机B的端口P2发送数据，同时交换机B的端口P3向交换机A的端口P4发送数据，两条流互相等待对方释放缓冲区，结果谁都不释放，形成环路。更常见的场景是多个ToR交换机之间的多跳拓扑中，某个队列的PFC反压帧沿着上行链路逐级传递，最终导致整个网络的某个优先级队列全部停摆。

检测PFC死锁，最直接的方法是监控交换机端口的PFC暂停帧计数。在正常情况下，PFC暂停帧是动态发送和停止的，计数会增长但不会持续停留。如果发现某个端口的`tx_pause`计数在几秒内停止增长，而`rx_pause`计数持续增长，并且该端口的队列深度一直处于满状态，那么基本可以判定发生了PFC死锁。我们来看一个实际的检测脚本思路：

```python
# 简化版PFC死锁检测逻辑（基于交换机SNMP或CLI输出）
# 假设我们每隔1秒采集一次端口统计

import time
import subprocess

def get_pause_counters(port):
    # 实际中通过SSH或SNMP获取
    # 返回 (tx_pause, rx_pause)
    result = subprocess.run(
        ["show", "interface", port, "priority-flow-control"],
        capture_output=True
    )
    # 解析输出...
    return (tx_pause, rx_pause)

def detect_deadlock(port, threshold=5):
    prev_tx, prev_rx = get_pause_counters(port)
    time.sleep(1)
    curr_tx, curr_rx = get_pause_counters(port)

    tx_delta = curr_tx - prev_tx
    rx_delta = curr_rx - prev_rx

    # 死锁特征：tx暂停帧停止发送，rx暂停帧持续收到
    if tx_delta < 2 and rx_delta > threshold:
        print(f"端口{port}疑似PFC死锁，rx暂停帧持续增加")
        return True
    return False
```

一旦检测到PFC死锁，恢复手段有两种：软件恢复和硬件恢复。软件恢复就是手动或者自动重置端口，这会中断业务几十毫秒。硬件恢复依赖芯片自带的死锁恢复机制，比如NVIDIA Spectrum芯片的**PFC Deadlock Recovery**功能。它会在检测到某个队列长时间处于暂停状态（比如超过100ms）时，自动丢弃该队列的部分报文，打破死锁环路。开启这个功能需要在交换机上配置：

```bash
# 开启PFC死锁恢复
configure terminal
priority-flow-control deadlock-recovery enable
priority-flow-control deadlock-recovery threshold 100
priority-flow-control deadlock-recovery action drop
exit

# 验证配置
show priority-flow-control deadlock-recovery
```

这个功能非常实用，但要注意，丢包动作会触发RoCEv2的重传，虽然会损失一点性能，但总比整个网络卡死要好。死锁恢复之后，我们需要进一步分析死锁的根因，通常和ECN水线配置不当有关，这把我们引向下一节内容。

### 20.3 ECN水线误配置与信号风暴

ECN（Explicit Congestion Notification）是RoCEv2网络实现拥塞控制的关键信号。交换机在检测到队列深度超过ECN水线时，会在IP头中打上CE标记，接收端收到后生成CNP报文通知发送端降速。如果ECN水线配置得太低，交换机就会频繁打CE标记，导致发送端频繁降速，带宽利用率下降；如果配置得太高，拥塞信号来得太晚，队列已经溢出丢包。更严重的问题是，如果整个网络所有交换机都配置了过低的水线，可能引发**ECN信号风暴**——大量CNP报文充斥网络，占用带宽并导致CPU中断过载。

我们来看一个实际案例。某数据中心部署了200台GPU服务器，使用RoCEv2做AllReduce通信。运维人员为了追求低延迟，把ECN水线设置得很低：`min_threshold=8KB, max_threshold=24KB`。结果运行分布式训练时，网络吞吐量只有理论值的60%。通过抓包分析发现，CNP报文占比高达15%。这就是典型的ECN信号过多导致发送端过度降速。正确的做法是参考网卡厂商的推荐值，对于25Gbps端口，一般建议`min_threshold=64KB, max_threshold=128KB`。对于100Gbps端口，建议`min_threshold=256KB, max_threshold=512KB`。

```bash
# 配置ECN水线（以端口为单位）
configure terminal
interface ethernet 1/1
priority-flow-control ecn min-threshold 65536 bytes
priority-flow-control ecn max-threshold 131072 bytes
exit

# 查看当前ECN标记计数
show interface ethernet 1/1 priority-flow-control ecn statistics
```

ECN水线的配置需要结合网络拓扑和流量模型来调优。如果是存储集群的读密集型流量，ECN水线可以适当调高，因为读流量通常是大象流，对延迟不敏感；如果是HPC集群的同步通信，需要调低水线来保证低延迟。调优的方法是通过A/B测试：先设置一组参数，运行基准测试，观察吞吐量和CNP比例，然后微调。我们一般把CNP比例控制在1%~3%之间，超过5%就说明水线太低了。

#### 设计要点

ECN水线不是越小越好，也不是越大越好。核心原则是：让ECN信号在拥塞发生前发出，但不要频繁发出。建议从厂商推荐值开始，每次调整步长不超过20%，并配合CNP监控来验证效果。

解决了ECN信号风暴问题，我们还需要关注另一个常见的故障：网卡缓存溢出导致的丢包。

### 20.4 网卡接收缓存溢出与丢包定位

RoCEv2的网卡（比如ConnectX-6/7）内部有接收缓存（Rx Buffer），用于暂存从网络接收到的数据，等待PCIe DMA传输到主机内存。如果主机侧的处理速度跟不上网络接收速度，接收缓存就会溢出，导致网卡丢弃报文。这种丢包在网卡统计中表现为`rx_discards`或者`out_of_buffer`。与链路层的CRC丢包不同，这种丢包是软性的，通常发生在突发流量场景下。

定位网卡缓存溢出，第一步是查看网卡内部的硬件计数器：

```bash
# 使用mlxreg_ext读取网卡内部计数器
mlxreg_ext -d /dev/mst/mt4123_pciconf0 --reg_name PCNT --get

# 重点关注以下字段
#     rx_no_buf: 0          # 接收时无缓冲区可用
#     rx_discard: 0         # 接收丢弃
#     rx_prio0_buf_discard: 0  # 优先级0的丢弃

# 如果rx_no_buf非零，说明接收缓存不足
# 解决方案：增加网卡接收缓冲区大小
ethtool -G ens1f0 rx 8192
# 将RX ring buffer从默认的4096增加到8192
```

如果调整缓冲区大小后仍然有丢包，说明主机侧的处理能力存在瓶颈。常见原因包括：PCIe带宽不足（比如插在x8槽上）、CPU中断负载过高、或者内存带宽不够。这时候需要检查PCIe链路状态：

| 检查项 | 命令 | 正常值 | 异常值 |
| --- | --- | --- | --- |
| PCIe链路宽度 | lspci -vvv -s 03:00.0 | Width x16 | Width x8或x4 |
| PCIe链路速率 | lspci -vvv -s 03:00.0 | 16 GT/s (Gen4) | 8 GT/s (Gen3)或更低 |
| 网卡温度 | ethtool -m ens1f0 | < 70°C | > 85°C |
| CPU中断负载 | cat /proc/interrupts | 中断均匀分布 | 单个CPU核中断占比>80% |

在工程实践中，我们遇到过因为BIOS设置导致PCIe降速到Gen3 x8的情况，网卡理论带宽从200Gbps降到100Gbps，但上层应用不知道，仍然以200Gbps的速率发送，导致接收缓存频繁溢出。解决方法是进入BIOS强制设置PCIe为Gen4 x16。网卡缓存溢出问题排查清楚后，我们还需要关注网络层面的路由黑洞和ECMP哈希不均问题。

### 20.5 ECMP哈希不均与流冲突检测

RoCEv2网络通常使用ECMP（Equal Cost Multi Path）实现多路径负载均衡。ECMP基于五元组（源IP、目的IP、源端口、目的端口、协议）计算哈希值，选择一条路径。问题在于，如果哈希算法不够随机，或者流量中存在大量大象流（比如分布式训练中的AllReduce流），多条大象流可能被哈希到同一条链路上，造成拥塞，而其他链路空闲。这就是ECMP哈希不均。

检测ECMP哈希不均，需要在所有链路上采集端口利用率。如果某条链路的利用率超过80%，而其他链路利用率低于30%，就说明哈希不均。更精确的方法是使用交换机的**Flowlet**功能或者**Dynamic Load Balancing**。NVIDIA Spectrum交换机支持基于流的实时重哈希（Flowlet Switching），它会根据流之间的空闲间隔重新分配路径，避免哈希冲突。配置方法如下：

```bash
# 开启Flowlet负载均衡
configure terminal
load-balance flowlet enable
load-balance flowlet inactivity-interval 100
load-balance flowlet max-flows 65536
exit

# 查看当前ECMP组的流量分布
show ip route ecmp-group 1
```

除了ECMP哈希不均，还有一种情况是**流冲突**：两个不同的流（比如存储写流和训练同步流）共享同一个优先级队列，互相抢占资源。解决方案是使用PFC优先级隔离：将存储流量映射到优先级3，训练流量映射到优先级4，分别配置不同的ECN水线和带宽分配。这样即使某个优先级出现拥塞，也不会影响另一个优先级。

#### 信息

ECMP哈希不均的排查思路：先看链路利用率分布，再看每条链路上的流数量。如果链路利用率差异大但流数量差异不大，说明是大象流冲突；如果流数量差异也大，说明哈希种子需要调整。可以尝试修改哈希因子（比如加入源端口或目的端口）来改善。

解决了ECMP问题，接下来我们看一个更隐蔽的故障：链路闪断导致的ARP/ND缓存失效。

### 20.6 链路闪断与ARP/ND缓存快速恢复

RoCEv2网络依赖IP层路由，所以ARP（IPv4）或ND（IPv6）缓存的正确性至关重要。当一条物理链路发生闪断（比如光模块松动后又恢复），交换机的路由表会更新，但网卡上的ARP缓存可能还指向旧的下一跳MAC地址。这时候RoCEv2报文会被发送到错误的端口，导致丢包。更糟糕的是，ARP缓存超时时间通常设置为300秒，在这5分钟内，网络可能持续丢包。

快速恢复的方法是启用**Gratuitous ARP**或**快速ND**。当交换机的端口状态从Down变回Up时，立即发送Gratuitous ARP报文，通知所有邻居更新ARP缓存。在NVIDIA交换机上，可以配置**arp-gratuitous-update**：

```bash
# 配置端口状态变化时发送Gratuitous ARP
configure terminal
interface ethernet 1/1
ip arp gratuitous update enable
exit

# 同时缩短ARP缓存超时时间（从300秒改为30秒）
configure terminal
ip arp timeout 30
exit
```

但缩短ARP超时也有代价：会增加ARP请求广播流量。对于大型数据中心（超过1000台服务器），每秒大量的ARP请求可能占用控制面CPU。所以更推荐使用**BGP-EVPN**的**ARP抑制**功能，由交换机代理ARP响应，减少广播。在RoCEv2网络中，我们建议将ARP超时设置为60秒，同时开启Gratuitous ARP更新，这样既能快速收敛，又不会产生太多广播。

链路闪断的另一个影响是导致RoCEv2的QP（Queue Pair）连接断开。对于RC（Reliable Connection）模式的QP，一旦链路中断超过一定时间，QP状态会变成Error，需要应用层重新建立连接。这引出了我们下一个话题：QP连接断开后的快速重建。

### 20.7 QP连接断开与自动重建机制

在RoCEv2的RC传输模式下，每个QP对应一条可靠的连接。如果链路闪断或者网卡复位，QP的状态会从RTS（Ready To Send）变为ERR（Error）。此时应用层需要销毁这个QP并重新创建，这个过程通常需要几百毫秒到几秒。对于分布式训练框架（比如NVIDIA NCCL），它会自动检测QP错误并重建连接。但如果是自定义的RDMA应用，就需要自己实现QP重建逻辑。

检测QP状态的方法是通过`ibv_query_qp`接口。我们来看一个简单的QP健康检查代码片段：

```c
#include <infiniband/verbs.h>

int check_qp_state(struct ibv_qp *qp) {
    struct ibv_qp_attr attr;
    struct ibv_qp_init_attr init_attr;

    // 查询QP属性
    ibv_query_qp(qp, &attr, IBV_QP_STATE, &init_attr);

    // 检查QP状态
    if (attr.qp_state == IBV_QPS_ERR) {
        printf("QP 0x%x 处于ERROR状态，需要重建\n", qp->qp_num);
        return -1;
    }
    return 0;
}

// 重建QP的简化流程
int reconnect_qp(struct ibv_pd *pd, struct ibv_cq *cq,
                 uint32_t remote_qpn, uint16_t dlid) {
    struct ibv_qp *new_qp;
    struct ibv_qp_attr attr;

    // 创建新的QP
    new_qp = ibv_create_qp(pd, &qp_init_attr);
    // 初始化QP到RTS状态
    // ... (设置QP属性、连接参数)
    return 0;
}
```

在实际工程中，我们建议在应用层实现一个**QP监控线程**，每隔100ms轮询一次所有QP的状态。一旦发现QP进入ERR状态，立即触发重建流程。同时，配合交换机的**Bidirectional Forwarding Detection (BFD)**功能，可以更快地检测到链路故障。BFD的检测间隔可以配置到3.3ms，远快于ARP超时。配置BFD后，交换机可以在几毫秒内通知服务器链路故障，服务器立即触发QP重建。

#### 设计要点

QP重建的耗时主要花在建立连接时的握手过程。对于RC QP，重建需要经历Reset→Init→RTR→RTS四个状态转换，每个状态转换都需要发送一次RMDA CM消息。如果使用SRQ（Shared Receive Queue），可以减少内存注册的开销，加速重建。

QP重建是应用层的最后一道防线。但更理想的做法是网络层本身就具备快速恢复能力，比如通过冗余路径实现无缝切换。这就是我们下一节要讲的：Graceful Link Failover。

### 20.8 Graceful Link Failover与无缝切换

Graceful Link Failover（优雅链路切换）是指当一条链路故障时，网络能够在用户无感知的情况下将流量切换到备用链路上。在RoCEv2网络中，实现这一目标主要依赖两种技术：**链路聚合（LAG）**和**主备转发**。LAG将多条物理链路捆绑成一条逻辑链路，其中一条链路故障时，流量自动分布到其他链路上。但LAG要求所有链路连接到同一台交换机，对于跨交换机的场景，需要用到**MLAG**或**EVPN多归属**。

对于RoCEv2的RDMA流量，LAG有一个关键限制：哈希颗粒度是流级别的。如果故障发生时，某条大象流正好哈希在故障链路上，这个流的所有报文都会丢失，直到流被重新哈希。为了减少影响，我们建议使用**Per-Packet Load Balancing**（逐包负载均衡），但逐包均衡会导致报文乱序，RoCEv2的接收端需要支持重排序。好在最新的ConnectX-7网卡支持**Multi-Path**功能，可以在网卡层面实现逐流或逐包的负载均衡，并且支持故障切换。

配置MLAG实现跨交换机链路冗余的示例如下：

```bash
# 在两台交换机上配置MLAG
# 交换机A
configure terminal
mlag-vip 10.10.10.100
interface mlag 1
mlag-peer-ip 10.10.10.2
exit
interface ethernet 1/1
mlag-group 1
exit

# 交换机B
configure terminal
mlag-vip 10.10.10.100
interface mlag 1
mlag-peer-ip 10.10.10.1
exit
interface ethernet 1/1
mlag-group 1
exit
```

配置完成后，服务器端只需要将网卡绑定到MLAG VIP上，交换机之间通过peer link同步状态。当一条上行链路故障时，流量在50ms内切换到另一条链路。对于RoCEv2的PFC和ECN状态，MLAG也会同步，确保切换后拥塞控制机制正常工作。Graceful Failover是故障恢复的终极方案，但需要硬件和软件的双重支持。在预算有限的情况下，至少要做到快速检测和快速重建。

### 20.9 故障恢复的自动化脚本与监控体系

前面讲了各种故障的定位和恢复手段，但靠人工一个一个去排查肯定不行。实际运维中，我们需要一个自动化的监控和恢复体系。这个体系应该包括三个层次：数据采集层、故障检测层、自动恢复层。数据采集层通过SNMP、gRPC、或者CLI抓取交换机和网卡的计数器；故障检测层运行我们前面提到的PFC死锁检测、ECN风暴检测、CRC错误检测等算法；自动恢复层根据故障类型触发相应的恢复动作，比如重置端口、调整ECN水线、重建QP等。

下面是一个简化的自动化恢复脚本框架（使用Python）：

```python
import time
import subprocess
import json

class RoCEv2FaultRecovery:
    def __init__(self, switch_ip, nic_port):
        self.switch_ip = switch_ip
        self.nic_port = nic_port

    def collect_stats(self):
        # 采集网卡和交换机统计
        stats = {}
        # 网卡CRC错误
        result = subprocess.run(
            ["ethtool", "-S", self.nic_port],
            capture_output=True, text=True
        )
        stats['crc_errors'] = self._parse_ethtool(result.stdout, 'crc_errors')
        # 交换机PFC暂停帧
        # ... 通过SSH获取
        return stats

    def detect_and_recover(self):
        while True:
            stats = self.collect_stats()
            # 1. 检测CRC错误
            if stats.get('crc_errors', 0) > 0:
                print("检测到CRC错误，尝试重置端口")
                self._reset_switch_port()
            # 2. 检测PFC死锁
            if self._detect_pfc_deadlock():
                print("检测到PFC死锁，触发死锁恢复")
                self._trigger_deadlock_recovery()
            # 3. 检测ECN风暴
            if stats.get('cnp_ratio', 0) > 0.05:
                print("CNP比例过高，调整ECN水线")
                self._adjust_ecn_threshold()
            time.sleep(5)

    def _reset_switch_port(self):
        # 通过SSH执行端口重置
        cmd = f"ssh admin@{self.switch_ip} 'interface ethernet 1/1 shutdown; no shutdown'"
        subprocess.run(cmd, shell=True)

    def _trigger_deadlock_recovery(self):
        cmd = f"ssh admin@{self.switch_ip} 'priority-flow-control deadlock-recovery trigger'"
        subprocess.run(cmd, shell=True)

    def _adjust_ecn_threshold(self):
        # 将ECN水线提升20%
        cmd = f"ssh admin@{self.switch_ip} 'priority-flow-control ecn min-threshold 78643 bytes'"
        subprocess.run(cmd, shell=True)
```

这个脚本只是一个演示框架，实际生产环境中需要结合告警系统（如Prometheus + Alertmanager）来实现分级告警。对于CRC错误这类硬件故障，自动恢复后如果仍然出现，需要自动创建工单通知硬件更换。对于PFC死锁这类软故障，自动恢复后要记录日志，并分析根因。监控体系的核心指标包括：每端口的CRC错误率、PFC暂停帧频率、CNP报文占比、QP错误率、链路利用率标准差。这些指标可以汇总到一个Grafana仪表盘上，方便快速定位问题。

#### 信息

建议将监控间隔设置为5秒，故障检测窗口设置为30秒（连续6次采样均异常才触发告警），避免误报。自动恢复动作执行后，需要等待至少10秒再重新检测，给网络稳定时间。

自动化脚本跑起来之后，日常运维的工作量会大大降低。但有一个问题不能忽略：恢复动作本身会不会引入新的问题？比如重置端口会导致该端口上的所有QP断开，影响正在运行的业务。所以，自动恢复需要和业务方协调，最好在业务低峰期执行。或者采用更温和的方式：先尝试调整参数（如ECN水线），如果无效再重置端口。

#### 章节小结

这一章我们完整地走了一遍RoCEv2网络的故障诊断与快速恢复流程。从最底层的链路CRC错误排查开始，到PFC死锁的检测与自动恢复，再到ECN水线误配置导致的信号风暴，以及网卡缓存溢出、ECMP哈希不均、ARP缓存失效、QP连接断开等常见故障。针对每个故障，都给出了具体的检测方法和恢复配置。最后，我们搭建了一个自动化监控恢复脚本的框架，把前面所有的检测逻辑和恢复动作整合到一起。这套方法论在实际的数据中心运维中已经验证过，能够将故障恢复时间从分钟级降低到秒级，甚至毫秒级。

#### 本章重点回顾

- 链路层CRC错误必须为零，光模块接收光功率保持在-3dBm到-7dBm之间。
- PFC死锁的特征是tx暂停帧停止发送而rx暂停帧持续增长，通过硬件死锁恢复功能自动打破死锁。
- ECN水线配置应使CNP比例控制在1%~3%之间，过高或过低都会影响性能。
- 网卡接收缓存溢出通过增加RX ring buffer或检查PCIe链路状态解决。
- ECMP哈希不均通过Flowlet负载均衡或调整哈希因子改善。
- 链路闪断后通过Gratuitous ARP和缩短ARP超时（60秒）快速恢复。
- QP连接断开后应用层需在100ms内检测并重建，配合BFD加速故障感知。
- Graceful Link Failover通过MLAG或网卡Multi-Path实现50ms内无缝切换。
- 自动化恢复脚本需集成CRC检测、PFC死锁检测、ECN调整等逻辑，并设置合理的检测窗口避免误报。
