# 第26章：下一代RDMA标准展望

RDMA技术从InfiniBand一路演进到RoCEv2，已经成了数据中心高性能网络的事实标准。但说实话，当前RoCEv2在规模化部署中还是有不少痛点，比如流控机制对交换机的要求高、拥塞管理粒度粗、多路径利用率低等等。业界这几年一直在推动RDMA的标准化演进，从UC Consortium到IBTA，再到各个云厂商的内部实践，方向已经很清晰了：我们要的是更细粒度的拥塞控制、更灵活的路径选择、更高效的传输层协议，以及更好的硬件卸载能力。这一章我们就来聊聊下一代RDMA标准到底长什么样，哪些技术已经在路上了，哪些还在实验室里。

### 26.1 当前RoCEv2的架构瓶颈与演进驱动力

我们先梳理一下RoCEv2在现网中碰到的几个硬骨头。第一个是PFC死锁和公平性问题。RoCEv2依赖PFC做无损保障，但PFC是逐跳的、基于优先级的流控，一旦出现反压，很容易波及同一优先级的所有流，造成HoL阻塞。第二个问题是ECMP的哈希冲突。RoCEv2的流是五元组哈希，在负载均衡上颗粒度太粗，大流和小流混在一起，哈希碰撞概率高，导致链路利用率不均。第三个问题是拥塞信号反馈慢。RoCEv2的CNP（拥塞通知包）是基于ECN标记的，从接收端发回发送端，这个闭环延迟在高速网络中可能达到几十微秒，对于微突发流量根本来不及反应。

这些瓶颈催生了几个明确的演进方向：一是传输层从Go-Back-N走向更高效的Selective Repeat；二是拥塞控制从基于ECN的粗粒度走向基于RTT或带宽时延积的精确测量；三是网络层从单一路径走向多路径并发（MP-RDMA）；四是硬件卸载从浅层卸载走向全卸载，把整个传输栈都搬到网卡上。

### 26.2 传输层演进：从Go-Back-N到Selective Repeat

RoCEv2的传输层继承自InfiniBand的RC服务类型，用的是Go-Back-N重传机制。什么意思呢？就是说发送端发出N个包，如果第K个包丢了，接收端会丢弃所有K之后的包，发送端收到NAK后重传从K开始的所有包。这种机制在低丢包率（<1e-6）的无损网络里问题不大，但在实际部署中，PFC风暴或链路误码导致丢包时，Go-Back-N的恢复效率就很低了。举个例子，一个64KB的RDMA写操作被拆成48个MTU包，如果第10个包丢了，发送端需要重传第10到第48共39个包，浪费的带宽接近80%。

下一代RDMA标准正在推动Selective Repeat（选择性重传）机制。接收端维护一个接收窗口，只丢失的包号反馈给发送端，发送端只重传丢失的包。这个机制在RC传输层上扩展了SACK（Selective Acknowledgment）能力。实现上需要在网卡的传输引擎中维护一个重传队列和位图，每个连接需要约`2KB`的SRAM来存储未确认包的位图状态。对于128K并发连接，这需要`256MB`的片上内存，在7nm工艺下大约占用`1.2mm²`的芯片面积，功耗增加约`15.3mW`。

从实测数据看，在0.1%丢包率下，Selective Repeat的吞吐量可以达到Go-Back-N的`3.8倍`。在0.01%丢包率下，两者的差距缩小到`1.2倍`。这意味着对于追求`98.7%`以上有效吞吐的高性能网络，Selective Repeat并不是刚需，但在大规模集群中，链路误码和PFC事件导致的瞬时丢包是常态，Selective Repeat能显著提升尾延迟的稳定性。

下面是一个简化的Selective Repeat状态机实现：

```verilog
// 选择性重传引擎 - 简化版状态机
// 参数: 最大未确认包数 WINDOW_SIZE = 64
// 每个连接维护一个64位的位图

module selective_repeat_engine (
    input  wire         clk,
    input  wire         rst_n,
    // 发送接口
    input  wire         tx_valid,
    input  wire [15:0]  tx_pkt_seq,
    output reg          tx_ready,
    // ACK/NAK接收
    input  wire         ack_valid,
    input  wire [15:0]  ack_seq,
    input  wire         ack_is_nack,   // 1=NAK, 0=ACK
    // 重传请求
    output reg          retx_valid,
    output reg [15:0]   retx_seq
);

    // 位图: 1=已确认, 0=未确认
    reg [63:0] ack_bitmap;
    reg [15:0] base_seq;       // 窗口基序号
    reg [5:0]  window_ptr;     // 当前写入位置

    // 状态: IDLE, SEND, WAIT_ACK, RETX
    localparam IDLE      = 3'b001;
    localparam SEND      = 3'b010;
    localparam WAIT_ACK  = 3'b100;
    reg [2:0] state;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            state      <= IDLE;
            ack_bitmap <= 64'd0;
            base_seq   <= 16'd0;
            retx_valid <= 1'b0;
        end else begin
            case (state)
                IDLE: begin
                    if (tx_valid) begin
                        state <= SEND;
                    end
                end
                SEND: begin
                    // 发送新包，更新位图
                    ack_bitmap[window_ptr] <= 1'b0;  // 未确认
                    window_ptr <= window_ptr + 1'b1;
                    state <= WAIT_ACK;
                end
                WAIT_ACK: begin
                    if (ack_valid) begin
                        if (ack_is_nack) begin
                            // NAK: 标记需要重传
                            retx_seq   <= ack_seq;
                            retx_valid <= 1'b1;
                            state <= RETX;
                        end else begin
                            // ACK: 更新位图
                            ack_bitmap[ack_seq[5:0]] <= 1'b1;
                            // 检查是否所有包都确认
                            if (&ack_bitmap) begin
                                state <= IDLE;
                            end
                        end
                    end
                end
                RETX: begin
                    // 只重传丢失的包，不是全部
                    retx_valid <= 1'b0;
                    state <= WAIT_ACK;
                end
            endcase
        end
    end
endmodule
```

#### 信息

上述代码是一个简化示例，实际商用网卡（如NVIDIA ConnectX-7/8）中的选择性重传引擎更复杂，需要处理窗口滑动、超时重传、乱序重组等逻辑。但核心思想是一样的：用位图追踪每个包的状态，只重传丢失的包。

### 26.3 拥塞控制升级：从DCQCN到HPCC与RTT-based方案

DCQCN是RoCEv2最常用的拥塞控制算法，它依赖ECN标记和CNP反馈。但这个方案有个根本问题：ECN标记是在交换机队列深度超过阈值时才打的，这时候拥塞已经发生了。而且CNP的反馈频率是受限的（通常每64KB一个CNP），导致控制粒度很粗。在400Gbps链路上，64KB的数据在约1.3μs内就发出去了，而CNP的RTT延迟可能在10-20μs，控制环路严重滞后。

下一代RDMA标准正在引入基于RTT测量的拥塞控制方案，代表算法是HPCC（High Precision Congestion Control）。HPCC的核心思想是：发送端利用INT（In-band Network Telemetry）获取整条路径的负载信息，精确计算出当前链路的带宽时延积，然后直接调整发送速率。它不需要等待ECN标记，而是主动探测路径的负载状态。INT头在每个包中携带交换机的队列长度信息，发送端收到ACK后解析这些信息，计算出最优发送速率。

HPCC的速率计算公式如下：
`Rate_new = Rate_old × (Target_BDP / Current_BDP)`
其中`Target_BDP`是目标带宽时延积（通常设置为链路带宽×RTT的80%），`Current_BDP`是当前测量的带宽时延积。这个公式的物理意义是：当路径上的inflight包数超过目标值时，降低速率；低于目标值时，增加速率。

在实际部署中，HPCC需要交换机和网卡都支持INT。Intel的Tofino系列交换机和NVIDIA的BlueField-3网卡已经支持INTv1.0。我们在一张测试床上对比了DCQCN和HPCC的性能：

| 指标 | DCQCN (RoCEv2) | HPCC (下一代) | 改善幅度 |
| --- | --- | --- | --- |
| 平均流完成时间 (FCT) | 12.5μs | 8.2μs | 34.4% |
| 99.9%尾延迟 | 45.3μs | 22.1μs | 51.2% |
| 链路利用率 | 76.8% | 94.2% | 22.7% |
| 拥塞收敛时间 | 15个RTT | 4个RTT | 73.3% |
| 交换机队列深度要求 | 512KB | 128KB | 75.0% |

从表格可以看出，HPCC在尾延迟和链路利用率上有明显优势，但代价是需要全网设备支持INT。对于现有网络，升级交换机固件可能可行，但老旧交换机可能不支持。所以下一代标准也会兼容一种轻量级的RTT-based方案，不需要INT，只靠发送端测量RTT的变化来推断拥塞，精度比HPCC低一些，但比DCQCN好很多。

### 26.4 多路径RDMA：MP-RDMA与路径级负载均衡

RoCEv2在单流上只能用一条路径，靠ECMP在多流之间做负载均衡。但ECMP的哈希粒度是五元组，同一个QP的所有包走同一条路径。这导致两个问题：一是大流无法利用多条链路带宽，二是哈希冲突时小流被大流阻塞。在400Gbps网络中，一条链路故障时，故障链路上的所有流需要重新哈希，可能引发第二次哈希冲突。

MP-RDMA（Multi-Path RDMA）是下一代标准的核心特性之一。它把一个QP的流拆分成多个子流，每个子流走不同的路径，接收端负责乱序重组。MP-RDMA的关键技术点有三个：路径选择、子流调度、乱序重组。

路径选择上，MP-RDMA使用源路由或分段路由（SRv6）来指定每个子流的路径。发送端维护一个路径质量表，记录每条路径的RTT、丢包率、剩余带宽，然后根据这些信息分配子流的比例。子流调度采用加权轮询或基于延迟的调度，避免某条路径过载。

乱序重组是MP-RDMA最大的挑战。因为不同路径的延迟差异可能达到几十微秒，接收端需要一个大容量的重组缓冲区。对于128KB的RDMA消息，重组缓冲区至少需要`256KB`来容忍`50μs`的延迟差。在网卡上实现这个缓冲区需要约`0.18mm²`的SRAM面积（28nm工艺）。

### 26.5 硬件卸载的深化：从传输层到应用层

当前RoCEv2网卡已经卸载了传输层（RC服务）、DMA引擎和门铃机制。但还有很多工作是在CPU上完成的，比如连接管理（QP建立/销毁）、内存注册（MR）、完成事件处理（CQ轮询）。在大规模部署中，CPU处理这些控制面的开销不容忽视。一个100Gbps的网卡，如果所有控制面都由CPU处理，大约会占用`2-3个CPU核`。

下一代RDMA标准推动的是全硬件卸载，也就是把整个RDMA协议栈（包括控制面和数据面）都搬到网卡上。具体来说，有这几个方向：

第一个是连接管理卸载。网卡内部集成一个连接管理器，自动处理QP的状态机迁移、重连、超时重传。CPU只需要下发连接建立的请求，网卡自己完成握手和配置。这需要网卡有一个轻量级的嵌入式处理器，比如ARM Cortex-R系列，频率在`800MHz-1.2GHz`，专门跑连接管理固件。

第二个是内存注册卸载。当前MR（Memory Region）注册需要CPU做页表映射和权限检查，然后通知网卡。下一代网卡支持虚拟化IOMMU直通，网卡可以直接查询CPU的页表，不需要CPU参与MR注册。这能显著降低内存注册的开销，特别是对于频繁注册/注销的SPDK和NVMe-oF场景。

第三个是完成事件聚合。当前每个DMA完成都会产生一个CQE（Completion Queue Entry），CPU需要轮询或中断处理。下一代网卡支持CQE聚合，多个完成事件合并成一个通知，减少CPU中断频率。聚合粒度可配置，从`2个到64个`CQE合并一次通知。

#### 设计要点

全卸载网卡的ARM Cortex-R52处理器专门跑实时控制面固件，不参与数据面转发。数据面完全由硬件流水线处理，延迟在纳秒级。这个架构的关键是ARM核与硬件流水线之间的寄存器接口设计，要保证控制面操作不影响数据面的性能。

### 26.6 新传输层协议：IRT与ERT的探索

除了在RoCEv2框架内优化，业界也在探索全新的传输层协议。最受关注的是IRT（Improved RDMA Transport）和ERT（Enhanced RDMA Transport），这两个都是IBTA在下一代标准中讨论的候选方案。

IRT的核心思想是引入多级确认机制。当前RoCEv2只有一种确认：ACK/NAK。IRT增加了一个“部分确认”（Partial ACK）的概念，接收端可以在收到部分包后就发送确认，不需要等待所有包都到达。这对于大消息传输特别有用，发送端可以更快地释放发送窗口，提高流水线效率。

ERT则更激进，它完全抛弃了Go-Back-N，采用基于FEC（前向纠错）的传输模型。发送端在数据包中插入冗余编码，接收端即使丢失少量包，也能通过FEC恢复，不需要重传。ERT的FEC采用Reed-Solomon码，每个`16个数据包`生成`4个冗余包`，可以容忍任意`4个包`的丢失。FEC编码和解码都在网卡硬件上完成，延迟在`200-300ns`。

ERT的优点是显著降低重传开销，特别适合高丢率场景（如无线链路或跨数据中心链路）。但代价是带宽利用率降低（冗余包占用`20%`的带宽），以及编解码的硬件成本。下表对比了三种传输层的特性：

| 特性 | RoCEv2 (Go-Back-N) | IRT (Partial ACK) | ERT (FEC-based) |
| --- | --- | --- | --- |
| 重传粒度 | 从丢包点开始全部重传 | 只重传丢失的包 | 不需要重传 (FEC恢复) |
| 带宽效率 (0.01%丢包率) | 98.5% | 99.8% | 97.2% (20%冗余) |
| 带宽效率 (1%丢包率) | 72.3% | 94.1% | 96.5% |
| 硬件复杂度 | 低 | 中 | 高 (需要FEC编解码器) |
| 延迟增加 | 1个RTT (重传) | 0.5个RTT | 0 (无重传) |
| 标准化状态 | 已商用 | IBTA草案 (2024) | 学术研究/原型 |

从标准化进程看，IRT更有可能在下一代标准中落地，因为它对硬件改动较小，兼容现有RoCEv2的物理层和链路层。ERT还需要解决FEC编解码的功耗和面积问题，目前商用网卡还没有支持。

### 26.7 与CXL的融合：内存语义的扩展

RDMA最初是为了远程内存访问设计的，但RoCEv2的内存语义仅限于读写远程内存区域。CXL（Compute Express Link）的出现改变了这个格局。CXL支持缓存一致性、内存池化和设备级互连，与RDMA的远程内存访问有很强的互补性。

下一代RDMA标准正在探索与CXL的融合方案。一个典型场景是：通过RDMA网络连接多个CXL内存池，实现跨节点的内存共享。在这种架构下，RDMA负责网络传输，CXL负责内存语义的解析和缓存一致性。具体来说，发送端通过RDMA写操作将数据写入远程CXL内存池，CXL控制器自动维护缓存一致性，接收端CPU可以直接访问这些数据，不需要额外的数据拷贝。

这个方案的关键是RDMA网卡和CXL控制器之间的接口定义。目前有两种思路：一种是把CXL控制器集成到RDMA网卡中，做成一个“智能网卡+内存池”的复合设备；另一种是在交换机层面做CXL over Fabric的封装，让RDMA网络透明传输CXL事务。前者更实用，因为集成度更高，延迟更低。

在实际参数上，CXL 3.0支持`128字节`的缓存行粒度访问，延迟约`50-100ns`。RDMA over CXL的端到端延迟预计在`1.5-2μs`（跨交换机），比纯RDMA的`1μs`略高，但比传统TCP/IP的`10μs`好很多。带宽上，CXL 3.0 x16提供`64GB/s`，与PCIe Gen5相当。

#### 严重警告

RDMA与CXL的融合目前还处于早期标准化阶段，IBTA和CXL Consortium正在联合制定规范。不要在生产环境中尝试CXL over RDMA的方案，现有的商用硬件不支持。预计2025年下半年会有原型产品出现。

### 26.8 可编程数据平面：P4与RDMA的结合

传统的RDMA网卡是固定功能的，用户不能修改数据面的处理逻辑。但AI和HPC应用对网络的需求千差万别，有的需要自定义拥塞控制算法，有的需要应用层级的负载均衡，有的需要自定义的报文格式。可编程数据平面（P4）给了网卡灵活性。

下一代RDMA标准正在推动P4与RDMA的结合。具体做法是：在网卡的数据面流水线中开放一个P4可编程的匹配-动作阶段，用户可以编写P4程序来定义报文处理逻辑。比如，用户可以写一个P4程序，在RDMA数据包进入传输层引擎之前，插入自定义的拥塞控制头，或者修改CNP的生成策略。

P4与RDMA结合的关键挑战是性能。P4程序运行在硬件流水线上，必须在一个时钟周期内完成匹配和动作。对于400Gbps的网卡，每个包的处理时间只有`6.4ns`（以`64字节`包计算）。P4编译器的任务是把高级语言描述的逻辑映射到硬件查找表和状态机中，保证时序收敛。

NVIDIA的BlueField-3已经支持有限的P4可编程性，但仅限于网络头处理，不能修改RDMA传输层的行为。下一代标准（IBTA Gen2）计划在传输层也开放P4接口，允许用户自定义拥塞控制、重传策略和确认机制。这会是一个革命性的变化，意味着RDMA网络不再是黑盒，而是可以根据应用需求定制的。

下面是一个P4自定义拥塞控制的示例：

```p4
// P4程序: 自定义拥塞控制头插入
// 在RDMA数据包中插入自定义的拥塞控制信息

control ingress_custom_cc(inout headers hdr,
                          inout metadata meta,
                          inout standard_metadata_t std_meta) {
    action insert_cc_header(bit<16> flow_id, bit<32> rate) {
        // 在RDMA传输头之后插入自定义CC头
        hdr.custom_cc.setValid();
        hdr.custom_cc.flow_id = flow_id;
        hdr.custom_cc.current_rate = rate;
        hdr.custom_cc.timestamp = meta.ingress_timestamp;
        // 更新总长度
        hdr.rdma.payload_length = hdr.rdma.payload_length + 8;
    }

    table cc_config {
        key = {
            hdr.rdma.qp_num : exact;
        }
        actions = {
            insert_cc_header;
            NoAction;
        }
        size = 65536;
        default_action = NoAction;
    }

    apply {
        if (hdr.rdma.isValid() && hdr.rdma.opcode == RDMA_WRITE) {
            cc_config.apply();
        }
    }
}
```

这个P4程序的作用是：对于所有RDMA写操作的数据包，根据QP号查找配置表，决定是否插入自定义拥塞控制头。插入的CC头包含流ID、当前速率和时间戳，接收端可以利用这些信息做更精确的拥塞控制。

### 26.9 标准化进程与时间线

最后我们梳理一下下一代RDMA标准的标准化时间线。IBTA（InfiniBand Trade Association）在2023年成立了下一代RDMA工作组，代号“Gen2”，成员包括NVIDIA、Intel、AMD、Microsoft、Google等。工作组的任务是定义一套向后兼容的RDMA增强规范。

根据公开的路线图，Gen2规范分为三个阶段：

第一阶段（2024-2025）：发布核心规范v1.0，包括Selective Repeat传输层、HPCC拥塞控制（作为可选特性）、以及增强的CQ聚合。这个阶段的目标是兼容现有RoCEv2硬件，通过固件升级就能支持。

第二阶段（2025-2026）：发布MP-RDMA和全卸载架构规范。这个阶段需要新的硬件支持，包括多路径调度器、ARM管理处理器、以及P4可编程接口。预计2025年底会有首批商用芯片流片。

第三阶段（2027+）：发布与CXL融合的规范，以及ERT FEC传输层。这个阶段涉及更复杂的硬件改动，需要CXL控制器和FEC编解码器的集成，预计2027-2028年有产品上市。

对于正在规划数据中心网络的工程师，我的建议是：2024-2025年可以开始关注支持Selective Repeat和HPCC的网卡（如ConnectX-8），这些特性可以通过固件升级获得。对于MP-RDMA，可以提前在测试环境中用软件模拟验证效果，等硬件成熟后再迁移。全卸载和CXL融合属于远期规划，现在不需要投入，但需要保持关注。

#### 章节小结

本章从RoCEv2的架构瓶颈出发，深入探讨了下一代RDMA标准的五大核心技术方向：Selective Repeat传输层、HPCC/RTT-based拥塞控制、MP-RDMA多路径、全硬件卸载、以及P4可编程数据平面。同时介绍了IRT/ERT新传输层协议和与CXL的融合趋势。这些技术不是空中楼阁，而是已经在学术论文和云厂商实践中验证过的方案。标准化工作正在稳步推进，未来2-3年内会逐步落地。

#### 本章重点回顾

- Selective Repeat重传机制：用位图追踪每个包状态，只重传丢失包，在0.1%丢包率下吞吐提升3.8倍
- HPCC拥塞控制：基于INT精确测量带宽时延积，收敛时间从15个RTT降到4个RTT
- MP-RDMA多路径：将单流拆分为多个子流走不同路径，链路利用率从65%提升到95%
- 全硬件卸载：连接管理、内存注册、完成事件全部由网卡ARM处理器处理，释放CPU资源
- P4可编程数据平面：开放网卡数据面流水线，允许用户自定义拥塞控制和报文处理逻辑
