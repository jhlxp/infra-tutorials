# 第1章：RDMA技术演进与协议栈

本章我们聚焦RDMA技术的核心演进路径，从最基础的DMA（直接内存访问）开始，逐步深入到ARM Cortex处理器在智能网卡中的角色，最后落脚到寄存器级的编程接口。目标是帮大家理清RDMA协议栈的硬件基础：为什么需要RDMA？传统DMA的瓶颈在哪里？ARM核在网卡里到底干了什么？以及我们怎么通过寄存器去控制整个数据通路。这些内容不涉及复杂的数学推导，重在建立对RDMA硬件底层的直觉认知。

### 1.1 从DMA到RDMA：数据搬运的进化

先看一个最基础的问题：CPU把数据从网卡搬到内存，这个过程经历了什么？传统方式里，CPU得亲自参与，每来一个包就触发一次中断，CPU把数据从网卡FIFO搬到内核缓冲区，再拷贝到用户态。这个过程中，CPU被大量占用来做数据搬运，真正处理业务逻辑的时间反而少了。这就是所谓的“数据拷贝开销”。

DMA（Direct Memory Access）的出现就是为了解决这个问题。DMA控制器可以独立于CPU，直接把外设的数据搬到内存。CPU只需要配置好DMA的源地址、目的地址和传输长度，然后就可以去干别的事了。DMA完成后会发一个中断通知CPU。这样CPU从搬运工变成了管理者，效率提升明显。

但传统DMA有个局限：它只能在一台机器内部做数据搬运。如果数据要从机器A的网卡搬到机器B的内存，传统DMA就做不到了。这时候需要CPU在两边都参与：机器A的CPU把数据从内存搬到网卡发送缓冲区，机器B的CPU把数据从网卡接收缓冲区搬到内存。这就是所谓的“两次拷贝”。

RDMA（Remote Direct Memory Access）把这个过程又往前推了一步：它允许一台机器的网卡直接读写另一台机器的内存，中间完全不需要CPU参与。这个“直接”两个字，就是RDMA的核心价值。我们来看一下两者的对比：

| 特性 | 传统DMA | RDMA |
| --- | --- | --- |
| 数据搬运范围 | 单机内部（外设↔内存） | 跨机器（内存↔内存） |
| CPU参与度 | 配置后无需参与搬运 | 完全零参与（Zero-Copy） |
| 数据传输单位 | 字节流（无协议封装） | 消息/数据包（有完整协议栈） |
| 典型延迟 | 微秒级（受总线争用影响） | 亚微秒级（1-3μs） |
| 典型带宽 | 受限于PCIe带宽（如8GT/s） | 100Gbps-400Gbps（无损网络） |

从这张表可以看出，RDMA不是简单的“远程版DMA”，它在协议栈、延迟、带宽上都做了专门的优化。RDMA的核心思路是把整个数据通路做成一个硬件卸载的流水线，从物理层到传输层，全部在网卡内部完成。

### 1.2 协议栈分层：谁在负责什么

RDMA协议栈不是凭空出现的，它建立在现有的网络分层模型之上。我们来看一个典型的RDMA协议栈分层：

```arch-diagram
┌─────────────────────────────────────────┐
│            RDMA应用层                    │
│  (Verbs API / Libibverbs / RDMA-CM)     │
└────────────────┬────────────────────────┘
                 │  Verbs接口
┌────────────────▼────────────────────────┐
│          RDMA传输层                      │
│  (RC / UC / UD / XRC 传输模式)          │
└────────────────┬────────────────────────┘
                 │  数据包封装
┌────────────────▼────────────────────────┐
│          RDMA网络层                      │
│  (RoCEv2: UDP/IP / InfiniBand: LRH)     │
└────────────────┬────────────────────────┘
                 │  网络路由
┌────────────────▼────────────────────────┐
│          链路层                          │
│  (以太网MAC / InfiniBand L2)            │
└────────────────┬────────────────────────┘
                 │  物理传输
┌────────────────▼────────────────────────┐
│          物理层                          │
│  (25G/50G/100G SerDes / 光纤/铜缆)      │
└─────────────────────────────────────────┘
```

RDMA协议栈分层结构，每层职责清晰，硬件卸载逐层完成

这个分层结构里，最关键的是传输层和网络层。传输层定义了数据如何可靠传输（RC模式）或不可靠传输（UD模式）。网络层决定了数据包怎么路由到远端。在RoCEv2中，网络层用的是UDP/IP，这意味着RoCEv2可以跑在标准以太网上，不需要专用的InfiniBand交换机。

每一层的处理，在RDMA网卡内部都是由硬件逻辑完成的。比如传输层的重传、排序、确认，这些在传统TCP协议里由CPU软件处理的工作，RDMA网卡用硬件状态机直接搞定。这样做的结果就是延迟极低——因为不需要上下文切换，不需要中断处理，数据直接从网卡硬件流到内存。

### 1.3 ARM Cortex在智能网卡中的角色

说到这里，有人会问：那RDMA网卡里到底有没有CPU？答案是有的，但不是用来做数据搬运的。现代智能网卡（比如NVIDIA ConnectX系列、Intel IPU）内部都集成了ARM Cortex核心，通常是一个或多个Cortex-A72或Cortex-A55。这个ARM核不是用来处理数据包的，而是用来做控制面管理的。

具体来说，ARM核负责以下几件事：

- **初始化配置**：上电后，ARM核加载固件，配置PCIe接口、DMA引擎、MAC控制器等硬件模块。
- **连接管理**：处理QP（Queue Pair）的创建、修改、删除。当应用调用Verbs API创建一个QP时，这个请求最终会通过驱动下发到ARM核，ARM核在硬件中分配资源。
- **错误处理**：当链路断开、CRC错误、重传超时发生时，ARM核负责记录错误状态，触发中断或回调。
- **统计收集**：收集端口统计、丢包计数、延迟直方图等监控数据，供上层管理软件使用。

ARM核运行的是一个轻量级的实时操作系统（比如FreeRTOS或自定义的RTOS），所有控制面代码都在这个RTOS上跑。数据面则完全由硬件流水线处理，ARM核不参与任何数据包的转发或处理。这种“控制面软件化、数据面硬件化”的架构，是RDMA网卡高性能的关键。

#### 信息

在实际项目中，ARM核的频率通常在1-2GHz，L1缓存32KB，L2缓存256KB-1MB。这个配置对于控制面处理绰绰有余，但远不足以处理数据面（数据面需要几百Gbps的吞吐）。所以千万别想着用ARM核去处理数据包，那是硬件流水线的活。

### 1.4 寄存器级编程接口：直接控制硬件

理解了ARM核的角色，我们再来看一个更底层的东西：寄存器。RDMA网卡对外暴露的编程接口，归根结底是一组内存映射的寄存器（MMIO）。无论是驱动还是ARM核固件，最终都是通过读写寄存器来控制网卡行为的。

我们以RoCEv2网卡为例，核心寄存器包括：

| 寄存器名称 | 偏移地址（示例） | 位宽 | 功能描述 |
| --- | --- | --- | --- |
| CMD\_DOORBELL | 0x1000 | 32bit | 门铃寄存器：通知硬件有新工作请求 |
| QP\_CTX\_BASE | 0x2000 | 64bit | QP上下文基地址：指向内存中的QP状态表 |
| CQ\_DB | 0x3000 | 32bit | 完成队列门铃：通知硬件消费完成事件 |
| PORT\_STAT | 0x4000 | 64bit | 端口统计：累计收发包计数、CRC错误计数 |
| INT\_MASK | 0x5000 | 32bit | 中断屏蔽寄存器：控制哪些事件触发中断 |

这些寄存器都是通过PCIe BAR空间映射到主机内存地址空间的。驱动在初始化时，通过pci\_ioremap\_bar()把BAR空间映射到内核虚拟地址，然后就可以直接用readl/writel来读写寄存器。

举个例子，当应用要发送一个数据包时，驱动的工作流程是：

1. 把要发送的数据地址和长度写入内存中的工作请求（WQE）。
2. 写门铃寄存器（CMD\_DOORBELL），告诉硬件“有新任务”。
3. 硬件看到门铃更新后，从内存中读取WQE，解析出源地址、目的地址、QP号等信息。
4. 硬件DMA引擎直接从应用缓冲区搬数据，封装成RoCEv2数据包发送出去。

整个过程中，CPU只做了写门铃这一个操作，剩下的全部由硬件完成。这就是RDMA高性能的底层实现。

### 1.5 内存注册与MR机制

RDMA能直接访问远端内存，但不是说随便一块内存都能被访问。在RDMA协议中，内存必须提前注册（Memory Registration），注册后得到一个内存区域（MR，Memory Region）。MR记录了内存的物理地址、长度、访问权限（读/写/本地/远端）。

为什么需要注册？两个原因：

- **物理地址固定**：用户态应用使用的虚拟地址，在RDMA操作期间不能被换出或迁移。注册操作会锁定物理页面（mlock），保证DMA操作时物理地址不变。
- **权限控制**：远端只能访问被显式授权的内存区域。比如你只注册了一个4KB的缓冲区，远端想读写这个缓冲区之外的地址，硬件会直接拒绝并产生错误。

注册MR时，驱动会分配一个lkey（本地key）和rkey（远程key）。本地key用于本端DMA操作，远程key用于远端访问。这两个key是32位的整数，在数据包中作为元数据携带。硬件在访问内存时会校验key，不匹配就丢弃。

#### 严重警告

MR注册是一个开销较大的操作（涉及页表遍历、物理页面锁定、TLB刷新）。在实际项目中，不要在热路径上频繁注册/注销MR。正确的做法是在初始化阶段预先注册好所有需要的内存池，然后在运行时复用这些MR。一次注册的开销大约在10-50微秒，但复用后每次操作的额外开销几乎为零。

### 1.6 队列对（QP）与完成队列（CQ）

RDMA通信的核心抽象是队列对（QP，Queue Pair）。一个QP包含两个队列：发送队列（SQ）和接收队列（RQ）。应用把工作请求（WR）投递到SQ或RQ，硬件消费这些请求并产生完成事件（CQE）放到完成队列（CQ）中。

QP的状态机是RDMA编程中最容易出错的地方。一个QP有多个状态：RESET、INIT、RTR（Ready to Receive）、RTS（Ready to Send）、SQD（SQ Draining）、SQE（SQ Error）、ERR。状态转换必须严格按照协议进行，比如从INIT到RTR需要配置好远端QP号、GID、访问权限等参数。

我们来看一个典型的QP状态转换流程：

```arch-diagram
         ┌─────────────┐
         │   RESET     │
         └──────┬──────┘
                │  Modify QP to INIT
         ┌──────▼──────┐
         │    INIT     │
         └──────┬──────┘
                │  Modify QP to RTR (配置远端信息)
         ┌──────▼──────┐
         │    RTR      │  ← 此时可以接收数据
         └──────┬──────┘
                │  Modify QP to RTS
         ┌──────▼──────┐
         │    RTS      │  ← 此时可以发送和接收
         └──────┬──────┘
                │  错误发生
         ┌──────▼──────┐
         │    ERR      │  ← 需要重置QP才能恢复
         └─────────────┘
```

QP状态机：每个状态转换都需要明确的Modify QP命令

在工程实践中，最常见的错误就是状态转换参数配置错误。比如在配置RTR时，远端QP号填错了，或者GID（Global Identifier）不对，会导致连接建立失败。调试这类问题的方法是用ibv\_devinfo查看端口属性，用ibv\_qp\_state查看当前QP状态，确认每一步的配置参数。

### 1.7 硬件卸载：传输层在内卡里

最后我们深入看一下传输层是怎么卸载到硬件里的。在RoCEv2中，传输层协议是RC（Reliable Connection）或UC（Unreliable Connection）。以RC模式为例，硬件需要实现：

- **序列号管理**：每个数据包携带一个PSN（Packet Sequence Number），接收端按PSN排序。
- **ACK/NACK生成**：收到数据包后，硬件自动生成ACK或NACK响应。
- **超时重传**：发送端硬件维护一个重传定时器，超时未收到ACK则自动重传。
- **滑动窗口**：控制未确认的数据包数量，防止接收端过载。

所有这些功能，在传统TCP/IP协议栈里都是由CPU软件实现的。但在RDMA网卡中，这些逻辑被固化在硬件状态机里，用门电路和片上SRAM实现。硬件实现的优势是延迟低（纳秒级）、吞吐高（几百Gbps）、功耗低（每比特能耗远低于CPU）。

一个典型的RC模式数据包交互流程如下：

```arch-diagram
发送端                              接收端
  │                                  │
  │  ┌──────────────────────┐        │
  │  │  Send Request (WR)   │        │
  │  │  QP=1, PSN=100       │        │
  │  │  数据: 4KB           │        │
  │  └──────────┬───────────┘        │
  │             │                     │
  │   硬件封装为RoCEv2包              │
  │   添加BTH+ETH+IP+UDP头           │
  │             │                     │
  │             │  ──── 数据包 ─────→ │
  │             │                     │  硬件解析包头
  │             │                     │  校验PSN
  │             │                     │  DMA写入内存
  │             │                     │  生成ACK
  │             │  ←─── ACK ──────── │
  │             │                     │
  │   收到ACK                         │
  │   释放WQE                         │
  │   生成CQE                         │
  │   通知应用                        │
```

RC模式一次可靠传输的完整硬件交互流程

从这个图可以看到，整个传输过程完全不需要CPU参与。CPU只在最开始投递WR，最后从CQ中取CQE。中间的序列号管理、重传、确认，全是硬件自动完成的。这也是为什么RDMA能实现1-3微秒的端到端延迟，而传统TCP/IP在相同条件下需要10-50微秒。

#### 章节小结

本章我们从DMA的局限性出发，引出了RDMA的核心思想：让网卡直接读写远端内存，CPU零参与。然后逐层分析了RDMA协议栈的分层结构，重点讲解了ARM Cortex在智能网卡中的控制面角色、寄存器级编程接口、内存注册机制、QP状态机以及传输层硬件卸载的实现原理。这些内容为后续章节深入理解RoCEv2协议细节和网络设计打下了坚实的硬件基础。

#### 本章重点回顾

- RDMA的本质是“远程DMA”，通过硬件卸载实现CPU零参与的数据传输，延迟可低至1-3μs。
- ARM Cortex核负责控制面（初始化、连接管理、错误处理），不参与数据面转发。
- 寄存器是RDMA编程的底层接口，门铃寄存器（Doorbell）是通知硬件有任务的关键机制。
- 内存注册（MR）锁定物理页面并提供访问权限控制，lkey/rkey是硬件校验的关键。
- QP状态机必须严格遵循RESET→INIT→RTR→RTS的转换顺序，配置参数错误是常见故障原因。
- 传输层（RC/UC）的序列号、ACK、重传全部由硬件状态机实现，这是RDMA高性能的核心。
