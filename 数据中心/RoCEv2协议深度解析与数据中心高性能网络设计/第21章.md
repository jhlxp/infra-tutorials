# 第21章：大规模集群部署案例

本章我们直接进入实战，来看一个典型的RoCEv2大规模集群部署案例。很多工程师在实验室里搭过小规模RoCE网络，跑几个节点没问题，但一旦扩展到几百甚至上千节点，各种问题就冒出来了：PFC死锁、ECMP哈希不均、拥塞控制参数调不灵。这一章我们就拿一个真实的AI训练集群项目来拆解，从网络架构设计、交换机配置、服务器网卡调优，到最终性能验收，把整个部署流程和踩坑点过一遍。你会看到每个环节的具体参数和配置命令，而不是泛泛而谈的架构图。

### 21.1 项目背景与需求分析

这个项目是为一家自动驾驶公司搭建的AI训练集群，规模是128台GPU服务器，每台服务器8张A100 GPU，总共1024块GPU。训练任务主要是大规模Transformer模型，数据并行加模型并行混合训练，节点间通信量巨大。客户要求端到端带宽利用率达到95%以上，并且在大规模allreduce操作中，尾延迟不超过200微秒。

我们来看一下具体的需求参数。网络层面，每台服务器配置两张ConnectX-7 200Gbps网卡，分别连接到两台leaf交换机做冗余。Spine-leaf架构，leaf交换机16台，spine交换机4台，每台leaf上行到所有spine。服务器网卡开启RoCEv2，使用DCQCN拥塞控制，PFC开启，MTU设置为9000字节。

| 参数项 | 规格 | 说明 |
| --- | --- | --- |
| 服务器数量 | 128台 | 每台8×A100 GPU |
| 网卡型号 | ConnectX-7 200Gbps | 双口，每服务器2张 |
| 交换机型号 | Spectrum-4 64×200G | leaf/spine同型号 |
| 网络拓扑 | Spine-Leaf 4×16 | 4 spine, 16 leaf |
| 链路带宽 | 200Gbps | 全链路对称 |
| MTU | 9000字节 | Jumbo帧 |
| 拥塞控制 | DCQCN | 参数按推荐配置 |
| 流控机制 | PFC | 每个优先级独立 |
| 目标带宽利用率 | ≥95% | 端到端 |
| 尾延迟要求 | ≤200μs | allreduce操作 |

这个需求在单机测试时看起来不难，但一旦扩展到全集群，问题就来了。我们先从网络架构设计开始，看看怎么满足这些指标。

### 21.2 网络拓扑与布线方案

我们采用标准的Spine-Leaf架构，但布线方案上有讲究。128台服务器，每台2张网卡，共256个200G端口。16台leaf交换机，每台leaf有64个端口，我们用其中16个端口接服务器（每leaf接8台服务器×2网卡=16端口），剩下的48个端口全部上行到4台spine交换机。

具体来说，每台leaf到每台spine连12条200G链路，总共48条上行。这样spine交换机也是64端口，每台spine连16台leaf，每leaf12条链路，占用192端口，剩余端口留作扩展。

布线时要注意，每台leaf上连接的8台服务器，它们的网卡端口要均匀分布到leaf的不同端口组，避免端口组内共享背板带宽。比如Mellanox Spectrum-4交换机内部有多个ASIC，端口0-15共享一个ASIC，16-31共享另一个，如果8台服务器全连到同一个ASIC上，上行带宽就会受限。我们把服务器端口打散到不同ASIC上，确保每台leaf的上行带宽充足。

### 21.3 交换机配置要点

交换机配置是部署中最容易出错的地方。我们先说PFC配置。RoCEv2依赖PFC做无损传输，但PFC配置不好会导致死锁。我们只在RoCE流量的优先级上开启PFC，其他流量不开启。具体配置如下：

```bash
# 进入配置模式
configure terminal

# 创建优先级组，只对优先级3开启PFC
interface ethernet 1/1-48
  priority-flow-control mode on
  priority-flow-control priority 3 on
  priority-flow-control priority 0-2,4-7 off

# 设置流量类别映射
qos rewrite-pcp enable
qos trust pcp

# 将RoCE流量映射到优先级3
class-map type qos match-all roce
  match cos 3

policy-map type qos roce-policy
  class roce
    set priority 3

# 应用策略
interface ethernet 1/1-48
  service-policy type qos input roce-policy
  service-policy type qos output roce-policy
```

另一个关键配置是ECMP哈希算法。RoCEv2流是五元组（源IP、目的IP、源端口、目的端口、协议），默认的ECMP哈希基于IP和端口，但AI训练中大量流的目的IP都是同一个集合（比如同一个allreduce组的IP），导致哈希不均。我们改用自定义哈希，加入流标签（Flow Label）字段。在IPv6 RoCEv2中，Flow Label是20位的，可以极大增加哈希随机性。

```bash
# 启用IPv6 Flow Label哈希
ip load-sharing hash include-flow-label

# 设置哈希因子
ip load-sharing hash ipv6-fields source-ip dest-ip flow-label

# 验证配置
show ip load-sharing
```

#### 严重警告

不要在所有端口上开启PFC！很多工程师图省事，在交换机上全局开启PFC，结果导致控制流量和存储流量也被PFC反压，造成死锁。必须只对RoCE流量所在的优先级开启PFC，其他优先级关闭。另外，PFC的watchdog定时器要设置合理，默认值通常太大，建议设置为100ms，超过就触发恢复动作。

### 21.4 服务器网卡调优

服务器端网卡配置同样关键。ConnectX-7网卡有大量可调参数，我们直接给出生产环境验证过的配置。首先是DCQCN参数，这是拥塞控制的核心。

```bash
# 查询当前DCQCN参数
mlxreg -d /dev/mst/mt4125_pciconf0 --reg_name DCQCN --get

# 设置DCQCN参数（推荐值）
mlxreg -d /dev/mst/mt4125_pciconf0 --reg_name DCQCN --set \
  --field rtt_response_time 10 \
  --field rate_reduce_monitor_period 4 \
  --field initial_alpha_value 128 \
  --field alpha_gain 1 \
  --field rate_to_set 100

# 启用硬件拥塞控制
mlxconfig -d /dev/mst/mt4125_pciconf0 set \
  ROCE_CC_ALGORITHM=DCQCN \
  ROCE_CC_PRIORITY=3 \
  ROCE_CC_FORCE=1
```

然后是网卡中断绑定。200Gbps网卡中断频率极高，如果不做CPU亲和性绑定，中断会分散到所有CPU核心，导致缓存抖动。我们把网卡中断绑定到专用的几个CPU核心上，这些核心不运行业务进程。

```bash
# 查看网卡中断号
cat /proc/interrupts | grep mlx5

# 绑定中断到CPU 2-5（示例）
echo 2 > /proc/irq/124/smp_affinity
echo 3 > /proc/irq/125/smp_affinity
echo 4 > /proc/irq/126/smp_affinity
echo 5 > /proc/irq/127/smp_affinity

# 使用irqbalance工具，设置白名单
systemctl stop irqbalance
# 手动绑定后，确保irqbalance不干扰
systemctl mask irqbalance
```

另外，MTU设置为9000字节可以大幅减少报文头开销。在200Gbps链路上，小包（1500字节）的报文头占比约3%，而9000字节时不到0.5%。这对带宽利用率影响很大。

### 21.5 性能验证与调优

部署完成后，我们做全集群性能验证。使用NVIDIA的HPC-X工具包中的`osu_allreduce`基准测试，测量不同消息大小下的带宽和延迟。

测试结果如下表所示。注意看，在消息大小4MB时，带宽利用率达到98.7%，尾延迟98微秒，满足客户要求。但在128KB小消息时，带宽只有65%，这是因为小消息的协议开销占比大，且DCQCN在小流上收敛慢。

| 消息大小 | 平均带宽(GB/s) | 带宽利用率 | 平均延迟(μs) | 尾延迟(μs) |
| --- | --- | --- | --- | --- |
| 128KB | 12.8 | 65% | 45 | 78 |
| 512KB | 16.5 | 84% | 52 | 92 |
| 2MB | 18.9 | 96% | 68 | 110 |
| 4MB | 19.4 | 98.7% | 72 | 98 |
| 16MB | 19.6 | 99% | 85 | 120 |

针对小消息性能差的问题，我们做了优化：将NVIDIA集合通信库中的`NCCL_ALGO`从`Ring`改为`Tree`，因为Tree算法在小消息上延迟更低。同时调整了DCQCN的`rate_reduce_monitor_period`从4降到2，让拥塞响应更快。优化后128KB消息带宽提升到14.2GB/s，延迟降到35μs。

### 21.6 常见问题与排障

部署过程中遇到几个典型问题，这里记录一下。第一个是PFC死锁。现象是某个leaf交换机端口持续发送PFC反压帧，导致该端口下所有服务器通信中断。排查后发现是某个服务器网卡驱动版本过旧，在收到PFC反压后没有正确停止发送，导致交换机缓冲溢出。解决办法是统一升级网卡固件到最新版本。

第二个问题是ECMP哈希不均。我们用`perfquery`工具查看链路利用率，发现某条spine到leaf的上行链路利用率达到98%，而另一条只有30%。原因是默认哈希算法只用了IP和端口，而AI训练中大量流的目的IP是同一个集合（比如同一组GPU的IP），导致哈希碰撞。解决办法就是前面说的启用Flow Label哈希。

第三个问题是DCQCN参数不匹配。交换机侧和网卡侧的DCQCN参数必须一致，否则拥塞信号无法正确解析。我们遇到过交换机发送的CNP报文格式不对，网卡无法识别，导致拥塞控制失效。使用`mlxreg`工具检查双方参数，确保`rtt_response_time`和`rate_reduce_monitor_period`一致。

#### 信息

排障时推荐使用Mellanox的`ufm`工具（Unified Fabric Manager），它可以可视化展示全网PFC状态、缓冲区占用和链路利用率。另外，`ethtool -S`命令可以查看网卡侧的PFC计数，如果`tx_pause`和`rx_pause`计数不对称增长，说明有PFC风暴。

### 21.7 运维监控体系

集群上线后，运维监控必须跟上。我们部署了Prometheus + Grafana监控体系，采集以下关键指标：

- 每端口PFC帧计数（tx\_pause, rx\_pause）
- 每端口缓冲区占用率（通过交换机MIB获取）
- ECMP链路利用率标准差（衡量哈希均衡度）
- DCQCN速率变化曲线（每个网卡）
- allreduce操作的平均延迟和尾延迟

我们设置告警阈值：当PFC帧速率超过1000帧/秒时触发警告，超过5000帧/秒触发严重告警。当ECMP链路利用率标准差超过20%时触发告警，需要重新调整哈希策略。当allreduce尾延迟超过200μs时触发严重告警，需要立即介入。

另外，我们还做了自动化巡检脚本，每天凌晨低负载时执行全集群的带宽和延迟测试，生成报告。如果发现性能下降超过5%，自动通知运维人员。

### 21.8 经验总结与最佳实践

这个项目做完后，我们总结了几条关键经验。第一，RoCEv2大规模部署，PFC配置是最大的坑。必须严格限制PFC的作用范围，只在RoCE优先级开启，并且交换机侧和网卡侧的PFC参数要一致。第二，ECMP哈希必须用Flow Label，否则大规模集群下哈希不均必然发生。第三，DCQCN参数不是通用的，需要根据实际流量模型调优。我们的经验是，AI训练场景下，`rtt_response_time`设置在8-12之间，`rate_reduce_monitor_period`设置在2-4之间效果最好。

另外，不要迷信默认配置。Mellanox交换机出厂默认PFC是关闭的，默认ECMP哈希算法也不包含Flow Label。所有参数必须根据实际需求手动配置。还有，网卡固件版本一定要统一，我们遇到过混合版本导致PFC死锁的问题。

#### 章节小结

本章通过一个128节点AI训练集群的部署案例，详细拆解了RoCEv2大规模集群从网络架构设计、交换机配置、服务器网卡调优到性能验证的全过程。我们看到了PFC死锁、ECMP哈希不均、DCQCN参数不匹配等实际问题的表现和解决方法。关键结论是：PFC必须精准配置、ECMP必须使用Flow Label、DCQCN参数必须与流量模型匹配。这些经验直接适用于任何大规模RoCEv2集群的部署。

#### 本章重点回顾

- PFC只对RoCE优先级开启，其他优先级关闭，避免死锁
- ECMP哈希必须包含Flow Label字段，确保大流量的负载均衡
- DCQCN参数需要根据实际流量调优，AI训练场景推荐rtt\_response\_time=10, rate\_reduce\_monitor\_period=2
- 网卡中断绑定到专用CPU核心，避免缓存抖动影响性能
- 小消息场景下使用Tree算法替代Ring算法，提升带宽利用率
