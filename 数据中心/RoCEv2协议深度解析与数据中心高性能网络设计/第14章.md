# 第14章：网络延迟测量与优化

RoCEv2网络的核心优势在于超低延迟，但实际部署中，从网卡到交换机再到对端网卡，每一个环节都可能引入额外的延迟。我们做网络工程师的，最怕的就是用户说“网络卡”，但到底卡在哪里？是网卡处理慢了，还是交换机拥塞了，还是对端应用没及时收？这一章我们就来系统地聊一聊RoCEv2网络中的延迟构成、测量方法以及优化手段。你会学到如何用硬件时间戳精确测量一跳延迟，如何识别微突发导致的抖动，以及通过调整PFC水位、中断合并、PCIe参数等手段把延迟压到极致。

### 14.1 延迟的构成：从端到端的拆解

在RoCEv2网络中，一次数据包从源端应用发送到目的端应用，经历的延迟可以分为几个部分。我们来看一个典型的路径：应用调用`ibv_send`，数据从用户态拷贝到内核态（如果是Kernel Bypass则跳过），然后通过PCIe总线到达网卡，网卡进行DMA操作，封装RoCEv2头部，再通过SerDes发送到交换机，交换机做转发，最后对端网卡接收、DMA到内存，应用读到数据。这里面每一个环节的延迟都要单独看。

我整理了一个典型RoCEv2场景下的延迟分解表，基于Mellanox ConnectX-5网卡和25Gbps链路，CPU为Intel Xeon Gold 6248，关闭中断合并，实测数据如下：

| 延迟环节 | 典型值(us) | 影响因素 | 优化空间 |
| --- | --- | --- | --- |
| 应用发送API调用 | 0.2 - 0.5 | CPU频率、缓存命中率 | 使用DPDK/SPDK，减少上下文切换 |
| PCIe传输(Gen3 x16) | 0.3 - 0.6 | PCIe负载、Max Payload Size | 调整MPS为512B，开启TLP处理 |
| 网卡处理(封装+序列化) | 0.5 - 1.0 | 网卡型号、固件版本 | 升级固件，关闭不必要的offload |
| 链路传输(25Gbps, 10m线缆) | 0.05 - 0.1 | 线缆长度、FEC模式 | 使用DAC线缆，开启RS-FEC |
| 交换机转发(单跳) | 0.3 - 0.8 | 交换机芯片、ACL规则数量 | 使用低延迟交换机，简化ACL |
| 对端网卡接收+DMA | 0.5 - 1.2 | 中断处理、内存带宽 | 使用CPU亲和性，NUMA绑定 |
| 应用接收API调用 | 0.2 - 0.5 | 应用轮询效率 | 使用Busy Polling模式 |
| **端到端总延迟(典型)** | **2.1 - 4.7** | 综合因素 | 逐项优化可降至1.5us以下 |

从这个表可以看出来，网卡处理和对端DMA是最大的两个大头，加起来占了将近一半。交换机转发虽然看起来不大，但在多跳拓扑中，每跳累加就很可观了。所以优化延迟，首先要搞清楚你的瓶颈在哪一环。

#### 信息

实际项目中，我们常用`perftest`工具中的`ib_write_lat`来测量端到端延迟，但要注意这个测量值包含了所有环节，无法直接定位问题。需要配合硬件计数器逐段排查。

### 14.2 硬件时间戳：精确测量延迟的基石

要精确测量网络延迟，光靠软件打时间戳是不够的，因为软件本身就有调度延迟。RoCEv2网卡普遍支持硬件时间戳（HW Timestamping），在数据包进出网卡的物理层时，硬件会打上一个精确的时间戳，精度可以达到纳秒级。我们来看一下怎么在RoCEv2环境中启用硬件时间戳。

首先，确认网卡是否支持硬件时间戳。以Mellanox ConnectX-5为例，用`mlxconfig`工具查看：

```bash
# 查看当前网卡的PTP配置
mlxconfig -d /dev/mst/mt4119_pciconf0 query | grep PTP

# 如果显示 PTP_ENABLE = True，说明已启用
# 如果为 False，需要设置
mlxconfig -d /dev/mst/mt4119_pciconf0 set PTP_ENABLE=1

# 重启网卡驱动生效
/etc/init.d/openibd restart
```

启用之后，在应用程序中可以通过`struct ibv_cq_ex`的`wr_id`和`timestamp`字段获取硬件时间戳。这里有一个关键点：硬件时间戳只在`IBV_SEND_SIGNALED`的WR上才会被记录，而且需要CQ的`comp_mask`包含`IBV_CQ_EX_TIMESTAMP`。下面是一个简化的代码片段，展示如何读取发送和接收的时间戳：

```c
#include <infiniband/verbs.h>

struct ibv_cq_ex *cq_ex;
struct ibv_wc wc;
uint64_t send_ts, recv_ts;

// 创建CQ时开启扩展模式
struct ibv_cq_init_attr_ex cq_attr = {
    .cqe = 1024,
    .cq_context = NULL,
    .channel = NULL,
    .comp_mask = IBV_CQ_INIT_ATTR_MASK_FLAGS,
    .flags = IBV_CREATE_CQ_ATTR_SINGLE_THREADED
};
cq_ex = ibv_create_cq_ex(pd->context, &cq_attr);

// 轮询CQ并获取时间戳
while (1) {
    if (ibv_start_poll(cq_ex, &wc) == 0) {
        // 获取发送完成的时间戳
        if (wc.opcode == IBV_WC_SEND) {
            ibv_wc_read_completion_ts(cq_ex, &send_ts);
            printf("Send completion timestamp: %lu ns\n", send_ts);
        }
        // 获取接收完成的时间戳
        if (wc.opcode == IBV_WC_RECV) {
            ibv_wc_read_completion_ts(cq_ex, &recv_ts);
            printf("Recv completion timestamp: %lu ns\n", recv_ts);
        }
        ibv_end_poll(cq_ex);
    }
}
```

有了发送和接收的时间戳，我们就可以计算单向延迟了。不过要注意，发送端和接收端的时钟可能有偏差，所以更精确的做法是使用PTP（Precision Time Protocol）同步两边时钟，或者使用两端的差值取平均。在测试环境中，我们通常用`ib_write_lat -T`来启用硬件时间戳模式，它会自动处理时钟同步问题。

### 14.3 微突发与抖动：延迟的隐形杀手

很多时候，平均延迟看起来很低，但偶尔会有几个包延迟飙升到几十微秒甚至几百微秒。这种抖动（Jitter）对分布式存储和HPC应用的影响非常大，因为它会导致超时重传，进而引发雪崩效应。造成抖动的主要原因之一就是微突发（Microburst）。

微突发指的是在极短的时间窗口内（比如几十微秒），流量瞬间超过链路带宽，导致交换机缓存被填满，触发PFC反压或者直接丢包。因为时间极短，传统的SNMP轮询（5分钟一次）根本抓不到，必须用硬件计数器来抓。

我们来看一下如何在Mellanox交换机上捕获微突发。使用`mlxreg`工具读取端口的水线统计：

```bash
# 读取端口1的共享缓存占用情况
mlxreg -d /dev/mst/mt52100_pciconf0 --reg_name STB --get "port=1,page=0"

# 输出示例：
# STB: port=1, page=0
#   shared_occupancy: 1200 cells (每cell 64字节)
#   headroom_occupancy: 80 cells
#   total_occupancy: 1280 cells

# 如果 shared_occupancy 接近端口缓存上限（比如1500 cells），说明有微突发
```

这里有个工程经验：当`shared_occupancy`超过缓存上限的80%时，PFC就会开始反压，导致发送端暂停。而PFC本身又会引入额外的延迟和抖动。所以优化微突发的核心思路有两个：一是让流量更平滑（比如使用流量整形），二是增大缓存（但成本高）。对于RoCEv2网络，更推荐前者。

我们可以通过调整DCQCN（Data Center Quantized Congestion Notification）参数来缓解微突发。DCQCN是RoCEv2的标准拥塞控制算法，它通过CNP（Congestion Notification Packet）反馈来降速。但默认参数比较保守，容易过激。实际调优时，可以把`alpha`（增益因子）从默认的1/512调大到1/128，让降速更平滑，同时把`rate_reduce_period`从默认的55us缩短到30us，加快恢复速度。

#### 严重警告

不要随意调大`alpha`，否则会导致带宽利用率下降。建议在测试环境先做梯度实验，从1/256开始，逐步调整到1/64，观察延迟和吞吐的trade-off。

### 14.4 PFC水位调优：平衡延迟与丢包

PFC（Priority Flow Control）是RoCEv2网络保证无损传输的关键机制。但PFC是一把双刃剑：水位设得太低，容易触发反压，增加延迟；水位设得太高，又容易导致缓存溢出丢包。所以调优PFC水位是延迟优化的必修课。

PFC有两个关键参数：`xon`（开启阈值）和`xoff`（关闭阈值）。当缓存占用超过`xoff`时，交换机向发送端发送PFC暂停帧；当缓存占用降到`xon`以下时，发送端恢复发送。这两个参数的差值就是所谓的“滞后区间”。

对于延迟敏感的应用，我们希望滞后区间尽量小，这样反压能快速恢复。但太小的话，如果微突发频繁，PFC帧也会频繁触发，反而增加CPU开销。一般建议`xoff`设为端口缓存大小的40%，`xon`设为30%，差值10%。比如端口缓存为192KB，那么`xoff`=76.8KB，`xon`=57.6KB。

在Mellanox交换机上，可以通过以下命令调整：

```bash
# 设置端口1的PFC阈值
mlxreg -d /dev/mst/mt52100_pciconf0 --reg_name PPTB --set "port=1,priority=3,xoff=76800,xon=57600"

# 验证配置
mlxreg -d /dev/mst/mt52100_pciconf0 --reg_name PPTB --get "port=1,priority=3"
```

调完以后，一定要用`ethtool -S eth0 | grep pfc`观察PFC帧的数量。如果PFC帧数量明显下降，说明水位设置合理；如果PFC帧反而增多，说明滞后区间太小，需要适当调大。

### 14.5 中断合并与轮询模式的选择

网卡收到数据包后，有两种方式通知CPU：中断和轮询。中断延迟低，但高PPS下CPU开销大；轮询CPU占用高，但延迟更稳定。RoCEv2场景下，我们通常使用混合模式：在低流量时用中断，高流量时自动切换到轮询。

Mellanox网卡支持一个叫做`adaptive-rx`的功能，可以根据流量动态调整中断合并参数。我们来看一下如何配置：

```bash
# 查看当前中断合并设置
ethtool -c eth0

# 输出示例：
# Coalesce parameters for eth0:
# Adaptive RX: on  TX: off
# RX sample interval: 1us
# RX max coalesce frames: 16
# RX max coalesce usecs: 4

# 对于延迟敏感场景，建议关闭adaptive，手动设置
ethtool -C eth0 adaptive-rx off rx-usecs 1 rx-frames 1

# 对于吞吐优先场景，可以增大合并值
ethtool -C eth0 adaptive-rx on rx-usecs 4 rx-frames 16
```

如果应用本身是轮询模式（比如DPDK或SPDK），那么中断合并就完全没有意义了，反而会增加延迟。这种情况下，应该直接关闭中断，使用`ibv_create_cq`时设置`IBV_CQ_INIT_ATTR_MASK_FLAGS`为`IBV_CREATE_CQ_ATTR_SINGLE_THREADED`，然后在应用里用`ibv_poll_cq`轮询。实测下来，轮询模式比中断模式延迟能降低30%以上，但CPU占用率会接近100%。

这里有一个折中方案：使用`busy_poll`功能。Linux内核从4.5版本开始支持`SO_BUSY_POLL`，可以在套接字层面实现忙等轮询。对于RoCEv2的`ibv`接口，可以通过设置`IBV_QP_CREATE_IGNORE_SQ_OVERFLOW`和`IBV_QP_CREATE_IGNORE_RQ_OVERFLOW`来启用类似行为。不过这个功能在ConnectX-6以上的网卡才支持得比较好。

### 14.6 PCIe参数对延迟的影响

很多人容易忽略PCIe子系统对网络延迟的影响。实际上，数据从网卡到内存的PCIe传输延迟，在总延迟中占比不小。PCIe的`Max Payload Size`（MPS）和`Max Read Request Size`（MRRS）这两个参数，直接影响DMA的效率。

MPS决定了每个TLP（Transaction Layer Packet）能携带的最大数据量。对于RoCEv2，数据包大小通常为1500字节（标准MTU）或9000字节（巨型帧）。如果MPS设置为128字节，那么一个1500字节的包就需要拆成12个TLP，每个TLP都要经过PCIe交换层的仲裁，延迟自然就上去了。建议把MPS设置为512字节或1024字节。

查看和修改PCIe参数的方法：

```bash
# 查看网卡所在PCIe设备的MPS
lspci -s 0000:03:00.0 -vvv | grep MaxPayload

# 输出示例：
# DevCap: MaxPayload 512 bytes, PhantFunc 0, Latency L0s <64ns, L1 <1us
# DevCtl: MaxPayload 128 bytes, MaxReadReq 512 bytes

# 注意：DevCtl显示的是当前生效值，DevCap显示的是硬件支持的最大值
# 修改MPS需要进入BIOS或者通过setpci工具（需谨慎）
# 推荐在BIOS中将PCIe Root Port的MPS设为512或1024

# 另一个关键参数：TLP处理提示（TPH）
# 开启TPH可以让网卡告知PCIe控制器数据流的缓存行为
# 目前只在ConnectX-6及以上支持
mlxconfig -d /dev/mst/mt4129_pciconf0 set TPH_ENABLE=1
```

另外，NUMA亲和性也很重要。如果网卡插在CPU0的PCIe槽上，但应用跑在CPU1上，那么数据需要跨NUMA访问，延迟会增加0.5-1us。所以部署时一定要用`numactl`绑定网卡和应用的NUMA节点。

#### 设计要点

在服务器选型时，优先选择PCIe Gen4或Gen5的插槽，并确保网卡和CPU在同一个NUMA节点。对于双口网卡，两个端口可能分属不同NUMA节点，需要仔细查看拓扑图。

### 14.7 端到端延迟优化案例：从4.2us到1.8us

最后分享一个实际调优案例。某分布式存储集群，使用RoCEv2网络，25Gbps链路，Mellanox ConnectX-5网卡，测试`ib_write_lat`延迟为4.2us（64字节消息）。用户觉得太高，要求优化到2us以下。

我们按照以下步骤逐项排查和优化：

1. **检查PFC水位**：发现交换机PFC水位为默认值（xoff=50%），PFC帧频繁触发。调整为xoff=40%，xon=30%，PFC帧减少60%。
2. **调整中断合并**：关闭adaptive-rx，设置rx-usecs=1，rx-frames=1。延迟降低0.5us。
3. **绑定NUMA**：发现应用运行在CPU1，网卡在CPU0。使用`numactl --cpubind=0 --membind=0`重新运行，延迟降低0.8us。
4. **调整PCIe MPS**：BIOS中确认MPS为256字节，改为512字节，延迟降低0.3us。
5. **优化DCQCN参数**：将`alpha`从1/512改为1/256，`rate_reduce_period`从55us改为40us，延迟抖动降低50%。
6. **升级固件**：将网卡固件从16.26.1016升级到16.32.2010，修复了一个DMA延迟bug，延迟降低0.4us。

最终，`ib_write_lat`延迟从4.2us降至1.8us，抖动从±1.2us降至±0.3us。这个案例说明，延迟优化不是单一手段能解决的，需要系统性地从网络、PCIe、CPU、固件等多个维度入手。

#### 章节小结

本章我们深入拆解了RoCEv2网络延迟的构成，从硬件时间戳的精确测量，到微突发的识别与DCQCN调优，再到PFC水位、中断合并、PCIe参数等具体优化手段。延迟优化是一个系统工程，需要结合测量数据逐项排查。记住，没有银弹，只有方法论。

#### 本章重点回顾

- RoCEv2延迟由应用处理、PCIe传输、网卡处理、链路传输、交换机转发、对端DMA等多个环节构成，网卡处理和DMA是主要瓶颈。
- 硬件时间戳是精确测量延迟的基础，需要启用PTP并正确配置CQ扩展接口。
- 微突发是抖动的主要来源，通过DCQCN参数调优（alpha、rate\_reduce\_period）可以缓解。
- PFC水位设置原则：xoff=40%，xon=30%，滞后区间10%，避免频繁反压。
- PCIe MPS建议设为512字节以上，并绑定NUMA亲和性，可降低1us以上延迟。
