# 第5章：PFC与ECN流量控制原理

RoCEv2网络要实现低延迟、零丢包的高性能传输，光有协议封装是不够的。在数据中心里，多个服务器同时发数据，交换机的缓存很容易被占满，导致丢包。一旦丢包，RoCEv2的重传机制会引发延迟剧增，性能断崖式下跌。本章要讲清楚两个核心机制：PFC（优先级流控）和ECN（显式拥塞通知）。PFC负责在链路层面做逐跳的流量刹车，防止缓存溢出；ECN则在网络层面给发送端一个“轻踩刹车”的信号，让流量主动降速。这两个机制配合使用，才能让RoCEv2网络跑出接近理论极限的性能。我们先从最基础的PFC原理讲起。

### 5.1 PFC的基本原理与工作机制

PFC的全称是Priority-based Flow Control，也就是基于优先级的流量控制。它的核心思想很简单：当交换机某个端口的接收缓存快要满的时候，向对端设备发一个暂停帧，让对端暂时停止发送该优先级的数据。这就像高速公路上的收费站，如果出口堵了，收费站就亮红灯让后面的车先等等。

在RoCEv2网络里，我们通常把流量分成不同的优先级。比如，把RDMA流量放在优先级3，把普通TCP流量放在优先级0。PFC只对特定优先级生效，不会影响其他优先级的流量。这样一来，即使TCP流量把缓存占满了，RDMA流量也能有自己的“专用通道”。

PFC的暂停帧是一个标准的以太网MAC控制帧，目的MAC地址是01-80-C2-00-00-01，这个地址是专门给MAC控制帧用的。帧里包含一个关键参数：暂停时间，单位是暂停时间片（一个时间片等于512比特时间的传输时长）。当接收端发现缓存超过阈值（比如80%），就会发暂停帧，告诉对端“你停X个时间片”。对端收到后，会在指定时间内停止发送该优先级的帧。如果接收端缓存恢复正常，可以再发一个暂停时间为0的帧来恢复发送。

这里有个工程上的关键点：PFC的阈值设置非常讲究。阈值设得太低，容易触发频繁的暂停-恢复振荡，导致链路利用率下降；设得太高，缓存又可能溢出丢包。实际项目中，我们一般把触发阈值设为缓存深度的70%-80%，恢复阈值设为50%-60%。

| 参数 | 典型值 | 说明 |
| --- | --- | --- |
| 触发阈值 | 80% | 缓存使用率达到此值时发送暂停帧 |
| 恢复阈值 | 50% | 缓存使用率低于此值时恢复发送 |
| 暂停时间片 | 65535 | 最大暂停时间，约13ms @100Gbps |
| PFC优先级 | 3 | RoCEv2常用优先级3或4 |

PFC虽然能防止丢包，但它有个副作用：可能引发死锁。如果两个交换机互相发暂停帧，谁都不发数据，链路就卡死了。所以实际部署时，必须配合合理的拓扑设计和流量规划，避免出现环路。

### 5.2 PFC的配置方法与调试工具

在数据中心交换机上配置PFC，一般通过命令行或者SDN控制器。我们以常见的交换机为例，看一下具体的配置步骤。首先，需要开启PFC功能，并指定要控制的优先级队列。然后，设置缓存阈值和恢复阈值。最后，还要在服务器网卡上做相应的配置，确保网卡能响应PFC暂停帧。

服务器端的配置也很关键。如果网卡不响应PFC暂停帧，交换机发了也是白发。以Mellanox ConnectX-5网卡为例，可以通过mlxconfig工具开启PFC响应。具体命令是：`mlxconfig -d /dev/mst/mt4123_pciconf0 set DCB_PFC_UP_PPORT_RSP=1`。配置完成后，需要重启网卡驱动才能生效。

调试PFC问题，最常用的工具是交换机的`show interface priority-flow-control`命令，可以看到每个端口的PFC帧收发统计。如果发现某个端口收到的暂停帧数量异常多，说明对端缓存压力大，需要优化流量调度。另外，用`ethtool -S eth0`也可以查看网卡侧的PFC统计信息。

这里有个实际案例：某数据中心部署RoCEv2后，发现RDMA通信延迟波动很大。排查后发现，交换机的PFC触发阈值设成了50%，导致频繁发送暂停帧，链路利用率只有60%。把阈值调整到80%后，暂停帧数量减少了90%，延迟恢复稳定。

#### 严重警告

PFC配置不当会导致PFC风暴。当网络中出现环路时，交换机会互相发送暂停帧，最终导致整个网络瘫痪。务必在部署前确认STP或SPB等环路保护机制已启用。

### 5.3 ECN显式拥塞通知的核心原理

PFC是链路层的“硬刹车”，而ECN是网络层的“软刹车”。ECN的全称是Explicit Congestion Notification，它让交换机在IP包头上打一个标记，告诉接收端“网络有点堵了”。接收端看到标记后，会通知发送端降低发送速率。整个过程不需要丢包，也不需要重传，延迟影响很小。

ECN工作在IP层和传输层之间。IP头里有2个bit用于ECN：ECT（ECN-Capable Transport）和CE（Congestion Experienced）。发送端在发出数据包时，把ECT设为1，表示“我这个包支持ECN”。交换机如果发现缓存超过阈值，就把CE位设为1，表示“我经历拥塞了”。接收端收到CE=1的包后，会在确认包（ACK）中把ECN Echo标志置位，通知发送端降速。

在RoCEv2网络里，ECN的阈值设置比PFC更精细。因为ECN的目的是在拥塞发生前就发出预警，让发送端主动降速，避免触发PFC。所以ECN的触发阈值一般设得比PFC低。比如，PFC阈值是80%，ECN阈值可以设在60%。这样，网络轻度拥塞时先触发ECN，发送端降速；如果降速不够，缓存继续增长到80%，再触发PFC做硬刹车。

ECN的另一个关键参数是标记概率。不是所有超过阈值的包都会被标记，而是按一定概率标记。概率太高会导致发送端过度反应，概率太低又起不到预警作用。实际部署中，标记概率通常设在10%-30%之间，具体值需要根据网络流量模型调优。

```arch-diagram
┌─────────────────────────────────────────────────────────────┐
│                    ECN标记与反馈流程                          │
│                                                              │
│  发送端                   交换机                      接收端  │
```

### 5.4 ECN的配置与参数调优

ECN的配置主要涉及交换机和接收端两个方面。在交换机上，需要开启ECN功能，设置触发阈值和标记概率。不同厂商的配置命令不同，但核心参数是一样的。以某主流交换机为例，配置命令如下：

```bash
!
! 开启ECN功能
!
interface Ethernet1/1
  priority-flow-control mode on
  ecn profile ECN_PROFILE_1
    ecn threshold 60000 bytes
    ecn mark-probability 20
    ecn min-threshold 40000 bytes
    ecn max-threshold 80000 bytes
  exit
!
! 应用ECN profile到队列
!
class-map type qos match-any RDMA_TRAFFIC
  match cos 3
policy-map type qos ECN_POLICY
  class RDMA_TRAFFIC
    set ecn-profile ECN_PROFILE_1
interface Ethernet1/1
  service-policy type qos input ECN_POLICY
```

在接收端（服务器），需要确保网卡支持ECN并开启相关功能。Linux系统下，可以通过`sysctl`命令调整ECN相关参数：`sysctl -w net.ipv4.tcp_ecn=1`。对于RoCEv2，还需要在网卡驱动中开启ECN支持，比如Mellanox网卡的`mlxconfig`工具中设置`ECN_EN=1`。

ECN的调优是个迭代过程。先设置一个初始阈值，然后监控网络的延迟和吞吐量。如果延迟偏高，说明ECN阈值太高，需要降低；如果吞吐量上不去，说明ECN反应过度，需要提高阈值或降低标记概率。实际项目中，我们一般先用默认参数跑一周，收集数据后再做调整。

### 5.5 PFC与ECN的协同工作机制

PFC和ECN不是互相替代的关系，而是互补的。ECN是“预防性”的，它在拥塞刚出现时就通知发送端降速；PFC是“兜底”的，当ECN来不及降速或者降速不够时，PFC做最后的保护。一个设计良好的RoCEv2网络，应该让ECN处理90%以上的拥塞情况，PFC只作为最后一道防线。

协同工作的关键在于阈值设置。一般来说，ECN的触发阈值要比PFC低15%-20%。比如，交换机端口缓存是1MB，ECN阈值设在600KB，PFC阈值设在800KB。这样，当缓存达到600KB时，ECN开始标记包，发送端降速；如果发送端降速不及时，缓存继续增长到800KB，PFC才介入。

还有一个重要参数是ECN的标记概率和PFC的暂停时间之间的配合。如果ECN标记概率太高，发送端会急剧降速，导致链路利用率下降；如果标记概率太低，PFC会频繁触发，同样影响性能。实际调优时，我们会先固定PFC阈值，然后调整ECN参数，直到找到一个平衡点。

| 参数 | ECN | PFC | 协同效果 |
| --- | --- | --- | --- |
| 触发阈值 | 60%缓存 | 80%缓存 | ECN先预警，PFC兜底 |
| 响应时间 | RTT级别（微秒） | 暂停帧传输时间（纳秒） | ECN反应慢但平滑，PFC反应快但粗暴 |
| 影响范围 | 端到端 | 逐跳 | ECN全局调控，PFC局部保护 |
| 副作用 | 可能的吞吐量下降 | 可能的PFC风暴 | 合理配置可避免副作用 |

### 5.6 缓存管理与阈值计算

无论是PFC还是ECN，核心都是对交换机缓存的精细管理。现代数据中心交换机的端口缓存一般在几MB到几十MB之间。以一台32口100G交换机为例，每个端口可能只有4MB缓存。这点缓存对于高速流量来说，瞬间就能填满。所以，阈值计算非常关键。

缓存阈值通常用两种方式表示：绝对字节数和相对百分比。百分比方式更直观，但不够精确；字节数方式更精确，但需要根据链路速率调整。一个常用的计算公式是：`阈值 = 带宽 × RTT × 安全系数`。比如，100Gbps链路，RTT是10μs，安全系数取1.5，那么阈值 = 100Gbps × 10μs × 1.5 = 1.5Mb = 约188KB。这个阈值可以保证在收到暂停帧之前，缓存不会溢出。

实际工程中，我们还会考虑Headroom缓存。Headroom是发送暂停帧期间，对端可能继续发来的数据量。计算公式是：`Headroom = 带宽 × (暂停帧传输时间 + 链路延迟)`。100Gbps下，暂停帧传输时间约0.5μs，链路延迟约1μs，Headroom = 100Gbps × 1.5μs = 150Kb = 约19KB。所以，总缓存需求 = 触发阈值 + Headroom。

#### 设计要点

在规划缓存时，一定要留足Headroom。很多PFC丢包问题都是因为Headroom不够，导致暂停帧还没生效，缓存就已经溢出了。建议Headroom至少预留20%的余量。

### 5.7 死锁检测与预防策略

PFC最让人头疼的问题就是死锁。死锁发生的场景通常是：交换机A的端口1向交换机B的端口2发数据，同时交换机B的端口3向交换机A的端口4发数据。如果两个端口的缓存都满了，A发暂停帧给B，B发暂停帧给A，双方都在等对方先发，结果谁都不发，链路卡死。

预防死锁最有效的方法是避免环路。在数据中心网络里，通常使用CLOS架构（Leaf-Spine），这种架构天然无环。但如果配置了冗余链路或者VPC（虚拟端口通道），就可能出现环路。这时必须启用STP（生成树协议）或者更快的TRILL/SPB协议来破环。

另一个预防策略是设置PFC的超时机制。交换机在发送暂停帧后，启动一个定时器。如果超时后缓存还没恢复，就强制恢复发送，宁可丢包也不能死锁。这个超时时间一般设为暂停时间的2-3倍。比如，暂停时间片是65535，对应约13ms，超时时间可以设到30ms。

实际项目中，我们还会部署死锁检测工具。比如，交换机会监控每个端口的暂停帧收发频率。如果发现某个端口连续收到暂停帧超过一定次数（比如100次/秒），就触发告警，管理员可以手动干预。

### 5.8 性能调优实战案例

我们来看一个实际的调优案例。某互联网公司的RoCEv2网络部署后，发现RDMA写操作的延迟在P99（99分位）达到了200μs，远超预期的50μs。经过排查，发现PFC暂停帧数量非常高，平均每个端口每秒收到5000个暂停帧。

第一步，检查ECN配置。发现ECN没有开启，所有拥塞都靠PFC处理。于是开启ECN，设置阈值为60%，标记概率20%。部署后，PFC暂停帧下降到每秒500个，P99延迟降到100μs。

第二步，优化ECN参数。发现延迟仍然偏高，于是将ECN阈值降到50%，标记概率提高到30%。调整后，P99延迟降到60μs，但吞吐量下降了5%。这是因为标记概率太高，发送端降速过度。

第三步，精细调优。将标记概率回调到25%，同时调整PFC阈值到85%。最终，P99延迟稳定在45μs，吞吐量恢复到了99.5%。这个案例说明，PFC和ECN的调优需要反复迭代，找到最适合当前网络流量模型的参数组合。

### 5.9 常见故障排查与处理方法

PFC和ECN相关的故障，通常表现为延迟飙升、吞吐量下降或者偶发丢包。排查时，可以按以下步骤进行：

第一步，查看交换机的PFC统计。用`show interface priority-flow-control statistics`命令，看每个端口的暂停帧收发数量。如果某个端口收到的暂停帧特别多，说明对端缓存压力大，需要检查对端的流量模型。

第二步，检查ECN标记情况。用`show ecn statistics`命令，看被标记的包数量。如果标记数量很少但延迟很高，说明ECN阈值太高，需要降低。如果标记数量很多但延迟仍然高，说明发送端没有响应ECN，需要检查接收端配置。

第三步，抓包分析。在服务器端用tcpdump或Wireshark抓包，过滤ECN标记的包。如果发现CE标记的包很多，但ACK中没有ECN Echo，说明接收端网卡驱动不支持ECN反馈，需要升级驱动。

| 故障现象 | 可能原因 | 排查命令 | 解决方案 |
| --- | --- | --- | --- |
| 延迟飙升 | PFC频繁触发 | show interface pfc | 降低PFC阈值或开启ECN |
| 吞吐量下降 | ECN标记概率过高 | show ecn statistics | 降低标记概率或提高阈值 |
| 偶发丢包 | Headroom不足 | show buffer headroom | 增加Headroom缓存 |
| PFC风暴 | 网络环路 | show spanning-tree | 启用STP或破环 |

### 5.10 数据中心部署最佳实践

最后，总结一下PFC和ECN在数据中心部署的最佳实践。首先，必须同时开启PFC和ECN，不能只开一个。只开PFC会导致死锁风险，只开ECN无法应对突发流量。其次，阈值设置要遵循“ECN低、PFC高”的原则，一般ECN阈值比PFC低15%-20%。

第三，要合理规划优先级。RoCEv2流量建议使用独立的优先级（如优先级3），不要和TCP流量混在一起。如果必须混用，要确保不同优先级的队列隔离，避免互相影响。第四，部署监控工具，实时查看PFC和ECN的统计信息，及时发现异常。

第五，定期进行压力测试。用流量发生器模拟极端流量场景，验证PFC和ECN的响应是否正常。测试通过后再上线，可以避免很多线上问题。最后，建立应急预案。当出现PFC风暴或死锁时，能快速定位和恢复，比如通过自动化脚本批量重置端口。

#### 章节小结

本章详细讲解了PFC和ECN两个流量控制机制的原理、配置方法和调优技巧。PFC是链路层的逐跳流控，通过暂停帧防止缓存溢出；ECN是网络层的端到端流控，通过标记包通知发送端降速。两者协同工作，才能实现RoCEv2网络的高性能低延迟。重点掌握了阈值设置、缓存计算、死锁预防等工程实践方法。下一章将深入讲解RoCEv2的拥塞控制算法，看看发送端如何根据ECN反馈精确调整发送速率。

#### 本章重点回顾

- PFC基于优先级的逐跳流量控制，防止缓存溢出丢包
- ECN通过IP头CE标记实现端到端拥塞通知，避免触发PFC
- ECN阈值应低于PFC阈值15%-20%，实现先预警后兜底
- Headroom缓存必须预留足够余量，防止暂停帧生效前溢出
- 死锁预防需避免环路、设置超时机制、部署监控工具
