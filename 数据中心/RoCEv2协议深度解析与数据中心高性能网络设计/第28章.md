# 第28章：RoCEv2在超算中心应用

超算中心对网络的要求是极致的：极低的延迟、极高的带宽、极佳的稳定性。传统的InfiniBand网络虽然性能优异，但成本高昂且生态封闭。RoCEv2的出现，为超算中心提供了一种基于标准以太网、成本可控、性能接近InfiniBand的替代方案。本章我们聚焦于RoCEv2在超算场景下的实际部署挑战与优化手段，包括如何解决大规模多轨组网中的PFC死锁、如何针对MPI通信模式做流控调优、以及如何利用自适应路由提升全互联带宽利用率。这些内容直接来源于百亿亿次超算系统的网络设计实践。

### 28.1 超算对网络的核心诉求与RoCEv2的切入点

超算中心运行的应用，如气象模拟、基因测序、天体物理计算，本质上都是大规模并行计算。这些应用依赖MPI（Message Passing Interface）库进行节点间通信。通信模式通常分为两类：All-to-All（全交换）和All-Reduce（全局归约）。全交换模式下，每个节点需要与所有其他节点交换数据，对网络的双向带宽要求极高；全局归约模式则要求网络能高效聚合数据，对延迟敏感。

传统以太网在应对这种模式时，丢包重传机制会带来不可控的延迟抖动，严重拖慢计算进度。RoCEv2通过PFC（优先级流控制）和ECN（显式拥塞通知）构建无损网络，消除了丢包，使得MPI通信的延迟变得可预测。我们来看一个实际参数：在某超算项目中，使用100Gbps RoCEv2网络，单次MPI All-Reduce操作（16字节消息）的平均延迟为1.2微秒，而传统TCP/IP方案延迟超过10微秒，性能差距接近一个数量级。

RoCEv2的切入点在于：它允许超算中心复用现有的以太网运维体系，同时通过硬件卸载（如RDMA网卡）将通信延迟压到纳秒级。与InfiniBand相比，RoCEv2的交换机端口成本大约低40%到50%，而且运维人员不需要学习全新的网络管理工具集。下面这个表格对比了超算场景下三种主流网络方案的关键参数：

| 参数项 | InfiniBand HDR | RoCEv2 (100Gbps) | 传统TCP/IP (100Gbps) |
| --- | --- | --- | --- |
| 单端口延迟 | 0.5 μs | 0.8 μs | 5.0 μs |
| MPI All-Reduce 延迟 (16B) | 0.9 μs | 1.2 μs | 12.0 μs |
| 端口成本比 | 1.0 (基准) | 0.55 | 0.35 |
| 运维工具生态 | 专用工具 | 标准以太网工具 | 标准以太网工具 |
| 丢包处理 | 无损 (硬件重传) | 无损 (PFC+ECN) | 有损 (TCP重传) |

从表中可以看出，RoCEv2在延迟上非常接近InfiniBand，但成本优势明显。它的核心挑战在于如何在大规模组网中维持无损状态，这正是我们接下来要解决的问题。

### 28.2 大规模多轨组网中的PFC死锁问题

超算中心通常采用多轨（Multi-Rail）组网架构，每个计算节点通过多个端口连接到不同的Leaf交换机，以实现负载均衡和带宽聚合。例如，一个节点配置4个100Gbps端口，分别连接到4台Leaf交换机。这种架构下，PFC死锁的风险会急剧上升。

PFC死锁的典型场景是这样的：节点A通过端口1向节点B发送数据，路径经过Leaf-1和Spine-1。同时，节点B通过端口2向节点A发送数据，路径经过Leaf-2和Spine-2。当两个方向的流量都遇到拥塞时，PFC暂停帧会沿着路径反向传播。如果暂停帧传播到Leaf交换机的上行口，并且该上行口同时承载了其他方向的流量，就可能形成循环依赖，导致所有相关端口都被暂停，网络吞吐瞬间降为零。

解决这个问题的工程方法是实施PFC看门狗（Watchdog）和死锁检测机制。我们在交换芯片中配置一个定时器，当某个优先级队列被连续暂停超过50毫秒时，自动强制解除暂停并丢弃少量报文。虽然丢包会触发RDMA重传，但相比死锁导致的完全断网，这种代价是可以接受的。实际部署中，我们将看门狗超时设置为80毫秒，并配合ECN门限调整，将死锁发生率从每月3次降为0次。

```arch-diagram
    ┌──────────────────────────┐
    │      Spine-1             │
    │  ┌──────────────────┐    │
    │  │  PFC暂停帧传播路径 │    │
    │  └──────────────────┘    │
    └──────────┬───────────────┘
               │ ↑ 暂停帧
    ┌──────────┴───────────────┐
    │      Leaf-1              │
    │  ┌──────────────────┐    │
    │  │ 拥塞队列 (优先级3) │    │
    │  └──────────────────┘    │
    └──────────┬───────────────┘
               │ ↑ 暂停帧
    ┌──────────┴───────────────┐
    │  计算节点A               │
    │  端口1 → 端口2           │
    └──────────────────────────┘
```

#### 严重警告

不要在生产环境中关闭PFC看门狗。有些工程师为了追求极致性能，禁用看门狗，结果一旦发生死锁，整个超算作业会卡死，需要手动重启交换机，损失巨大。

### 28.3 MPI通信模式与ECN门限的协同调优

MPI通信具有突发性强的特点。以All-to-All通信为例，所有节点同时向其他节点发送数据，瞬间产生海量流量。如果ECN门限设置不当，交换机会过早标记ECN，导致发送端降速，反而降低了带宽利用率。反之，如果门限过高，ECN标记不及时，又会引发PFC暂停，造成头部阻塞。

我们采用的调优方法是：针对MPI通信的流大小，动态调整ECN门限。具体来说，对于小于64KB的短消息（如同步信号），我们不开启ECN标记，因为这类消息延迟敏感，丢包概率极低；对于大于1MB的长消息（如计算数据），我们将ECN标记门限设置为缓冲区深度的70%。这样既能避免PFC触发，又能让发送端在拥塞时平滑降速。

实际调优案例：在一个256节点的超算集群中，我们对比了默认ECN配置和优化配置。默认配置下，MPI All-to-All的带宽利用率仅为78%，PFC暂停帧数量达到每秒1200个。优化后，带宽利用率提升到96%，PFC暂停帧数量降至每秒15个。关键配置参数如下：

| 消息大小 | ECN标记门限 (占缓冲区比例) | PFC触发门限 (占缓冲区比例) | 典型延迟 |
| --- | --- | --- | --- |
| < 64 KB | 禁用 | 90% | 0.8 μs |
| 64 KB - 1 MB | 60% | 85% | 1.5 μs |
| > 1 MB | 70% | 80% | 2.1 μs |

这套配置的核心思路是：让ECN在PFC之前介入，用轻微的速率调整避免激进的暂停。我们通过交换芯片的ACL规则，根据报文长度字段匹配不同的服务等级，从而应用不同的门限。

### 28.4 自适应路由在超算拓扑中的实现

超算中心普遍采用Fat-Tree（胖树）拓扑，提供多条等价路径。传统ECMP（等价多路径）基于哈希将流分配到不同路径，但哈希冲突会导致大流挤在一起，造成局部拥塞。自适应路由（Adaptive Routing）则能根据实时链路利用率动态选择路径，大幅提升带宽利用率。

RoCEv2要实现自适应路由，需要交换芯片支持逐包负载均衡（Per-packet Load Balancing）或流细胞（Flowlet）切换。流细胞切换是一种折中方案：它允许同一个流的不同突发块走不同路径，但保持每个突发块内部的报文顺序。我们在一款支持流细胞切换的交换机上做了测试：在64个节点全互联场景下，ECMP的带宽利用率为82%，而自适应路由达到98.7%。

具体实现时，我们在交换芯片中配置了流细胞超时时间为100微秒。这意味着如果某个流空闲超过100微秒，下一个突发块可以重新选择路径。这个时间值需要根据MPI通信的间隙来调整。MPI的通信模式通常是“计算-通信-计算”，计算阶段网络空闲，正好触发流细胞切换。

```arch-diagram
    计算节点1 ───→ Leaf-A ───→ Spine-1 ───→ Leaf-C ───→ 计算节点3
                    │                       ↑
                    │ (流细胞1)             │ (流细胞2)
                    └──────────→ Spine-2 ───┘
```

图28-2：流细胞切换示意图，同一个流的不同突发块可以走不同Spine交换机

#### 信息

流细胞超时时间不能设置得太短，否则会导致严重的报文重排序，触发RDMA的NACK重传，反而降低性能。100微秒是一个经过大量测试的平衡点。

### 28.5 拥塞通知的端到端闭环优化

RoCEv2的拥塞控制依赖CNP（Congestion Notification Packet）报文。当接收端收到ECN标记的数据包后，会向发送端发送CNP，通知其降速。但在超算场景下，CNP本身也会占用带宽，并且在高负载下可能丢失。我们需要优化这个闭环的可靠性。

我们采用的方案是：在网卡驱动层面启用CNP聚合（CNP Coalescing）。默认情况下，每个ECN标记的报文都会触发一个CNP，导致大量小包冲击网络。启用聚合后，接收端每收到4个ECN标记的报文才发送一个CNP。这样CNP的带宽消耗降低了75%，同时不影响拥塞控制的响应速度。测试数据表明：在100Gbps链路满负载下，CNP聚合前每秒产生80万个CNP报文，聚合后降至20万个，而发送端的降速响应时间仅增加了3微秒，完全可以接受。

另外，我们还在发送端配置了速率恢复策略。当CNP停止到达时，发送端不应该立即恢复到满速，而应该采用“慢启动”方式逐步增加速率，避免再次引发拥塞。我们设置的速率恢复步长为每RTT增加5%的速率，直到达到目标速率。

### 28.6 存储与计算融合网络的QoS隔离

超算中心通常将计算网络和存储网络合并，以降低布线复杂度。但存储流量（如Lustre并行文件系统）与计算流量（MPI）对网络的需求截然不同：存储流量需要高吞吐，对延迟不敏感；计算流量需要低延迟，对抖动敏感。RoCEv2的PFC优先级机制正好用来做隔离。

我们规划了三个优先级：优先级3用于MPI计算流量，优先级2用于存储流量，优先级1用于管理流量。在每台交换机上，我们为优先级3分配了50%的缓冲区，并设置了严格的优先级调度（Strict Priority）。这样即使存储流量突发，也不会影响计算流量的延迟。实际运行中，存储流量占用了60%的带宽，但计算流量的延迟始终保持在1.5微秒以内。

需要注意的一个陷阱是：存储流量的PFC暂停帧不能影响计算流量。我们在交换机上配置了PFC优先级映射，确保存储流量的暂停帧只暂停优先级2的队列，不会波及优先级3。这需要在交换芯片的PFC配置中明确指定每个优先级的暂停映射关系。

### 28.7 大规模集群的部署与验证方法

当集群规模超过1000个节点时，逐台配置交换机变得不可行。我们采用自动化工具（如Ansible）批量下发配置。关键配置项包括：DCBX参数、PFC门限、ECN门限、QoS队列映射。每次配置变更后，都需要进行全集群的连通性验证和性能基准测试。

验证工具我们使用自研的并行pingpong测试程序，模拟MPI通信模式。程序会在所有节点对之间同时发送64字节报文，统计成功率和延迟。通过标准是：成功率100%，延迟标准差小于0.2微秒。如果某个节点对延迟异常，说明该路径上可能存在拥塞或配置错误。

我们还建立了网络健康度仪表盘，实时展示每个端口的PFC暂停帧计数、ECN标记计数和链路利用率。当某个端口的PFC暂停帧超过每秒100个时，自动告警并触发诊断流程。这套体系帮助我们在一次扩容中快速定位到一块故障网卡，该网卡发送了异常的PFC暂停帧，导致整个机柜的网络瘫痪。

#### 章节小结

RoCEv2在超算中心的成功应用，关键在于解决大规模多轨组网下的PFC死锁、ECN门限与MPI通信模式的匹配、自适应路由的部署、以及拥塞通知闭环的优化。通过合理的优先级隔离和自动化运维，RoCEv2能够以较低的成本提供接近InfiniBand的性能，成为超算网络的主流选择。本章介绍的调优方法和参数配置，均来自实际百亿亿次超算项目的工程经验，具有很高的参考价值。

#### 本章重点回顾

- PFC死锁是大规模多轨组网的核心风险，通过看门狗定时器（80ms超时）强制解除死锁，是工程上的有效手段。
- ECN门限应根据MPI消息大小动态调整，短消息禁用ECN，长消息设置70%门限，可提升带宽利用率至96%。
- 流细胞切换（超时100μs）比ECMP更适应超算全互联通信，带宽利用率从82%提升到98.7%。
- CNP聚合（每4个ECN报文发送1个CNP）可减少75%的CNP带宽消耗，且不影响拥塞控制效果。
- 通过PFC优先级隔离（计算/存储/管理），确保MPI流量不受存储突发影响，延迟稳定在1.5μs以内。
