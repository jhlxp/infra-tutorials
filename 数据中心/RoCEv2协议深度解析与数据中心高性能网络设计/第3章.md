# 第3章：RoCEv2报文封装与解析

这一章咱们直接来看RoCEv2的报文长什么样。很多搞网络的人一听到RDMA，就觉得很神秘，好像数据能直接从一个机器的内存飞到另一个机器的内存，不需要CPU参与。但说到底，数据在物理介质上传输，还是得靠以太网帧。RoCEv2就是把RDMA的数据，塞到标准的UDP/IP/Ethernet报文里。这一章，我们就来拆解这个报文，从最外层的以太网帧头，一路拆到最核心的RDMA操作码，把每个字段的含义、大小、作用都讲清楚。搞懂了报文结构，你才能理解为什么RoCEv2需要无损网络，为什么PFC和ECN能起作用，这些都是后续章节的基础。

### 3.1 报文整体封装层次

我们先来看一下RoCEv2报文在OSI模型里的位置。传统的以太网报文，从下往上依次是：MAC头、IP头、TCP/UDP头、应用层数据。RoCEv2把应用层数据替换成了InfiniBand的传输层报文。这里有个关键点：RoCEv2用的是UDP，不是TCP。为什么不用TCP？因为TCP的可靠性机制（重传、拥塞控制）会引入不确定的延迟，这对于需要极低延迟的RDMA来说是致命的。UDP只管发，不保证可靠，那可靠性和流控怎么办？这些功能下沉到了RoCEv2的传输层和网络层，由硬件来保证。

整个封装过程，就像套娃。最里面是InfiniBand的传输层报文（BTH + Payload），外面套上UDP头，再外面套上IP头，最外面是MAC头和CRC。这个结构决定了RoCEv2报文在标准以太网交换机上也能转发，因为交换机只看MAC和IP头。但问题也来了：交换机不认识UDP里面的内容，所以它无法感知RDMA的流控状态。这就需要我们通过其他手段（比如PFC）来告诉交换机“这个流的数据很重要，别丢包”。

```arch-diagram
        ┌─────────────────────────────────────────────────────────┐
        │                     以太网帧 (Layer 2)                  │
        ├────────────┬──────────┬──────────┬─────────────────────┤
        │ MAC 目的   │ MAC 源   │  Ether   │                     │
        │ 地址 (6B)  │ 地址 (6B)│ Type     │      Payload        │
        │            │          │ (0x0800) │                     │
        ├────────────┴──────────┴──────────┴─────────────────────┤
        │                      IP 头 (Layer 3)                   │
        ├────────────────┬──────────────────┬────────────────────┤
        │ Version=IHL    │   ToS/DSCP       │   总长度           │
        │ (1B)           │   (1B)           │   (2B)             │
        ├────────────────┼──────────────────┼────────────────────┤
        │   源IP地址 (4B)│   目的IP地址 (4B)│   协议=17 (UDP)    │
        ├────────────────┴──────────────────┴────────────────────┤
        │                     UDP 头 (Layer 4)                    │
        ├──────────┬──────────┬──────────┬───────────────────────┤
        │ 源端口   │ 目的端口 │  长度    │  校验和               │
        │ (2B)     │ (4791)   │  (2B)    │  (2B)                 │
        ├──────────┴──────────┴──────────┴───────────────────────┤
        │                 InfiniBand 传输层 (BTH + Payload)       │
        │  ┌──────────┬──────────┬──────────────────────────┐    │
        │  │  BTH     │   Payload│   ICRC (4B)              │    │
        │  │  (12B)   │   (可变) │                           │    │
        │  └──────────┴──────────┴──────────────────────────┘    │
        ├─────────────────────────────────────────────────────────┤
        │                      FCS (4B)                           │
        └─────────────────────────────────────────────────────────┘
```

图3-1: RoCEv2报文完整封装层次，从MAC到IB传输层

从上图可以看到，整个报文长度是动态的，最小约64字节（空载），最大可以达到9000字节（巨型帧）。实际部署中，我们建议使用MTU=4096或9000，这样能减少报文个数，提升吞吐。但要注意，交换机和网卡都得支持相同的MTU，否则会被分片，分片后的报文在接收端重组会很麻烦，甚至导致性能下降。

### 3.2 以太网MAC头与FCS

MAC头是报文的最外层，它决定了报文在局域网内怎么转发。对于RoCEv2来说，MAC头里的目的MAC地址，通常是下一跳路由器的MAC，或者是目标服务器的MAC。源MAC就是发送端网卡的MAC。EtherType固定为0x0800，表示上层是IPv4。如果用的是IPv6，EtherType就是0x86DD。目前绝大多数数据中心部署RoCEv2都使用IPv4，因为IPv4的地址空间在数据中心内部足够用，而且IPv6的头部处理开销更大。

FCS（帧校验序列）是CRC32校验，覆盖整个以太网帧（从目的MAC到Payload末尾）。这个校验由硬件自动计算和校验。如果FCS校验失败，网卡会直接丢弃这个帧，不会上报给协议栈。这就是为什么我们说物理层的误码率很重要——哪怕只有1bit错误，整个帧就废了。对于RoCEv2来说，丢一个帧可能意味着整个RDMA操作都要重传，延迟飙升。所以在实际项目中，我们要求光纤和光模块的误码率低于10^-12。

#### 严重警告

不要试图在软件层面修改FCS字段。网卡硬件在发送时会自动覆盖这个字段，你写什么都会被冲掉。调试时如果发现FCS错误，先查物理链路，别查代码。

### 3.3 IP头与DSCP标记

IP头负责路由。RoCEv2使用标准的IPv4头，没有特殊字段。但有一个字段非常关键：ToS（Type of Service）字节，具体来说是其中的DSCP（Differentiated Services Code Point）部分。DSCP占6个bit，用于区分报文的服务等级。在RoCEv2网络中，我们通常把RoCEv2流量的DSCP值设置为某个特定值（比如26或46），然后交换机根据这个DSCP值，将报文映射到对应的优先级队列里，实现无损队列的隔离。

这里有个常见的坑：DSCP标记必须在发送端设置好。如果发送端没设，或者设错了，交换机就不会把它当作高优先级流量处理。我们来看一个实际的配置：在Mellanox（现NVIDIA）网卡上，可以通过以下命令设置RoCEv2流量的DSCP值：`mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_DSCP=26`。设置完后，所有从这个网卡发出的RoCEv2报文，IP头里的DSCP字段都会变成26。交换机那边也要做对应的映射：将DSCP 26映射到优先级队列3（通常是无损队列）。

| DSCP值 | 二进制 | 用途 | 队列映射 |
| --- | --- | --- | --- |
| 0 | 000000 | 尽力而为流量 | 队列0 |
| 26 | 011010 | RoCEv2高优先级 | 队列3 |
| 46 | 101110 | RoCEv2紧急流量 | 队列5 |
| 48 | 110000 | 网络控制流量 | 队列7 |

表3-1: 常见DSCP值及其在RoCEv2网络中的映射关系

### 3.4 UDP头与端口号4791

UDP头是RoCEv2报文最核心的“身份标识”。IANA（互联网号码分配局）为RoCEv2分配了固定的目的端口号：4791。也就是说，任何目的端口为4791的UDP报文，都应该被接收端当作RoCEv2报文来处理。源端口号通常由发送端随机生成，用于实现等价多路径（ECMP）的负载均衡。因为交换机在做哈希转发时，会看IP五元组（源IP、目的IP、源端口、目的端口、协议）。如果所有RoCEv2报文的源端口都一样，那它们就会走同一条路径，无法利用多路径的带宽。

在实际部署中，有些网卡驱动支持配置源端口的生成策略。比如，可以基于QP号（Queue Pair）来生成源端口，这样不同的RDMA流就会有不同的源端口，从而实现流级别的负载均衡。但要注意，ECMP的哈希粒度是流，不是报文。同一个流（五元组相同）的所有报文，一定会走同一条路径。如果路径出现拥塞，这个流的所有报文都会受影响。这个问题我们会在后面讲负载均衡时详细讨论。

UDP校验和是可选的。但在RoCEv2中，强烈建议启用UDP校验和。因为UDP校验和覆盖了UDP头和数据部分，可以检测出报文在传输过程中的比特错误。如果校验和错误，接收端会直接丢弃报文，不会触发RDMA层的重传（RDMA重传依赖BTH里的PSN）。这能避免将损坏的数据提交给应用程序。但开启校验和会消耗一些CPU资源（虽然硬件可以offload）。在性能敏感的场景下，有些团队会关闭校验和，但这会增加数据静默损坏的风险。我的建议是：在存储场景（比如NVMe over Fabric）必须开启；在HPC场景（比如MPI）可以关闭，以换取微秒级的延迟降低。

### 3.5 BTH（Base Transport Header）深度解析

BTH是InfiniBand传输层的头部，固定12字节。它包含了RDMA操作的核心控制信息。我们来看一下它的结构：

```arch-diagram
        ┌──────────────────────────────────────────────────────────┐
        │                    BTH (12 Bytes)                        │
        ├────────┬────────┬────────┬────────┬────────┬────────────┤
        │ OpCode │  Flags  │  PSN   │  P_Key │  LNH   │  Reserved  │
        │ (1B)   │  (1B)   │  (4B)  │  (2B)  │  (2B)  │  (2B)      │
        ├────────┴────────┴────────┴────────┴────────┴────────────┤
        │  OpCode示例:                                             │
        │  0x01: SEND First / Last                                 │
        │  0x04: RDMA Write First                                  │
        │  0x05: RDMA Write Middle                                 │
        │  0x06: RDMA Write Last                                   │
        │  0x07: RDMA Write Only                                   │
        │  0x08: RDMA Read Request                                 │
        │  0x09: RDMA Read Response First / Only                   │
        │  0x0C: ACK                                               │
        │  0x0D: NAK                                               │
        └──────────────────────────────────────────────────────────┘
```

图3-2: BTH头部结构及常见操作码

我们来逐个字段解释：
**OpCode（操作码）**：8bit，标识这个报文是什么类型的RDMA操作。比如0x07表示“RDMA Write Only”，意味着这个报文包含完整的写数据，不需要分片。0x08是“RDMA Read Request”，请求对端返回数据。0x0C是ACK，确认收到。这个字段是接收端硬件解析报文的关键，它决定了硬件该把数据写到内存的哪个位置。
**Flags（标志位）**：8bit，包含了一些控制信息，比如是否请求ACK，是否是Solicited事件等。
**PSN（Packet Sequence Number）**：32bit，报文的序列号。接收端用这个序列号来检测报文是否丢失、乱序。如果发现PSN不连续，就说明丢了报文，需要触发重传。PSN是RoCEv2可靠性机制的基础。
**P\_Key（分区键）**：16bit，用于隔离。只有P\_Key匹配的QP才能通信，类似于VLAN。在数据中心内部，通常所有节点使用同一个P\_Key（0xFFFF）。
**LNH（Link Next Header）**：2bit，指示BTH后面是否还有额外的头部（比如RETH、Atomic ETH）。对于简单的SEND操作，LNH=0。

### 3.6 RETH与Atomic扩展头部

当OpCode指示这是一个RDMA Read或Atomic操作时，BTH后面会紧跟一个扩展头部。最常见的是RETH（RDMA Extended Transport Header），用于RDMA Read操作。RETH包含了三个关键信息：虚拟地址（VA）、R\_Key（远程内存访问密钥）、和数据长度。接收端硬件根据RETH里的VA和R\_Key，将数据直接写入到指定的内存地址。这个过程完全由硬件完成，不需要CPU介入。

RETH的结构如下：

```arch-diagram
        ┌──────────────────────────────────────────────────────────┐
        │                   RETH (16 Bytes)                        │
        ├────────────────────────┬─────────────────────────────────┤
        │  虚拟地址 (VA, 8B)     │    R_Key (4B)    │  长度 (4B)  │
        ├────────────────────────┴─────────────────────────────────┤
        │  VA: 0x7F2A_0000_1000  (目标内存起始地址)                │
        │  R_Key: 0x1234_5678    (权限校验)                        │
        │  长度: 0x0000_1000     (4KB数据)                         │
        └──────────────────────────────────────────────────────────┘
```

图3-3: RETH头部结构，用于RDMA Read操作

这里有个关键点：R\_Key是内存注册时生成的，它和虚拟地址、权限绑定在一起。接收端硬件在收到RDMA Read请求时，会检查R\_Key是否合法。如果R\_Key不匹配，硬件会直接丢弃报文，并生成一个错误事件。这提供了一种内存保护机制。在实际项目中，我们经常遇到R\_Key配置错误导致通信失败的问题。调试方法是：在发送端打印出R\_Key的值，在接收端用`ibv_query_qp`工具查看QP的R\_Key配置，对比是否一致。

### 3.7 ICRC与数据完整性

在BTH和Payload之后，还有一个ICRC（Invariant CRC）。这个CRC覆盖了从BTH开始到Payload结束的所有数据，但不包括IP头和UDP头。ICRC是InfiniBand规范里定义的，用于保证数据在传输过程中的完整性。注意，ICRC和以太网的FCS不同：FCS覆盖整个帧，而ICRC只覆盖IB传输层部分。这样设计的好处是，如果报文在路由器里被修改了IP头（比如TTL减1），FCS会重新计算，但ICRC不会变，接收端仍然可以验证IB数据部分的完整性。

ICRC的计算由硬件完成，对软件透明。在调试时，如果发现ICRC错误，通常意味着内存故障或者PCIe链路有问题。因为数据在发送端从内存搬到网卡时，如果PCIe传输发生了错误，可能导致数据被篡改，但FCS可能还没计算（或者计算的是错误的数据），最终接收端会发现ICRC不匹配。我们曾在一个项目中遇到频繁的ICRC错误，排查了光纤、光模块、交换机都没问题，最后发现是服务器主板的PCIe插槽接触不良，更换插槽后问题解决。

### 3.8 报文解析实战：用Wireshark抓一个RoCEv2报文

光说不练假把式。我们来看看怎么用Wireshark抓取并解析一个真实的RoCEv2报文。首先，你需要在服务器上安装Wireshark。然后，在发送端运行一个RDMA程序（比如`ib_send_bw`），在接收端启动抓包。抓包时要注意：由于RoCEv2报文经过网卡硬件加速，普通tcpdump可能抓不到（因为数据没经过内核协议栈）。你需要使用支持“原始包捕获”的工具，或者配置网卡的“loopback”模式。Mellanox网卡提供了`tcpdump -i mlx5_0`的方式，可以直接捕获硬件队列里的报文。

抓到报文后，Wireshark会自动识别RoCEv2（因为目的端口是4791）。你可以展开各个头部查看字段。我们来看一个典型的RDMA Write Only报文的解析结果：

```text
Frame 1: 1024 bytes on wire (8192 bits)
Ethernet II, Src: Mellanox_ab:cd:ef, Dst: Mellanox_12:34:56
Internet Protocol Version 4, Src: 10.0.0.1, Dst: 10.0.0.2
    Differentiated Services Field: 0x68 (DSCP 26)
User Datagram Protocol, Src Port: 54321, Dst Port: 4791
InfiniBand Base Transport Header
    OpCode: RDMA Write Only (0x07)
    Flags: 0x00
    Packet Sequence Number: 123456
    P_Key: 0xFFFF
    LNH: 0x0
    Reserved: 0x0000
Data (1024 bytes)
    [ICRC: 0xA1B2C3D4]
```

从这个输出可以看到：DSCP值为26，目的端口4791，OpCode=0x07，PSN=123456。这些信息对于诊断网络问题非常有用。比如，如果你发现某个流的PSN不连续，说明中间丢了包。如果DSCP值不对，说明发送端配置有问题。

#### 信息

抓包时如果看不到IB头部，可能是Wireshark版本太低，或者没有启用RoCEv2解析器。建议使用Wireshark 3.6以上版本，并确保编译时启用了InfiniBand支持。

### 3.9 常见封装错误与排查

在实际部署中，报文封装环节经常出现一些问题，导致通信失败。我总结了三个最常见的：

**问题1：UDP目的端口不是4791。** 有些网卡驱动默认使用RoCEv1（基于IB的GRH头），而不是RoCEv2。你需要检查网卡配置，确保开启了RoCEv2模式。在Mellanox网卡上，命令是：`mlxconfig -d /dev/mst/mt4123_pciconf0 set ROCE_MODE=2`。如果没改，发出来的报文目的端口是0，交换机不认识，直接丢弃。

**问题2：DSCP标记不一致。** 发送端设了DSCP 26，但交换机把DSCP 26映射到了尽力而为队列。这会导致RoCEv2流量和普通流量争抢缓冲区，引发丢包。解决方法是在交换机的全局配置里，将DSCP 26映射到无损队列，并启用PFC。

**问题3：MTU不匹配。** 发送端网卡MTU=9000，但中间交换机某个端口的MTU=1500。大包经过这个端口时会被分片。分片后的报文在接收端重组，但UDP校验和只覆盖第一个分片，后续分片的数据完整性无法保证。更严重的是，如果某个分片丢了，整个RDMA操作都得重传。所以必须确保端到端MTU一致。

#### 章节小结

这一章我们完整拆解了RoCEv2的报文结构，从外到内：MAC头、IP头（含DSCP）、UDP头（端口4791）、BTH（含OpCode和PSN）、扩展头部（RETH）、以及ICRC。每个字段都有其特定的作用，共同保证了RDMA数据能在以太网上高效、可靠地传输。理解了报文结构，你就能看懂抓包结果，也能快速定位配置错误。下一章我们将深入讨论无损网络的核心机制——PFC和ECN，看看它们是如何与这些报文头部协同工作的。

#### 本章重点回顾

- **RoCEv2封装层次**：MAC→IP→UDP→IB BTH，UDP目的端口固定为4791。
- **DSCP标记**：通过IP头ToS字段设置，用于区分RoCEv2流量优先级，典型值26。
- **BTH核心字段**：OpCode（操作类型）、PSN（序列号，用于可靠性）、P\_Key（分区隔离）。
- **RETH扩展头**：用于RDMA Read操作，包含虚拟地址和R\_Key，实现远程内存直接访问。
- **ICRC与FCS**：ICRC保护IB传输层数据完整性，FCS保护整个以太网帧，两者分工不同。
